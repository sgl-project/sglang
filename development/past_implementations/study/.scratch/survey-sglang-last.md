## 3. Channel Config Artifact
- JSON: keys = `"model.layers.<i>.self_attn.<q|k>_proj"`; values = list of per-head channel-index lists (each list length = head_dim, e.g. 128).
- Only first `ds_heavy_channel_num` indices kept per head (`model_runner.py:2167`).
- Generated by external tool: comment in `test/manual/test_double_sparsity.py:25` cites `https://github.com/andy-yang-1/DoubleSparse/blob/main/evaluation/group_channel_config.py`.
- Loading: `model_runner.py:2160–2171 init_double_sparsity_channel_config()` — builds `self.sorted_channels` list of CUDA tensors, shape per layer `[num_heads, heavy_channel_num]`.

## 4. Where Sparsity Lives
### A. KV Cache pool — DoubleSparseTokenToKVPool
- File `python/sglang/srt/mem_cache/memory_pool.py:1972–2060`.
- Dual buffers per layer: full `k_buffer`, `v_buffer` `[size+page_size, head_num, head_dim]` **plus** `label_buffer` `[size+1, head_num, heavy_channel_num]`.
- New methods: `get_label_buffer(layer_id)` (2038), `set_kv_buffer(layer, loc, cache_k, cache_v, cache_label)` (2047–2059).
- Instantiated in `model_runner_kv_cache_mixin.py:412–425`.

### B. Attention backend — DoubleSparseAttnBackend
- File `python/sglang/srt/layers/attention/double_sparsity_backend.py` (258 lines).
- Loads `sorted_channels` (line 34) and `sparse_decode_threshold` (35–36).
- Working buffers (60–96): `att_out_approx[num_head, batch, max_seq_len]`, `mid_out[batch, num_head, block_seq_num, head_dim]`, `mid_o_logexpsum[batch, num_head, block_seq_num]`.
- `forward_extend()` (113–165): **always dense**; extracts label channels from K (128–134) and writes K, V, K_label to pool (137–139); calls `extend_attention_fwd`.
- `forward_decode()` (167–257): routing decision (211–214):
  ```
  if (min_seq_len < heavy_token_num or max_seq_len < sparse_decode_threshold):
      decode_attention_fwd(...)   # dense fallback
  else:
      decode_sparse_attention_fwd(...)  # sparse 3-stage
  ```
  Sparse path extracts Q_label (231–237) and calls 3-stage kernel (238–255).

### C. Triton kernels — triton_ops/double_sparsity_attention.py (1,106 lines)
- Dense fallback: `_fwd_kernel_flash_decode_stage1` (29–136), `_fwd_kernel_flash_decode_stage2` (140–188); wrapper `flash_decode_attention_fwd` (284–326).
- **Sparse pipeline (3-stage):**
  1. `_sparse_fwd_kernel_flash_decode_stage1` (329–398) — Q_label @ K_label_bufferᵀ → `Att_Out[head, batch, seq_len]` approximate logits via BGEMV; wrapper `sparse_flash_decode_stage1` (561–609).
  2. CPU step (line 759): `topk_token_indices = torch.topk(att_out_approx, heavy_token_num, dim=-1).indices` → `[num_head, batch, heavy_token_num]`.
  3. `_sparse_fwd_kernel_flash_decode_stage2` (400–513) — full Q,K,V attn over selected top-k positions only (using req_to_token + topk indices); wrapper `sparse_flash_decode_stage2` (613–670).
  4. `_sparse_fwd_kernel_flash_decode_stage3` (517–558) — Welford-style block reduction → final O; wrapper `sparse_flash_decode_stage3` (674–696).
- Master orchestrator: `flash_decode_sparse_attention_fwd` (700–776).
- Extend/prefill kernel `_fwd_kernel` (782–990); wrapper `extend_attention_fwd` (996–1105). Adapted from SGLang v0.4.2 reference.
- Head dims supported: 16, 32, 64, 128 (stage1–3); 192, 256, 288, 576 supported in extend (1021–1032).
- SM-tuned BLOCK_M/BLOCK_N (1040–1053): Hopper SM9.0 vs Ampere SM8.x vs HIP (AMD).

### D. Model wiring
- **No per-model changes.** Works for any model with standard `model.layers.<i>.self_attn.{q,k}_proj` naming and Triton-compatible attention.
- Channel-config loading: `model_runner.py:668–674, 2156–2171`.
- Backend dispatch: `python/sglang/srt/layers/attention/attention_registry.py:101–106`.

### E. CPU offloading
- **Not implemented.** Label cache is GPU-resident; full KV stays in pool.

## 5. Scheduler / Runner Integration
- CUDA graphs disabled (line 932) — because top-k selection is dynamic per request.
- Prefill always dense; sparsity is decode-only.
- Routing decision is **per-batch** (not per-request): batch must be homogeneous in path.
- `init_forward_metadata()` (52–111) precomputes `seq_lens`, `req_to_token` alignment, max/min seqlen, allocates working buffers.

## 9. Open Questions
1. How exactly are channel rankings computed in the external tool? (mean |Q·K|? other metric?)
2. Why removed in #23009 — no rationale in commit msg. Maintenance burden? Low usage? Superseded?
3. Why is prefill never sparse? Paper might suggest it; not implemented.
4. `heavy_token_num` static per launch — comment hints at dynamic future but unimplemented.
5. No accuracy delta benchmark (sparse vs dense on same model).
6. Multi-GPU TP/PP not validated — likely transparent but untested.
7. Config keys assume LLaMA-style projection naming; non-LLaMA models would need separate config generation. No coverage for DeepSeek-V3.2 / GLM-5.1 MLA-style attention.
