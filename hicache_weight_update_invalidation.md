# HiCache Storage Invalidation After Online Weight Update

## Summary

When online weight update is called with `flush_cache=True`, SGLang must invalidate all KV cache entries generated by the previous weights. Today the scheduler flush clears live GPU/host KV state, but it does not clear persistent HiCache storage. As a result, the next request with the same prompt can reload stale KV pages from L3 storage and run with KV data produced by old weights.

This document records the reproduction, root cause, and implementation plan for fixing GitHub issue [#26792](https://github.com/sgl-project/sglang/issues/26792). The validation strategy is E2E-only; no unit or CI tests are required for this change plan.

## Root Cause

The online weight update flow and HiCache storage invalidation are currently disconnected.

Relevant code paths:

- `python/sglang/srt/managers/scheduler_components/weight_updater.py`
  - `SchedulerWeightUpdaterManager.flush_cache_after_weight_update()` only calls `flush_cache()` when `recv_req.flush_cache` is true.
  - All online weight update entry points share this helper: disk, distributed, tensor, and IPC.
- `python/sglang/srt/managers/scheduler.py`
  - `Scheduler.flush_cache()` resets live scheduler state, the tree cache, request/token pools, grammar state, metrics, draft worker cache, and optionally the CUDA allocator cache.
  - It does not clear HiCache storage.
  - `Scheduler.clear_hicache_storage_wrapped()` exists as a separate admin path for `/hicache/storage-backend/clear`.
- `python/sglang/srt/mem_cache/hicache_storage.py`
  - `HiCacheFile.config_suffix` includes model name and parallelism information, but not weight version or cache generation.
  - Existing files stay addressable after an online weight update.
- `python/sglang/srt/mem_cache/radix_cache.py`
  - `RadixKey.hash_page()` hashes token pages and the previous page hash.
  - It does not include weight version.

Therefore, `flush_cache=True` invalidates L1/L2 and in-memory radix metadata, but the L3 storage namespace remains the same. The following request rebuilds the in-memory tree by matching the same token-page hashes and can fetch KV pages created before the weight update.

## Goals

- Preserve the correctness meaning of `flush_cache=True` for online weight update: no KV produced by old weights may be reused after a successful update.
- Cover all online weight update sources through one shared implementation point.
- Make the change safe for HiCache file storage, distributed storage backends, TP/PP/CP deployments, and dynamic storage attach/detach.
- Keep the first fix narrow enough to land quickly.
- Validate by a real E2E reproduction flow rather than adding tests.

## Non-Goals

- Do not add unit tests or registered tests for this task.
- Do not change the public online weight update API.
- Do not require users to pass `weight_version`; correctness must hold even when it is absent.
- Do not make model-specific assumptions.
- Do not solve cache namespace design for all future use cases in the first patch if a smaller correctness fix is sufficient.

## Recommended Fix

### Phase 1: Correctness Fix

Clear HiCache L3 storage as part of `flush_cache=True` after a successful online weight update.

The shared implementation point should be `SchedulerWeightUpdaterManager.flush_cache_after_weight_update()` because every online weight update path already calls it after worker-side success.

Proposed flow:

```python
def flush_cache_after_weight_update(self, recv_req) -> None:
    if not recv_req.flush_cache:
        return

    flush_cache_success = self.flush_cache(
        empty_cache=recv_req.torch_empty_cache
    )
    if not flush_cache_success:
        raise RuntimeError("Cache flush failed after updating weights")

    if self.scheduler is None:
        raise RuntimeError("Scheduler is required to invalidate HiCache storage")
    storage_clear_success = self.scheduler.clear_hicache_storage_after_weight_update()
    if not storage_clear_success:
        raise RuntimeError("HiCache storage clear failed after updating weights")
```

Add a scheduler helper instead of directly reaching into `tree_cache` from the updater:

```python
def clear_hicache_storage_after_weight_update(self) -> bool:
    if not self.enable_hierarchical_cache:
        return True
    if not self.enable_hicache_storage:
        return True
    if not hasattr(self.tree_cache, "clear_storage_backend"):
        logger.error("HiCache storage is enabled but tree cache cannot clear storage backend.")
        return False

    ok = self.tree_cache.clear_storage_backend()
    if ok:
        logger.info("HiCache storage cleared after online weight update.")
    else:
        logger.error("Failed to clear HiCache storage after online weight update.")
    return ok
```

Rationale:

- `flush_cache()` already requires `Scheduler.is_fully_idle()`.
- `is_fully_idle()` also checks HiCache in-flight operations:
  - `ongoing_write_through`
  - `ongoing_load_back`
  - `ongoing_prefetch`
  - `ongoing_backup`
- Clearing storage after `flush_cache()` therefore happens after active requests and HiCache asynchronous operations have drained.
- A failed L3 clear is correctness-unsafe. The update path should fail loudly instead of silently serving with stale persistent KV.

The storage clear should happen only when:

- `recv_req.flush_cache` is true,
- hierarchical cache is enabled,
- HiCache storage is currently enabled.

When storage is not enabled, the helper returns success without doing work.

### Phase 1 Backend Hardening

The storage clear path needs two small robustness fixes.

First, propagate the storage backend clear result in `HybridCacheController.clear_storage_backend()`.

Current behavior:

```python
self.storage_backend.clear()
return True
```

Planned behavior:

```python
return bool(self.storage_backend.clear())
```

If the backend raises, keep the existing exception propagation or catch at the scheduler helper. The important part is that a backend returning `False` must not be reported as success.

Second, make `HiCacheFile.clear()` idempotent under multi-rank clearing.

The current file backend removes all files in the storage directory. In TP/PP/CP deployments, multiple ranks may clear the same directory nearly concurrently. That should not turn a correct clear into a false failure because another rank removed the file first.

Planned behavior:

- Ignore `FileNotFoundError` while unlinking files.
- Continue clearing remaining files when a single file disappears concurrently.
- Clear the file evictor metadata after file deletion.
- Return `False` only for real errors that prevent the clear from completing.

This hardening also benefits the existing admin endpoint `/hicache/storage-backend/clear`.

## Phase 2: Cache Namespace Design

Phase 1 is the fastest correctness fix, but it is destructive: it clears the whole configured storage backend. A more scalable design is to make the persistent cache namespace change whenever weights change.

Phase 2 should introduce a model cache namespace, also called a cache generation or weight epoch.

Requirements:

- The namespace must change after every successful online weight update.
- It must not depend only on user-provided `weight_version`; if `weight_version` is absent, SGLang must create an internal monotonic generation.
- It must be part of the L3 storage key, not only metadata.
- Old namespace entries must become unreachable immediately.
- Cleanup of old namespace entries can be asynchronous or explicit.

Possible namespace sources:

- If `recv_req.weight_version` is provided, use a stable digest of it.
- If absent, increment an internal scheduler-side generation such as `kv_cache_generation`.
- Include model identity and parallelism as today.
- Include rank-specific suffixes where needed for non-MLA, PP, and CP layouts.

Possible implementation options:

1. Add the namespace to `HiCacheStorageConfig` and let each backend include it in key construction.
2. Add the namespace to `HiCacheStorageExtraInfo` and require all get/set paths to include it.
3. Add the namespace to `RadixKey.hash_page()` or a storage-specific page-hash wrapper.

The safest long-term option is a storage-specific namespace field rather than overloading token hashes. That keeps in-memory matching and persistent storage versioning conceptually separate.

However, while designing Phase 2, inspect `RadixKey.extra_key` carefully. Today `extra_key` helps isolate in-memory cache matching, but `RadixKey.hash_page()` does not include it. If L3 storage keys are token-hash-only, then other isolation dimensions such as request `cache_salt`, LoRA identity, or future adapters may have similar persistent-cache collision risks. Phase 2 should define exactly which isolation dimensions belong in the storage namespace.

## Failure Semantics

For Phase 1, if worker-side weight update succeeds but cache flush or storage clear fails, the server is already in a risky partial state: weights may have changed while stale cache remains.

The least surprising behavior is:

- Log the failure at error level with the backend name.
- Return failure from the update request if the current output type allows it.
- If the helper currently uses assertions, keep the same failure style for consistency in the first patch.
- Do not continue silently after a failed storage clear.

If follow-up API refinement is desired, change `flush_cache_after_weight_update()` from `None`/assertions to an explicit `(success, message)` result and let each update path return a structured failure message. That can be done after the correctness fix.

Expected behavior:

- The implementation should not clear HiCache storage automatically.
- Existing semantics remain unchanged.

## Operational Notes

- Phase 1 may be expensive for large distributed storage backends because it clears the configured backend namespace. This is acceptable for correctness but should be documented in release notes.
- In shared storage deployments, users should avoid sharing one raw HiCache namespace across unrelated model deployments. This is already best practice and becomes more important when destructive clear is tied to online weight update.
- Phase 2 namespace support is the preferred production-grade design for frequent online weight updates.
- The admin endpoint `/hicache/storage-backend/clear` remains useful as an explicit operator action and as a manual recovery path.

- Call it from `SchedulerWeightUpdaterManager.flush_cache_after_weight_update()` after successful `flush_cache()`.

- Make storage-enabled clear failure fail the update path loudly.
- Propagate backend `clear()` return values from `HybridCacheController.clear_storage_backend()`.
- Make `HiCacheFile.clear()` tolerant of concurrent file deletion.
- Re-run the E2E validation flow and confirm request B no longer reloads storage KV from old weights.

Phase 2:

- Add a persistent cache namespace or generation.
