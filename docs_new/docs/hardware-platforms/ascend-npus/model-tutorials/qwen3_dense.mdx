## Introduction

Qwen3-Dense models are the dense architecture variants of the Qwen3 series developed by Alibaba, featuring Grouped-Query
Attention (GQA) with up to 256K context length. The series includes Qwen3-32B, Qwen3-14B, and Qwen3-8B, delivering
significant improvements in instruction following, logical reasoning, text comprehension, mathematics, science, coding,
and tool usage. These models support EAGLE3 speculative decoding for accelerated inference and are available in both
standard and thinking/reasoning-enhanced editions.

This document demonstrates the deployment of Qwen3-Dense models (Qwen3-32B, Qwen3-14B, Qwen3-8B) on Ascend NPUs using
SGLang, including single-node PD mixed mode, feature configuration, and performance optimization.

This document is validated and written based on **SGLang v0.5.10**. The current models (Qwen3-32B, Qwen3-14B, Qwen3-8B)
are fully supported in this version. To use the latest features (e.g., speculative decoding), it is recommended to use
v0.5.10 or a later version.

## Supported features

| Feature                       | Usage                                                                                         |
|-------------------------------|-----------------------------------------------------------------------------------------------|
| Tensor Parallelism            | `--tp-size 4`                                                                                 |
| Quantization                  | `--quantization modelslim`                                                                    |
| Chunked Prefill               | `--chunked-prefill-size -1 --max-prefill-tokens 16384`                                        |
| NPU Graph                     | `--cuda-graph-bs-decode 1 4 8 16`                                                                    |
| Speculative Decoding          | `--speculative-algorithm EAGLE3 --speculative-draft-model-path <draft-model-path> --speculative-num-steps 3 --speculative-eagle-topk 1 --speculative-num-draft-tokens 4 --speculative-draft-model-quantization unquant` |
| Overlap Schedule              | `export SGLANG_ENABLE_SPEC_V2=1 SGLANG_ENABLE_OVERLAP_PLAN_STREAM=1`                          |

For feature compatibility and conflict information between features,
see [Feature Compatibility](/docs/hardware-platforms/ascend-npus/ascend_npu_optimization#feature-compatibility).

## Prerequisites

### Model weights

<Warning>
If you need to download model weights, check the model size before downloading to reserve enough space.
</Warning>

- [Qwen3-32B](https://huggingface.co/Qwen/Qwen3-32B) (BF16)
- [Qwen3-14B](https://huggingface.co/Qwen/Qwen3-14B) (BF16)
- [Qwen3-8B](https://huggingface.co/Qwen/Qwen3-8B) (BF16)

Ensure the available device memory exceeds the model weight size before deployment. For optimal throughput and latency,
refer to the [best practice configurations](#best-practices) which may require additional cards.

It is recommended to download the model weights to a shared directory across multiple nodes.

## Online service deployment

### Single-node online deployment (Qwen3-32B)

Single-node deployment completes both prefill and decode within the same node (PD mixed mode), suitable for scenarios
with limited hardware resources. The following command is based on the Qwen3-32B model with W8A8 quantization.

```bash Launch server
#============================================================
# Before running, update the following variables:
#   MODEL_PATH: path to the model weights directory
#   DRAFT_MODEL_PATH: path to the speculative draft model weights directory
#============================================================

echo performance | tee /sys/devices/system/cpu/cpu*/cpufreq/scaling_governor
sysctl -w vm.swappiness=0
sysctl -w kernel.numa_balancing=0
sysctl -w kernel.sched_migration_cost_ns=50000

export SGLANG_SET_CPU_AFFINITY=1

unset https_proxy
unset http_proxy
unset HTTPS_PROXY
unset HTTP_PROXY
unset ASCEND_LAUNCH_BLOCKING
source /usr/local/Ascend/ascend-toolkit/set_env.sh
source /usr/local/Ascend/nnal/atb/set_env.sh
source /usr/local/Ascend/ascend-toolkit/latest/opp/vendors/customize/bin/set_env.bash
export PATH=/usr/local/Ascend/8.5.0/compiler/bishengir/bin:$PATH

export PYTORCH_NPU_ALLOC_CONF=expandable_segments:True

MODEL_PATH=/path/to/model-weights
DRAFT_MODEL_PATH=/path/to/draft-model-weights

export SGLANG_DISAGGREGATION_BOOTSTRAP_TIMEOUT=600

export HCCL_SOCKET_IFNAME=lo
export GLOO_SOCKET_IFNAME=lo
export HCCL_OP_EXPANSION_MODE="AIV"
export SGLANG_ENABLE_OVERLAP_PLAN_STREAM=1
export SGLANG_ENABLE_SPEC_V2=1

python -m sglang.launch_server --model-path $MODEL_PATH \
    --host 127.0.0.1 --port 7339 --trust-remote-code --nnodes 1 --node-rank 0  \
    --attention-backend ascend --device npu --quantization modelslim \
    --max-running-requests 16 \
    --disable-radix-cache \
    --speculative-draft-model-quantization unquant \
    --speculative-algorithm EAGLE3 --speculative-draft-model-path $DRAFT_MODEL_PATH \
    --speculative-num-steps 3 --speculative-eagle-topk 1 --speculative-num-draft-tokens 4 \
    --chunked-prefill-size -1 --max-prefill-tokens 16384  \
    --tp-size 4 --mem-fraction-static 0.843 --cuda-graph-bs-decode 1 4 8 16 --dtype bfloat16
```

For common issues, please refer to the [Ascend NPU FAQ](/docs/hardware-platforms/ascend-npus/ascend_npu_faq) for troubleshooting.

## Functional verification

After the service is started, you can invoke the model by sending a prompt:

```shell
curl http://localhost:7339/generate \
    -H "Content-Type: application/json" \
    -d '{
        "text": "What is the capital of France?",
        "sampling_params": {
            "max_new_tokens": 64,
            "temperature": 0
        }
    }'
```

Expected result: an HTTP 200 response with the generated text containing "Paris".

Once the server prints `The server is fired up and ready to roll!` in the logs, it is ready to accept requests. For more
testing examples (Health Check, Generate, Chat Completions, and port usage guidance),
see [Testing the Service](/docs/hardware-platforms/ascend-npus/ascend_npu#testing-the-service).

#### Scenario overview

| Scenario                      | Deployment mode | Total NPUs | Weight version |
|-------------------------------|-----------------|------------|----------------|
| Low latency (6K+1.5K)         | PD mixed        | 4 (A3)     | BF16           |
| Low latency (4K+1.5K)         | PD mixed        | 4 (A3)     | BF16           |
| Low latency (18K+4K)          | PD mixed        | 8 (A3)     | BF16           |
| Low latency (1K+0.3K)         | PD mixed        | 2 (A3)     | W8A8      |
| Low latency (6K+1.5K)         | PD mixed        | 2 (A3)     | W8A8      |
| Low latency (6K+1.5K)         | PD mixed        | 8 (A2)     | W8A8      |
| Low latency (4K+1.5K)         | PD mixed        | 8 (A2)     | BF16           |
| Low latency (1K+0.3K)         | PD mixed        | 1 (A3)     | W8A8      |
| Low latency (6K+1.5K)         | PD mixed        | 1 (A3)     | W8A8      |
| Low latency (3.5K+1.5K)       | PD mixed        | 1 (A3)     | W8A8      |
| High throughput (3.5K+1.5K)   | PD mixed        | 2 (A3)     | W8A8      |
| High throughput (2K+2K)       | PD mixed        | 2 (A3)     | W8A8      |
| High throughput (3.5K+1.5K)   | PD mixed        | 8 (A2)     | W8A8      |
| High throughput (2K+2K)       | PD mixed        | 8 (A2)     | W8A8      |
| High throughput (3.5K+1.5K)   | PD mixed        | 1 (A3)     | W8A8      |

`Total NPUs` indicates the total number of NPUs used across all nodes.
