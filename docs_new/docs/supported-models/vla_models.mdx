Vision-Language-Action (VLA) models accept multi-camera images, a language
instruction, and (optionally) robot proprioceptive state, and output a chunk
of **continuous robot actions** rather than text tokens. They are typically
deployed in robotics for manipulation, navigation, and other embodied tasks.

Unlike standard generative LLMs, VLA models in SGLang produce their entire
action chunk during prefill тАФ there is no decode loop, and `/generate`
returns a top-level `actions` field (not a token stream).

## Supported models

| Model Family (Variants) | Example HuggingFace Identifier | Description |
|-------------------------|-------------------------------|-------------|
| **╧А0** (`Pi0ForActionPrediction`) | `lerobot/pi0_base` | Vision-Language-Action flow-matching model from [Physical Intelligence](https://www.physicalintelligence.company/). PaliGemma backbone (SigLIP + Gemma 2B) + Gemma 300M action expert. Predicts a 50-timestep ├Ч 32-dim action chunk via 10-step Euler-integrated flow matching; supports up to 3 camera views (base, left wrist, right wrist). |

## ╧А0 (Pi-Zero)

### Architecture

- **PaliGemma backbone** тАФ SigLIP vision encoder (So400m/14) + Gemma 2B language model.
- **Gemma 300M action expert** тАФ Processes robot state and noisy actions with attention shared across the prefix.
- **Flow-matching head** тАФ Iteratively denoises from Gaussian noise to actions over a configurable number of Euler steps (default: 10).

### Quickstart

#### 1. Download the Model Weights

The ╧А0 checkpoint is published on the HuggingFace Hub at
[`lerobot/pi0_base`](https://huggingface.co/lerobot/pi0_base):

```bash Download Weights
pip install "huggingface_hub[cli]"
hf download lerobot/pi0_base --local-dir /path/to/pi0
```

#### 2. Patch `config.json`

`lerobot/pi0_base` ships a config shaped for LeRobot's own loader. SGLang
recognizes ╧А0 via the `model_type: "pi0"` tag and the
`Pi0ForActionPrediction` architecture, so edit the downloaded
`config.json` (merge these keys тАФ keep everything else as-is):

```jsonc config.json
{
  "model_type": "pi0",
  "architectures": ["Pi0ForActionPrediction"],
  "auto_map": {
    "AutoConfig": "sglang.srt.models.pi0--Pi0Config"
  }
}
```

#### 3. Install the PaliGemma Tokenizer

╧А0 uses the PaliGemma-3B tokenizer, which is **not** bundled with the
`lerobot/pi0_base` checkpoint. Pull just the tokenizer assets from the
upstream PaliGemma repo and drop them into the model directory.

<Note>
`google/paligemma-3b-pt-224` is gated тАФ you need a HuggingFace token with
the PaliGemma license accepted.
</Note>

```bash Install Tokenizer
export HF_TOKEN=hf_your_token_here
pip install huggingface_hub

python -c "
from huggingface_hub import snapshot_download
snapshot_download(
    'google/paligemma-3b-pt-224',
    allow_patterns=['tokenizer*', 'special_tokens_map*'],
    local_dir='/path/to/pi0',
)
"
```

Then patch `tokenizer_config.json` to enable the leading BOS token тАФ the ╧А0
prompt tokenization contract (see [`Pi0NewLineProcessor`](https://github.com/huggingface/lerobot/blob/main/src/lerobot/policies/pi0/processor_pi0.py)
in LeRobot) appends a trailing `\n` **and** expects a leading BOS:

```jsonc tokenizer_config.json
{
  "add_bos_token": true
}
```

#### 4. Launch the Server

```bash Launch Server
CUDA_VISIBLE_DEVICES=0 python -m sglang.launch_server \
    --model-path /path/to/pi0 \
    --port 8890 \
    --dtype float32
```

<Note>
`--dtype float32` is the default for the reference ╧А0 checkpoint; bfloat16
parity has not been validated end-to-end yet.
</Note>

### Sending Requests

#### Raw `/generate`

```python Python
import requests
import base64

# Encode your camera images as base64
with open("camera_base.jpg", "rb") as f:
    base_img = base64.b64encode(f.read()).decode()
with open("camera_left_wrist.jpg", "rb") as f:
    left_img = base64.b64encode(f.read()).decode()
with open("camera_right_wrist.jpg", "rb") as f:
    right_img = base64.b64encode(f.read()).decode()

response = requests.post(
    "http://localhost:8890/generate",
    json={
        "text": "pick up the red block",
        "image_data": [
            f"data:image/jpeg;base64,{base_img}",
            f"data:image/jpeg;base64,{left_img}",
            f"data:image/jpeg;base64,{right_img}",
        ],
        "sampling_params": {
            "max_new_tokens": 1,  # ╧А0 generates actions, not tokens
        },
        "extra_body": {
            "state": [0.1, 0.2, ...],         # 32-dim robot proprioceptive state
            "num_inference_steps": 10,        # denoising steps (optional, default=10)
        },
    },
)

# Response contains continuous actions
actions = response.json()["actions"]  # shape: (50, 32)
```

### Response Schema

VLA responses are **not** token streams. The `/generate` endpoint returns a
single JSON document with the following shape:

```jsonc Response
{
  "text": "",                        // always empty for VLA тАФ do not try to parse
  "output_ids": [],                  // always empty for VLA
  "actions": [[...], [...], ...],    // (action_horizon, action_dim) nested list
  "meta_info": {
    "id": "...",
    "finish_reason": {"type": "stop", "matched": "vla_done"},
    "prompt_tokens": N,
    "completion_tokens": 0,          // VLA emits no tokens
    ...
  }
}
```

The stable client-facing contract is:

- **Read `response.json()["actions"]`** тАФ a `(action_horizon, action_dim)` nested list.
- Don't parse `text`; don't rely on `completion_tokens`.
- `meta_info.finish_reason.matched == "vla_done"` is the explicit signal that this was a VLA response.

### Normalization Contract

╧А0 consumes **normalized** state and emits **normalized** actions by default.
SGLang supports two deployment modes:

<AccordionGroup>
  <Accordion title="Client-side normalization (default)">
    If `config.json` does **not** set `norm_stats`, SGLang logs an INFO message
    at load time and passes state / actions through unchanged. The client must
    pre-normalize the state vector with the training-dataset statistics and
    un-normalize the returned actions the same way. This is the most common
    setup with LeRobot datasets.
  </Accordion>
  <Accordion title="Server-side normalization">
    Add a `norm_stats` block to `config.json` and SGLang will apply it
    transparently (forward on the state input, inverse on the action output):

    ```json config.json
    {
      "model_type": "pi0",
      "norm_stats": {
        "state":  {"mode": "mean_std", "mean": [...], "std": [...]},
        "action": {"mode": "mean_std", "mean": [...], "std": [...]}
      }
    }
    ```

    `mode` may be `"mean_std"` or `"min_max"` (matches LeRobot's
    `NormalizerProcessorStep` semantics). Vector length can be less than
    `max_state_dim` / `max_action_dim` тАФ only the leading entries are
    transformed; the padded tail passes through unchanged.
  </Accordion>
</AccordionGroup>

### Configuration

Key configuration parameters (from `config.json`):

| Parameter | Default | Description |
|-----------|---------|-------------|
| `chunk_size` | 50 | Number of action timesteps per chunk |
| `max_action_dim` | 32 | Action vector dimension |
| `max_state_dim` | 32 | State vector dimension |
| `num_inference_steps` | 10 | Number of flow-matching denoising steps |
| `image_resolution` | `(224, 224)` | Input image resolution |
| `paligemma_variant` | `gemma_2b` | VLM backbone variant |
| `action_expert_variant` | `gemma_300m` | Action expert variant |

### How It Works

#### Inference Pipeline

1. **Image encoding**: Each camera image is resized to 224├Ч224 (with
   aspect-ratio-preserving padding) and encoded by SigLIP into 256 visual
   tokens.
2. **Language encoding**: The instruction is tokenized (max 48 tokens) and
   embedded by Gemma 2B's embedding layer, scaled by тИЪ(hidden_dim).
3. **Prefix KV cache**: Image + language tokens are concatenated and forwarded
   through the PaliGemma backbone to produce KV cache.
4. **Denoising loop**: Starting from Gaussian noise xтВБ, for each step:
    - Embed state + noisy actions + timestep тЖТ suffix tokens
    - Forward suffix through action expert with prefix KV cache
    - Euler step: `x_t = x_t + dt ├Ч v_t`
5. **Output**: The denoised xтВА is the predicted action chunk
   (`chunk_size ├Ч max_action_dim`, default 50 ├Ч 32).

#### Attention Pattern

```text
                 Images  Language  State  Actions
Images           тЬУ       тЬУ         тЬЧ      тЬЧ
Language         тЬУ       тЬУ         тЬЧ      тЬЧ
State            тЬУ       тЬУ         тЬУ      тЬЧ
Actions          тЬУ       тЬУ         тЬУ      тЬУ (causal)
```

- Prefix tokens (images + language): bidirectional attention.
- State token: attends to the prefix, but the prefix cannot attend to it.
- Action tokens: causal attention among themselves; attend to everything above.
