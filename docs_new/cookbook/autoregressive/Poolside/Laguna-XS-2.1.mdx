Pick your hardware + quantization + strategy to generate the launch command. Laguna-XS-2.1 ships **two operating points** per cell:

- **Low-Latency (DFlash)** — speculative decoding with a matched-precision DFlash draft. The draft proposes a block, the target verifies it in one forward pass (measured accept-length ranges ~3–7 depending on precision and parallelism — see per-cell numbers below), cutting per-token latency for interactive / few-stream serving. Accuracy-neutral (verification-based).
- **High-Throughput** — plain serving. At batch-saturated load the GPU is compute-bound and speculation's draft + rejected-token overhead costs more than it saves, so the dense path wins on aggregate tokens/sec.

On the 8-GPU HGX platforms (H200 / B300), BF16 uses plain `--tp 8`. The **FP8 and INT4 checkpoints use `--tp 8 --ep-size 8`** instead — plain `--tp 8` fails at weight load because their quantization scale granularity cannot shard the 512-dim MoE 8-way, so expert-parallelism is used to still saturate all 8 GPUs on one instance (FP8 additionally needs `SGLANG_SHARED_EXPERT_TP1=1`; see [Configuration Tips](#2-configuration-tips)). The 4-GPU Grace-Blackwell GB300 node uses plain `--tp 4` throughout (4 GPUs is already inside the plain-TP ceiling).

## 1. Model Introduction

[Laguna-XS-2.1](https://huggingface.co/poolside/Laguna-XS-2.1) is an open-weight **33B-parameter** hybrid sliding-window-attention Mixture-of-Experts model (**~3B activated per token**) from [poolside](https://poolside.ai), built for agentic coding and long-horizon software-engineering work — the extra-small sibling of [Laguna-M.1](./Laguna-M.1).

**Key Features:**

- **Sparse MoE**: 40-layer transformer with **256 routed experts, top-8 routing**.
- **Hybrid sliding-window attention**: 30 sliding-window layers (window 512) interleaved with 10 full-attention layers; 48 Q-heads / 8 KV-heads (head dim 128).
- **Long context**: 262,144 tokens, RoPE with YaRN on the full-attention layers.
- **DFlash speculative decoding**: poolside ships **matched-precision DFlash draft models** (5-layer, ~0.9 GB) calibrated per quantization for low-latency serving.
- **Hybrid reasoning**: `<think>…</think>` segments toggled per request via `chat_template_kwargs={"enable_thinking": …}`.

## 2. Configuration Tips

- **Trust remote code** (`--trust-remote-code`): Laguna-XS-2.1 ships custom modeling/config code on the Hugging Face Hub, so this flag is required.
- **Attention backend — leave it alone for dense, pin it for DFlash.** Laguna is hybrid-SWA and backend-sensitive:
  - **High-Throughput (dense)**: do **not** pass `--attention-backend`. Auto-select resolves the correct kernel per platform (`fa3` on Hopper, `trtllm_mha` on Blackwell).
  - **Low-Latency (DFlash)**: with a speculative algorithm active, auto-select falls back to **flashinfer**, which is broken for this model at `tp ≥ 4` on Blackwell (greedy GSM8K collapses 76% → 28%). The generated Low-Latency commands therefore pin the target backend explicitly (`trtllm_mha` on Blackwell, `fa3` on H200). The DFlash *draft* worker is fine on its automatic fallback — leave `--speculative-draft-attention-backend` unset.
  - **Never** use `--attention-backend triton` with Laguna — its SWA handling is broken for this model (GSM8K 13%).
- **DFlash memory**: the Low-Latency cells carry `--mem-fraction-static 0.7`. At `tp 4` on GB300 the default fraction OOMs in the draft vocab all-gather (`Failed to CUDA calloc`); 0.7 is validated. Dense cells use the default heuristic.
- **INT4 is mixed-precision**: the INT4 checkpoint quantizes MoE layers in *mixed* 4-bit / 8-bit config groups (regex targets, no `"Linear"` group). Builds older than [PR #29761](https://github.com/sgl-project/sglang/pull/29761) crash at load with `KeyError: 'Linear'` — use a build at or past its merge.
- **Chat template**: on transformers ≥ 5.10 the standalone `chat_template.jinja` auto-loads — no `--chat-template` flag needed (the server logs `Auto-detected template features: reasoning_parser=poolside_v1, tool_call_parser=poolside_v1` when it resolves correctly). On older pinned transformers (≤ ~5.8) the `{% include %}` stub in `tokenizer_config.json` cannot resolve and the server silently falls back to a generic template — there, pass `--chat-template <model-dir>/chat_template.jinja` explicitly.
- **Thinking default**: thinking is **off by default**; opt in per request with `extra_body={"chat_template_kwargs": {"enable_thinking": True}}`. The template gates on `enable_thinking` — the generic `thinking` key is ignored.
- **Served model id**: the server registers the model under whatever you pass to `--model-path`, so a client's `model` field must match it — `poolside/Laguna-XS-2.1` (BF16) or the `-FP8` / `-NVFP4` / `-INT4` id for the quantized cells.
- **TP degree**: 48 Q-heads / 8 KV-heads shard cleanly at `tp` 1 / 2 / 4 / 8, but the **FP8 and INT4 checkpoints cap plain TP at 4**: `moe_intermediate_size=512` with FP8 block `[128,128]` / INT4 `group_size=128` scales cannot shard 8-way (512/8 = 64 violates the 128-granularity — FP8 fails at weight creation, INT4 crashes in the Marlin kernel; reproduced on 8×H200, and the checks are hardware-independent). Rather than give up 4 of 8 GPUs, the generated H200/B300 commands for FP8 and INT4 use `--tp 8 --ep-size 8` — expert parallelism keeps whole experts per rank, sidestepping the shard-granularity wall while still using every GPU on a single instance (FP8 additionally needs `SGLANG_SHARED_EXPERT_TP1=1`, since its shared expert is also block-quantized; INT4's stays bf16). Other options: plain `--tp 4` (uses only 4 of 8 GPUs), or `--tp 4 --dp-size 2` (two replicas across all 8). Accuracy is parallelism-independent within noise (verified `tp 1` ≡ `tp 4` on GB300; on H200 full-set GSM8K, `tp 4` vs `tp 8 + ep 8` differ by ≤1pt on every combination — an initial ~2pt gap on INT4+DFlash didn't reproduce on a repeat run, confirming ordinary eval noise rather than a parallelism effect).

## 3. Advanced Usage

### 3.1 DFlash Speculative Decoding (Low-Latency strategy)

DFlash is a block-wise speculative decoder: the 5-layer draft proposes a block of tokens, the target model verifies the whole block in a single forward pass, and only target-approved tokens are emitted — so **output quality is the target's, by construction** (measured: GSM8K matches dense within noise on every quantization). The speedup lever is **accept-length** — how many draft tokens survive verification per target step:

- ~**6** tokens/step at `tp 1`, ~**4** at plain `tp 4` (measured on GB300 and 8×H200, matched-precision pairs; ~3 under mixed reasoning-heavy traffic) — versus 1 token/step dense. Under `--tp 8 --ep-size 8` on H200 (greedy GSM8K), FP8 measured ~6.75 and INT4 ~5–5.6 across repeat runs — comparable to or higher than plain-TP figures, though these numbers came from a different eval workload than the tp4/tp1 figures above, so treat exact cross-config comparisons loosely.
- Best for **interactive / few-stream** serving where the GPU has idle compute. Under batch-saturated load prefer the High-Throughput cells: speculation's draft + rejected-token overhead costs aggregate throughput once the GPU is compute-bound.
- **Match the draft to the target precision** (the generated commands do) — each draft is calibrated against its specific quantized target.

### 3.2 Reasoning

Launch with `--reasoning-parser poolside_v1` (baked into every generated command). Reasoning is **opt-in**: the Laguna chat template gates it on `enable_thinking=True` via `chat_template_kwargs`. The `<think>` trace lands in `message.reasoning_content`, separate from the final answer in `message.content`.

<Accordion title="Reasoning Example (Python)">

```python Example
from openai import OpenAI

client = OpenAI(base_url="http://localhost:30000/v1", api_key="EMPTY")

response = client.chat.completions.create(
    model="poolside/Laguna-XS-2.1",
    messages=[{"role": "user", "content": "What is 15% of 240? Explain briefly."}],
    max_tokens=2048,
    extra_body={"chat_template_kwargs": {"enable_thinking": True}},
)

message = response.choices[0].message
print("=============== Reasoning ===============")
print(message.reasoning_content)
print("=============== Answer ==================")
print(message.content)
```

</Accordion>

<Note>
XS-2.1 is an extra-small model — give it generous `max_tokens` when thinking is enabled
(AIME-class problems regularly reason for thousands of tokens), and keep thinking **off**
for short-form tasks where it isn't needed.
</Note>

### 3.3 Tool Calling

Launch with `--tool-call-parser poolside_v1` (baked into every generated command). The parser converts Laguna's `<tool_call>` output into the standard OpenAI `tool_calls` structure. Tool calling works with reasoning off (the default).

<Accordion title="Tool Calling Example (Python)">

```python Example
from openai import OpenAI

client = OpenAI(base_url="http://localhost:30000/v1", api_key="EMPTY")

tools = [
    {
        "type": "function",
        "function": {
            "name": "get_weather",
            "description": "Get the current weather for a location",
            "parameters": {
                "type": "object",
                "properties": {
                    "location": {"type": "string", "description": "The city name"},
                    "unit": {"type": "string", "enum": ["celsius", "fahrenheit"]},
                },
                "required": ["location"],
            },
        },
    }
]

response = client.chat.completions.create(
    model="poolside/Laguna-XS-2.1",
    messages=[{"role": "user", "content": "What's the weather in Beijing?"}],
    tools=tools,
)

message = response.choices[0].message
if message.tool_calls:
    for call in message.tool_calls:
        print(f"Tool: {call.function.name}")
        print(f"Args: {call.function.arguments}")
```
