Omni models can understand and generate across modalities such as text, image, audio, and video. Unlike a pure VLM or a pure diffusion model, an omni model may alternate between text reasoning and media generation in the same request or across multiple conversation turns.

In SGLang, the omni runtime uses an orchestrator layer to coordinate specialized workers:

- `srt` owns autoregressive text generation, chat state, and session KV cache.
- `multimodal_gen` owns image, audio, and video generation backends.
- `omni` owns the request/session orchestration that connects the pieces.

The HTTP surface is split by task:

- Use ordinary `/generate` or OpenAI-compatible `/v1/chat/completions` for VLM-style understanding when the model only needs to answer with text.
- Use `/v1/omni/generate` for media generation, image editing, and interleaved responses. This is SGLang's native segment API, not an OpenAI Chat Completions-compatible response shape.
- Use `/v1/responses` with the `image_generation` tool when you want an OpenAI-style non-streaming adapter for image-capable omni models.
- Use `/v1/omni/sessions/{session_id}/close` to release a persistent omni session after multi-turn work.

An interleaved response is an ordered sequence of output segments, for example text, image, then more text. With `keep_session: true`, the server returns a `session.id`; later requests can pass that `session_id` so the model can refer to previous turns and generated images.

## Deployment Modes

`sglang serve` can launch the standard SRT server, an omni SRT deployment, or the standalone diffusion server. `auto` is a dispatch choice, not a hybrid deployment: if a model is detected as diffusion, SGLang starts a separate `multimodal_gen` diffusion server instance and does not mount the SRT chat or omni session routes. Use `--model-type` when the model can match more than one deployment path.

| Deployment | Command selector | Use when |
| --- | --- | --- |
| SRT / LLM server | `--model-type llm` | Text generation and VLM chat |
| Omni SRT deployment | `--model-type omni` | Omni models that need SRT-owned chat state, sessions, KV cache, and colocated generation stages |
| Standalone diffusion server | `--model-type diffusion` | Pure image/video generation models served directly by `multimodal_gen` |
| Auto detection | `--model-type auto` or unset | Simple cases where the model path clearly belongs to only one server type |

Omni models that interleave text and media should use the omni SRT deployment. The omni route is mounted into the SRT process, and `multimodal_gen` can be used as a colocated generation backend from inside the omni orchestrator. Future omni models that can run generation independently can use a standalone diffusion serving backend while keeping the same orchestrator boundary.

## Engine Boundaries

The omni path is a runtime composition, not a replacement for the text or diffusion engines.

| Component | Process boundary | Responsibility |
| --- | --- | --- |
| SRT engine | SRT server process tree | HTTP serving, chat/OpenAI APIs, tokenization path, AR decoding, SRT sessions, and KV cache |
| Omni orchestrator | Mounted inside the SRT deployment for current sessionful models | Normalize omni requests, drive the decode/generate loop, maintain ordered segments, and commit generated media back into the live session |
| Diffusion / `multimodal_gen` engine | Colocated backend for current U1-style models, or standalone server for pure diffusion models | Image/audio/video generation stages, denoising, media decoding, and generation-specific parallel execution |

For SenseNova U1 today, the diffusion-style generation path is colocated with the SRT model runner because generated image segments need live model/session/KV handles. That is different from `--model-type diffusion`, which starts a complete standalone diffusion serving instance with its own HTTP server and scheduler. The omni backend interface keeps those options separate so future models can use a remote or standalone diffusion runtime when they do not need live SRT handles.

## Omni HTTP API

The omni HTTP API is shared by model cookbooks. Model-specific pages focus on supported tasks and sampling parameters, while the common request/response shape lives here.

| Task | Endpoint |
| --- | --- |
| Generate or edit media, or run interleaved generation | `POST /v1/omni/generate` |
| OpenAI-style non-streaming image generation adapter | `POST /v1/responses` with `tools: [{"type": "image_generation"}]` |
| Close a persistent omni session | `POST /v1/omni/sessions/{session_id}/close` |
| Text-only VLM understanding | Use ordinary `/generate` or OpenAI-compatible `/v1/chat/completions` |

### Request Shape

The request body is segment-based. `messages` contains ordered multimodal input segments; `sampling_params` is interpreted by the selected model plugin.

```json Example
{
  "model": "<omni-model>",
  "task": "interleave",
  "messages": [
    {"type": "text", "text": "Generate one image, then describe it."}
  ],
  "sampling_params": {
    "width": 1024,
    "height": 1024,
    "num_steps": 50
  },
  "max_images": 1,
  "max_text_segments": 2
}
```

Supported input segment fields:

| Segment type | Payload field | Notes |
| --- | --- | --- |
| `text` | `text` or `content` | User instructions, chat text, or edit prompts |
| `image` | `image` | Data URL, base64 payload, or model-supported image object |
| `audio` | `audio` | Reserved for audio-capable omni models |
| `video` | `video` | Reserved for video-capable omni models |

### Response Shape

The response contains ordered output segments. A model can return text, media, or both in the same response.

```json Output
{
  "segments": [
    {"type": "text", "text": "Here is the image."},
    {
      "type": "image",
      "image": {
        "b64_json": "...",
        "mime_type": "image/png"
      }
    }
  ],
  "stats": {},
  "metadata": {}
}
```

### Persistent Sessions

Set `keep_session: true` to keep the SRT-owned session and KV cache alive after a response. The server returns `session.id`; later turns pass that value as `session_id`.

```json Example
{
  "model": "<omni-model>",
  "task": "interleave",
  "keep_session": true,
  "messages": [
    {"type": "text", "text": "Create an image I can edit in the next turn."}
  ]
}
```

After the workflow is complete, close the session:

```bash Command
curl -X POST http://localhost:30000/v1/omni/sessions/<session_id>/close
```

### Responses Adapter

The `/v1/responses` adapter is useful for clients that already speak the OpenAI Responses shape. The MVP is non-streaming and maps the `image_generation` tool to the native omni orchestrator.

```json Example
{
  "model": "sensenova-u1",
  "input": [
    {
      "role": "user",
      "content": [
        {"type": "input_text", "text": "Create one image, then describe it."}
      ]
    }
  ],
  "tools": [
    {"type": "image_generation", "size": "1024x1024", "action": "auto"}
  ],
  "metadata": {
    "sampling_params": {"num_steps": 50},
    "max_images": 1,
    "max_text_segments": 2
  }
}
```

The adapter returns standard Responses output items: `message` for text segments and `image_generation_call` for generated images. Use `previous_response_id` to continue a stored omni session in a later turn.
