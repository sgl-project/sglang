A U1-style request is closer to a transfusion-like interleaving loop than a plain VLM call:

1. The AR path formats the conversation, runs prefill, and decodes text until it reaches an image-generation boundary token.
2. The generation backend runs the native pixel-flow denoising stages for that boundary.
3. The generated image is returned as an output segment and its model context is committed back into the live session.
4. Later AR continuation or later turns can refer to the generated image through the same session context.

This is why SGLang keeps the responsibilities split:

- `srt` handles chat formatting, autoregressive text decoding, sessions, and KV cache.
- `multimodal_gen` provides the colocated stage pipeline for generated image segments.
- `sglang.omni` contains the engine-agnostic coordinator, request/response types, HTTP entrypoints, model adapters, and backend adapters.
- `sglang.srt.omni_session` contains SRT-owned session execution: native SRT `Req` construction, session/KV binding capture, and temporary context forward when media generation needs attention over live SRT KV.

For U1, `omni` is mounted into the SRT deployment. It is not a separate FastAPI server, and it does not start a complete standalone diffusion serving instance. The generation backend runs colocated with the SRT model runner so it can use the live U1 model, session state, and KV-backed context. This is the current execution boundary for sessionful interleaved models; the same orchestrator protocol can later call a standalone diffusion serving runtime for models that do not need live SRT handles.

Use the ordinary SGLang HTTP APIs when U1 is acting like a VLM and only needs to return text:

- `/generate`
- OpenAI-compatible `/v1/chat/completions`

Use the omni HTTP API when U1 needs to generate or edit media:

- `POST /v1/omni/generate`
- `POST /v1/responses` with `tools: [{"type": "image_generation"}]`
- `POST /v1/omni/sessions/{session_id}/close`

`/v1/omni/generate` is a native SGLang segment API rather than an OpenAI Chat Completions-compatible response shape. OpenAI Chat-style paths remain text-output paths. Use `/v1/responses` when you want an OpenAI-style non-streaming adapter for image generation or editing; it maps `image_generation` tool calls to the same omni orchestrator.

An interleaved request can produce multiple ordered output segments, such as text, image, then text. A multi-turn interleaved conversation keeps the same omni session alive with `keep_session: true`; the next turn passes `session_id`, so the model can refer to previous text and generated images.

Supported U1 tasks:

| Task | Aliases | Use case |
| --- | --- | --- |
| `vlm` | `chat`, `vlm_chat` | Text answer from visual input through the omni path |
| `t2i` | `text_to_image`, `txt2img` | Text-to-image generation |
| `edit` | `image_edit`, `img2img` | Image editing with a source image |
| `interleave` | `interleaved` | Ordered text and image segments in one request/session |

## 2. Installation

Use a development build that includes native SenseNova U1 and the SGLang omni runtime.

```bash Command
git clone https://github.com/sgl-project/sglang.git
cd sglang
pip install -e "python[all,diffusion]"
```

Install any CUDA, PyTorch, or FlashInfer packages required by your target environment before starting the server.

## 3. Model Deployment

Start an omni SGLang server with the U1 model path. The omni routes are exposed from the same SRT deployment, and U1 image segments are executed colocated with the SRT model runner so they can reuse the live model/session/KV state.

```bash Command
sglang serve \
  --model-type omni \
  --model-path sensenova/SenseNova-U1-8B-MoT \
  --host 0.0.0.0 \
  --port 30000
```

`--model-type omni` is intentional. `sglang serve` also supports `--model-type diffusion` for standalone SGLang-Diffusion deployments. The default `auto` mode is a dispatch choice, not a hybrid deployment; with diffusion support installed, it may route registered generation models to a standalone `multimodal_gen` diffusion server instance. SenseNova U1 needs the SRT deployment because omni sessions, chat state, KV cache, `/generate`, `/v1/chat/completions`, and `/v1/omni/generate` all live in the SRT model execution path. For U1, `multimodal_gen` is used as a colocated pipeline/stage backend, not as a separate diffusion server.

For larger GPUs, add the same runtime flags you normally use for SRT, such as `--tp-size`, memory settings, or CUDA graph settings. `--diffusion-server-args` can pass generation-engine settings for omni backends that consume them, but the current U1 colocated path does not use standalone/disaggregated diffusion serving flags such as `--enable-cfg-parallel`: U1 image generation borrows live SRT model/session/KV state and runs through a colocated `multimodal_gen` pipeline.

Runtime topology for U1:

| Layer | Where it runs | Role |
| --- | --- | --- |
| SRT HTTP/API layer | SRT deployment | Serves `/generate`, OpenAI chat, and `/v1/omni/generate` |
| SRT scheduler/model runner | SRT deployment | Owns AR decoding, session lifecycle, and KV cache |
| Omni orchestrator | Inside the SRT deployment | Calls SRT for text boundaries, calls generation backend for image boundaries, and returns ordered segments |
| U1 generation backend | Colocated `multimodal_gen` pipeline/stages | Runs native pixel-flow image generation and commits generated image context back to SRT |
| Standalone diffusion server | Not used for U1 in this path | Used by pure diffusion deployments through `--model-type diffusion` |

Code layout:

| Area | Contents |
| --- | --- |
| `sglang.omni.core` | Generic request/response objects, interleaved segment helpers, and the thin coordinator loop |
| `sglang.omni.backends.ar` | AR backend boundary; the current implementation adapts omni orchestration to SRT sessions |
| `sglang.omni.backends.mm_gen` | Media-generation backend boundary; U1 currently uses a direct colocated pipeline-forward backend, while an executor-backed backend can drive a future standalone runtime |
| `sglang.omni.model_adapters.sensenova_u1` | U1-specific prompt grammar, image boundary token handling, thinking mode, CFG condition paths, and generated-image commit rules |
| `sglang.omni.entrypoints` | `/v1/omni/generate`, `/v1/responses`, streaming serialization, and session-close HTTP wiring |
| `sglang.omni.runtime` | Serve-time transport and scheduler handoff state for omni endpoints; it does not own SRT KV operations |
| `sglang.srt.omni_session` | SRT-side session runtime that owns native SRT request execution, session lifecycle, KV bindings, and temporary context forward |
| `sglang.multimodal_gen` | Native image/video generation stages, sampling parameters, and diffusion-style pipelines |

## 4. Usage

SenseNova U1 supports familiar SGLang text/VLM APIs, the native omni endpoint, and a non-streaming Responses adapter. Use the standard APIs when the response is text-only; use `/v1/omni/generate` when the response can include generated images or interleaved segments. Use `/v1/responses` when an OpenAI-style output item shape is more convenient. The common omni request/response schema is described in the [Omni overview](/cookbook/omni/intro).

| Workflow | Endpoint | Key fields | Output |
| --- | --- | --- | --- |
| Text-only generation | `/generate` | `text`, `sampling_params` | Text |
| OpenAI-compatible VLM chat | `/v1/chat/completions` | `messages[].content` with `image_url` and `text` | Text |
| Native VLM understanding | `/generate` | `text`, `image_data`, `sampling_params` | Text |
| Text-to-image | `/v1/omni/generate` | `task: "t2i"`, text segment, image sampling params | Image segment |
| Image editing | `/v1/omni/generate` | `task: "edit"`, image segment, text edit instruction | Image segment |
| Interleaved multi-turn generation | `/v1/omni/generate` | `task: "interleave"`, `keep_session` or `session_id` | Ordered text/image segments |
| Responses image generation adapter | `/v1/responses` | `tools: [{"type": "image_generation"}]`, `input`, optional `previous_response_id` | `message` and `image_generation_call` output items |

For single-turn text-to-image or image editing, only the `task` and `messages` fields change. For example, `task: "t2i"` uses a text segment as input, while `task: "edit"` uses an image segment plus a text instruction.

`/v1/omni/generate` supports both JSON and streaming responses. Set `stream: true`
to receive Server-Sent Events: text arrives as `text_delta` events, image
generation emits `image_start` so clients can render a placeholder, and the final
`image` event carries the generated payload.

```bash Command
curl -N http://localhost:30000/v1/omni/generate \
  -H "Content-Type: application/json" \
  -d '{
    "model": "sensenova-u1",
    "task": "interleave",
    "stream": true,
    "messages": [
      {"type": "text", "text": "Create one image, then describe it."}
    ],
    "sampling_params": {"width": 1024, "height": 1024, "num_steps": 50},
    "max_images": 1,
    "max_text_segments": 2
  }'
```

The `/v1/responses` adapter is a non-streaming MVP. It supports text input, base64/data URL image input, `size`, `action`, metadata sampling overrides, and `previous_response_id` for multi-turn continuation. Streaming, background mode, file IDs, and remote image URLs are not covered by this first adapter. When the adapter stores an omni session, the response metadata includes `omni_session_id`; close it with `/v1/omni/sessions/{session_id}/close` when the workflow is complete.

```python Example
import base64
import requests

url = "http://localhost:30000/v1/responses"

turn1 = {
    "model": "sensenova-u1",
    "input": [
        {
            "role": "user",
            "content": [
                {
                    "type": "input_text",
                    "text": (
                        "Create a clean studio image of one matte cobalt-blue "
                        "cube centered on a white tabletop, then describe it."
                    ),
                }
            ],
        }
    ],
    "tools": [
        {"type": "image_generation", "size": "1024x1024", "action": "auto"}
    ],
    "metadata": {
        "sampling_params": {
            "num_steps": 50,
            "cfg_text_scale": 4.0,
            "cfg_interval": [0.0, 1.0],
            "cfg_renorm_type": "none",
            "t_eps": 0.02,
        },
        "max_images": 1,
        "max_text_segments": 2,
    },
}

r1 = requests.post(url, json=turn1)
r1.raise_for_status()
data1 = r1.json()

turn2 = {
    "model": "sensenova-u1",
    "previous_response_id": data1["id"],
    "input": [
        {
            "role": "user",
            "content": [
                {
                    "type": "input_text",
                    "text": (
                        "Using the previous image, make the cube bright red "
                        "and add one glossy green sphere to its right."
                    ),
                }
            ],
        }
    ],
    "tools": [
        {"type": "image_generation", "size": "1024x1024", "action": "auto"}
    ],
    "metadata": {
        "sampling_params": {
            "num_steps": 50,
            "cfg_text_scale": 4.0,
            "cfg_img_scale": 1.0,
            "cfg_interval": [0.0, 1.0],
            "cfg_renorm_type": "none",
            "t_eps": 0.02,
        },
        "max_images": 1,
        "max_text_segments": 2,
    },
}

r2 = requests.post(url, json=turn2)
r2.raise_for_status()
data2 = r2.json()

for item in data2["output"]:
    if item["type"] == "message":
        for content in item["content"]:
            if content["type"] == "output_text":
                print(content["text"])
    elif item["type"] == "image_generation_call":
        with open(f'{item["id"]}.png', "wb") as f:
            f.write(base64.b64decode(item["result"]))
```

## 5. Developer WebUI

SGLang also includes a small local WebUI for validating interleaved omni sessions. Start the U1 server first, then run the UI against the same API base:

```bash Command
python -m sglang.omni.webui.server \
  --host 127.0.0.1 \
  --port 7860 \
  --api-base http://127.0.0.1:30000
```

Open `http://127.0.0.1:7860`, click `Preset: first image`, send the first turn, then click `Preset: edit previous` and send again. The UI streams text deltas, shows an image placeholder while generation is running, and keeps the returned `session.id` so turn 2 can refer to and edit the image generated in turn 1.

The WebUI is a local development proxy for `/v1/omni/generate`; do not expose it directly to an untrusted network.

## 6. Multi-Turn Interleave

Set `keep_session: true` on the first turn. The server returns `session.id`; pass that value as `session_id` on later turns.

```python Example
import base64
import requests

url = "http://localhost:30000/v1/omni/generate"

turn1 = {
    "model": "sensenova-u1",
    "task": "interleave",
    "keep_session": True,
    "messages": [
        {
            "type": "text",
            "text": (
                "Create a clean studio image of one matte cobalt-blue cube centered "
                "on a white tabletop, with a plain light-gray background. After the "
                "image, briefly describe the cube color and the table."
            ),
        }
    ],
    "sampling_params": {
        "width": 1024,
        "height": 1024,
        "num_steps": 50,
        "seed": 223,
        "cfg_text_scale": 4.0,
        "cfg_interval": [0.0, 1.0],
        "cfg_renorm_type": "none",
        "t_eps": 0.02,
    },
    "max_images": 1,
    "max_text_segments": 2,
}

r1 = requests.post(url, json=turn1)
r1.raise_for_status()
data1 = r1.json()
session_id = data1["session"]["id"]

turn2 = {
    "model": "sensenova-u1",
    "task": "interleave",
    "session_id": session_id,
    "messages": [
        {
            "type": "text",
            "text": (
                "Using the previous generated image as the reference, edit it so the "
                "cube becomes bright red and add one glossy green sphere to its "
                "right. Keep the same tabletop, camera angle, and background."
            ),
        }
    ],
    "sampling_params": {
        "width": 1024,
        "height": 1024,
        "num_steps": 50,
        "seed": 224,
        "cfg_text_scale": 4.0,
        "cfg_img_scale": 1.0,
        "cfg_interval": [0.0, 1.0],
        "cfg_renorm_type": "none",
        "t_eps": 0.02,
    },
    "max_images": 1,
    "max_text_segments": 2,
}

r2 = requests.post(url, json=turn2)
r2.raise_for_status()
data2 = r2.json()

for index, segment in enumerate(data2["segments"]):
    if segment["type"] == "text":
        print(segment["text"])
    elif segment["type"] == "image":
        with open(f"u1_turn2_{index}.png", "wb") as f:
            f.write(base64.b64decode(segment["image"]["b64_json"]))

requests.post(f"http://localhost:30000/v1/omni/sessions/{session_id}/close")
```

This is the path that demonstrates interleave behavior: the generated image from turn 1 is committed into the live session, and turn 2 can refer to it as "the previous image".

## 7. Sampling Parameters

Common U1 image sampling parameters:

| Parameter | Default | Notes |
| --- | --- | --- |
| `width` | `1024` | Image width; must be divisible by 32 |
| `height` | `1024` | Image height; must be divisible by 32 |
| `num_inference_steps` | `50` | Also accepts `num_steps` or `steps` |
| `cfg_text_scale` | `4.0` | Also accepts `guidance_scale` or `text_cfg_scale` |
| `cfg_img_scale` | `1.0` | Also accepts `img_cfg_scale` or `image_cfg_scale` |
| `cfg_interval` | `[0.0, 1.0]` | CFG active interval |
| `cfg_renorm_type` | `none` | One of `none`, `global`, `channel`, `text_channel` |
| `timestep_shift` | `3.0` | Pixel-flow timestep shift |
| `t_eps` | `0.02` | Clamp epsilon for the final pixel-flow velocity step |

Omni control fields:

| Field | Default | Notes |
| --- | --- | --- |
| `task` | `interleave` | Request-level task selector; can also be passed inside `sampling_params`. `mode` is accepted as a legacy alias. |
| `max_images` | `1` | Upper bound for generated image segments |
| `max_text_segments` | `8` | Upper bound for generated text segments |
| `stream` | `false` | Return `/v1/omni/generate` as SSE events instead of one JSON object |
| `keep_session` | `false` | Keeps the SRT session/KV alive for later turns |
| `session_id` | unset | Continue an existing session |
| `think` | `false` | Enables model-specific thinking behavior when supported |
| `think_max_new_tokens` | unset | Thinking token budget |

## 8. Scope Notes

- Standard `/generate` and `/v1/chat/completions` are intended for one-image VLM understanding and text output.
- Image generation, editing, and interleaved text/image output should use `/v1/omni/generate`.
- Close persistent sessions after use so the server can release the session KV cache.
- For quick smoke tests, use smaller multiples of 32 and fewer steps; use `1024x1024` or other validated model resolutions for quality checks.
