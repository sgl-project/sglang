# KV Transfer Bandwidth And Background Traffic Experiment Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Measure KV Cache transfer latency versus data size under different RDMA/TCP bandwidth profiles and stable background traffic levels, then fit each curve with a latency model.

**Architecture:** Keep the current SGLang/Mooncake GPU-buffer benchmark as the foreground KV transfer workload. Add an operational matrix around it: first discover RDMA/TCP topology, then run foreground transfers with selected HCA sets, then add stable background traffic generated by RDMA perftest or TCP iperf3, and finally fit `latency_ms = alpha_ms + bytes / bandwidth` for every condition. The first benchmark pair remains `099 -> 102`; spare nodes `100` and `101` are used as background generators when needed.

**Tech Stack:** SGLang MooncakeTransferEngine, Docker `sglang-pd-switch:tianciJ`, RDMA perftest tools (`ib_write_bw`/equivalent), TCP `iperf3`, shell scripts, Python CSV fitter.

## Experiment Concepts

Observed RDMA topology:

```text
mlx5_0       Rate 200, mapped to eth0; validated for Mooncake foreground KV
mlx5_bond_0  Rate 200, mapped to bond0; validated for ib_write_bw background only
mlx5_bond_1  Rate 200, mapped to bond1; validated for topology, foreground pending
mlx5_bond_2  Rate 200, mapped to bond2; validated for topology, foreground pending
mlx5_bond_3  Rate 200, mapped to bond3; validated for topology, foreground pending
```

Requested RDMA bandwidth profiles on this hardware:

```text
rdma_800_4x200: 4 lanes, each 200G
rdma_800_2x400: 2 logical groups, each group made from 2 physical 200G lanes
rdma_400_2x200: 2 lanes, each 200G
rdma_400_1x400: 1 logical group made from 2 physical 200G lanes
rdma_400_4x100: 4 lanes, each shaped to 100G
tcp_200: TCP transport, expected total 200G if iperf3 is available
```

Important interpretation:

```text
The cluster does not expose a literal 400G RDMA device in ibstat output.
Therefore "1x400" and "2x400" must be implemented as logical groups of 2x200 lanes.
The current single-process foreground benchmark can pass a comma-separated Mooncake device string, but it cannot by itself distinguish "2x400" from "4x200" grouping semantics.
To distinguish those profiles rigorously, use either Mooncake's verified multi-NIC striping behavior or a parallel sharded foreground runner that splits one logical KV transfer across lane groups and records max-lane latency.
```

Background traffic levels:

```text
0%: no active background traffic
1%: background consumes 1% of the profile bandwidth
10%: background consumes 10% of the profile bandwidth
50%: background consumes 50% of the profile bandwidth
90%: background consumes 90% of the profile bandwidth
```

Foreground sweep for every condition:

```text
SIZES=1MB:2GB:x2
WARMUP=3
REPEAT=20
```

Fit model:

```text
latency_ms = alpha_ms + bytes / effective_bandwidth_bytes_per_ms
```

Fit using `latency_ms_p50` and large sizes `>=64MiB` by default. Report `alpha_ms`, `effective_GiBps`, and `R2`.

- [ ] **Step 4.3: Decide how to realize `rdma_800_2x400` and `rdma_400_1x400`**

```text
Because the cluster exposes 200G physical lanes, "400G" is a logical group of two 200G lanes.
If Mooncake multi-device striping is the measurement method, rdma_400_1x400 and rdma_400_2x200 are not distinguishable by the current single-process script: both use two 200G devices.
If the distinction matters, implement a parallel sharded runner:
  rdma_800_2x400: 2 shard processes, each shard uses 2 physical 200G devices.
  rdma_800_4x200: 4 shard processes, each shard uses 1 physical 200G device.
  rdma_400_1x400: 1 shard process using 2 physical 200G devices.
  rdma_400_2x200: 2 shard processes, each shard uses 1 physical 200G device.
```

- [ ] **Step 4.4: Decide how to realize `rdma_400_4x100`**

```text
Mooncake foreground transfer does not currently expose a per-lane bandwidth limiter.
Therefore rdma_400_4x100 cannot be a hard foreground cap with the current script.
Approximate options:
  A. Generate 100G competing ib_write_bw traffic on each of four 200G lanes and label the result "background-approximated".
  B. Implement application-level throttling in a sharded foreground runner by chunking transfers and sleeping to target 100G per lane.
  C. Use NIC/QoS configuration if the cluster exposes a safe per-port rate limiter.
```
