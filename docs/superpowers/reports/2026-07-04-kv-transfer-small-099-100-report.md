# KV Transfer 099->100 小流量 RDMA 限速阶段报告

**日期:** 2026-07-04

**目标:** 记录 `lingjun-099 -> lingjun-100` 在小流量区间 `1MiB..32MiB`、步长 `1MiB` 下，机头 RDMA 与机尾 RDMA 在不限速、`4*100`、`2*200` 口径下的 Mooncake KV transfer 延迟和有效带宽；本轮不注入背景流量。

## 技术摘要

- 本轮 6 个条件全部完成，每个条件均有完整 32 行逻辑 size 汇总，`error_sum=0`。
- 机头 RDMA 的不限速 4 路均分在 32MiB 达到 `65.381767GiB/s` p50；加总目标同为 400Gbps 的 `4*100` 与 `2*200` 后，32MiB p50 分别为 `42.909733GiB/s` 与 `38.964844GiB/s`，明显低于不限速但接近 400Gbps 量级。
- 机尾 RDMA single NIC 的不限速与 cap200 在 32MiB 基本重合，分别为 `17.662163GiB/s` 与 `17.611289GiB/s`，说明该路径当前主要受 single NIC 或 tail path 上限约束；cap100 将 32MiB p50 压到 `11.400148GiB/s`。
- 小流量下 1MiB 点受固定开销影响更明显，带宽排序不完全代表大尺寸稳定带宽；本报告以 8/16/32MiB 的趋势作为主要解释依据。

## 限制、鲁棒性与解释边界

1. 本轮没有背景流量，因此只能解释前景小流量在不同拆分和限速口径下的表现，不能直接外推到 bg1/bg10/bg50/bg90。
2. 限速使用 `kv_transfer_latency.py --rate-limit-gbps` 的应用层 pacing，`chunk_size=16MB`；它不是 NIC QoS 或交换机侧硬限速。小于或接近 chunk size 的传输会更容易受固定开销、计时粒度和 sleep/pacing 策略影响。
3. 机头 RDMA 与机尾 RDMA 的实现语义不同：机头为 `mlx5_bond_*` 多路 split；机尾沿用历史 `tail_single_nic`，即 `mlx5_0` single NIC。两者比较反映本轮配置下的路径差异，不是完全对称的 head/tail 多 HCA 对照。
4. 聚合结果使用同一 iteration 下各 shard latency 的最大值作为逻辑传输 latency；如果某个 lane 偶发变慢，会直接影响该 iteration 的逻辑 p50/p90。
5. 本轮只覆盖 `lingjun-099 -> lingjun-100` 一个方向、一个时间窗口。跨节点、反方向或繁忙时段需要单独复测。
