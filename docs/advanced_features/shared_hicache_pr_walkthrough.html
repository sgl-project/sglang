Mental Model

1Plan is the contract The external router gives SGLang a shared_hicache_plan. The plan is not a static peer map. It says: "for this request, this target worker may reuse this prefix from this source worker at this HTTP endpoint, and the reusable medium is HiCache CPU."
2HiCache stays the owner of cache semantics The feature does not invent a new cache store. It indexes existing host-tier HiCache nodes and inserts transferred pages through a HiCache-facing primitive: insert_shared_hicache_device_blocks.
3Mooncake is only the transport Mooncake is reused for memory registration and transfer writes. It does not own the plan, prefix logic, node lifetime, or cache hit accounting.
4Scheduler fails open If the plan is bad, stale, unsupported, or the direct transfer is not usable, the request continues with ordinary local prefill. Only indeterminate direct-transfer timeouts quarantine target pages. 
Router/Dynamo
  builds SharedHiCachePlan from KV events and selected source endpoint
      |
      v
TokenizerManager / Engine
  carries shared_hicache_plan as opaque request metadata
      |
      v
Scheduler
  probes local HiCache prefix, then asks SharedHiCacheManager to stage missing prefix pages
      |
      v
Target worker HTTP client -> source worker HTTP server
  source resolves protected host pages and Mooncake writes them into target KV pages
      |
      v
HiRadixCache
  target inserts the newly written device pages as an ordinary prefix cache node
Worker id purpose. shared_hicache_worker_id is the stable routing identity that lets a plan prove it was meant for this SGLang worker. It is deliberately separate from process-local rank internals. The manager rejects plans whose target worker id or source worker id do not match the current target/source role.

End-to-End Flow

1. Request metadata Engine.generate, async_generate, and GenerateReqInput carry shared_hicache_plan.
2. Scheduler gate Scheduler calls _prepare_shared_hicache_for_schedule before final req.init_next_round_input.
3. Target staging Manager validates the plan, allocates target KV pages, and submits a direct-transfer RPC.
4. Source transfer Source resolves HiCache host pages, protects nodes, and asks Mooncake to write into target GPU pages.
5. Local insertion Target inserts the transferred pages into HiRadix, increments Shared HiCache hit length, and schedules.

State Diagram

no plan
  -> schedule normally

plan present
  -> parse and validate
      -> invalid/stale/wrong target/source: clear plan, schedule normally
      -> already satisfied locally: schedule normally
      -> transfer disabled/unavailable: schedule normally
      -> remote transfer submitted: pending
          -> pending and timeout policy says wait: skip this scheduling pass
          -> complete ok: insert pages, mark hit, schedule
          -> complete deterministic miss/error: free target pages, schedule normally
          -> indeterminate timeout: quarantine target pages, schedule normally

Plan Contract: shared_hicache/plan.py

This file intentionally makes the router-to-SGLang contract narrow. It removes old compatibility aliases except for the current block-hash naming bridge, requires CPU HiCache source medium, and keeps numeric fields strict so bad JSON cannot silently route to the wrong worker or page.

LinesWalkthrough
1-8 Imports only dataclasses, time, typing, StorageMedium, and block_hash_aliases. This keeps the plan parser dependency-light and usable from both manager and control paths.
10-14 Defines the public extra-args key, no-plan reason key, schema version, timeout reason, and SHARED_HICACHE_SOURCE_MEDIUM = CPU_PINNED. The source medium constant is the core native HiCache decision.
17-27 _now_ms and normalize_endpoint. Endpoint normalization accepts host:port, adds http://, and strips a trailing slash so control paths can append /transfer_direct exactly once.
30-35 _canonical_source_medium accepts only CPU_PINNED. This is deliberately not backward compatible with older remote-g2 medium names.
38-63 Strict coercion helpers. bool is rejected even though it is an int subclass, numeric strings are not accepted, block hashes must be integers, and arrays must be concrete sequences.
65-76 Hash alias and first-present helpers. Signed and unsigned int64 block hash aliases are tolerated because JSON and Dynamo/SGLang boundaries can disagree on int64 presentation.
79-96 SharedHiCachePlan dataclass. Important fields: target/source worker ids, target/source DP ranks, source endpoint, CPU source medium, identity block hashes, optional KV block hashes, block size, TTL, and start block index.
99-196 from_dict performs all validation in one place. It requires mapping input, accepts current block_hashes plus a short hashes bridge, clamps planned blocks to available hashes, validates optional kv_block_hashes length, normalizes endpoint, and rejects expired/negative/missing fields later through manager validation.
198-215 Serialization and read helpers: to_dict keeps the same field names, planned_hashes and planned_kv_block_hashes slice to the planned prefix, is_shared_hicache checks medium, and is_expired centralizes TTL checks.
Plan rule: Shared HiCache only stages CPU HiCache pages. It does not claim disk, external object storage, or Mooncake L3 store content as reusable source pages.

Target Orchestrator: shared_hicache/manager.py

The manager is the target-side control loop. Its job is to turn a router plan into local device-prefix pages without blocking the scheduler indefinitely or corrupting the allocator when a transfer result is ambiguous.

Review Hot Spots

Plan schema and naming The public names are now Shared HiCache names: --enable-shared-hicache, shared_hicache_plan, SharedHiCacheManager, and metrics prefixed with shared_hicache.
No static peer map Static endpoint maps are rejected in server_args.py. The source endpoint is carried per plan. That makes the scheduler behavior explicit and router-owned.
HiCache-native primitives The shared feature uses lookup_hicache_host_blocks and insert_shared_hicache_device_blocks rather than building an alternate tree/cache store.
Mooncake reuse boundary Reused: engine init, memory registration, batch transfer. Not reused: Mooncake store APIs, offload manager logic, or storage backend abstractions, because this is CPU HiCache peer reuse, not L3 object fetch.
Timeout safety Direct-transfer timeouts are treated as potentially in-flight writes. The target quarantines pages instead of returning them to the allocator.
Prefix warmup explicitness Prefix warmup copies the plan for the warmup request only. Expanded parallel samples clear the plan so they do not race the same remote staging implicitly.
