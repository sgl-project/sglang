## 📋 两类项目

### A 类：端到端"主路径"项目（从入口到输出）

**目标**：你能画出并解释这条链路，并能改动其中一个关键层而不崩。

#### SGLang 的主链路示例

**基于你的经验，你应该特别关注这些链路**：

**链路 1：LLM 推理服务主链路**
```
Entry / API layer（OpenAI-style 或 internal endpoint）
    ↓
Router / Scheduler（你修过高并发瓶颈的地方）
    ↓
Scheduler / Runtime（batching、pipeline stage、并发/队列）
    ↓
Model wrapper（把 HF/自研模型适配成统一接口）
    ↓
Parallelism（tp/sp/usp, all2all, comm）← 你做过 SP support
    ↓
Kernel & memory（triton/cuda, fused op, offload）
    ↓
Postprocess & output（decode, save, response return）
```

**链路 2：Diffusion / GLM-Image 主链路**（你做过 SP support）
```
Entry / API layer
    ↓
Diffusion Pipeline（你修过 T5 config、stage 对齐）
    ↓
Model wrapper（GLM-Image、T5 encoder）
    ↓
Parallelism（SP support - 你实现的部分）
    ↓
Kernel & memory（4D latents 处理）
    ↓
Postprocess & output（image generation）
```

**链路 3：Benchmark / 回归验证链路**（你做过 TTFT/ITL/p99/VRAM）
```
Benchmark Runner
    ↓
Request Generation（你做的自动化 benchmark）
    ↓
Performance Metrics Collection（TTFT/ITL/p99/VRAM）
    ↓
Regression Detection（你定位过 deadlocks、memory regressions）
    ↓
Report Generation
```

**你至少要完整走通一次，并且能解释每层的职责边界。**

#### 关键检查点

- [ ] 能画出完整的请求处理流程图
- [ ] 能解释每一层的职责和边界
- [ ] 能改动其中一层而不影响其他层
- [ ] 理解层与层之间的接口和数据流

---

### B 类：结构性改动项目（不是修 bug，是改抽象）

**这才是"学会系统设计思维"的关键**：你得做一次会影响多个模块协作方式的改动。

#### 典型的结构性改动

1. **职责重划分**
   - 把某个逻辑从 A 层移动到 B 层
   - 重新定义模块边界

2. **统一接口设计**
   - 引入统一接口（比如 scheme 化、kernel-call split）
   - 抽象化多个实现

3. **可观测性机制**
   - 加入一套可回归/可观测性机制（CI + perf regression harness）
   - 建立质量保证体系

## 🚀 推荐项目选题（基于你的经验）

### 选题 1：基于你 Benchmark 经验的 CI / Regression Test 标准化框架 ⭐⭐⭐ **最推荐**

**目标**：把你做过的 benchmark（TTFT/ITL/p99/VRAM、Qwen benchmark）标准化成可扩展的 CI 框架

#### 为什么这个选题最适合你？

**你已经有了基础**：
- ✅ 你做过自动化 benchmark、压测
- ✅ 你定位过回归（"2 concurrency deadlocks / 3 memory regressions"）
- ✅ 你做过 Qwen benchmark

**现在需要系统化**：
- 把你做过的 benchmark 经验抽象成标准框架
- 建立自动化的回归检测机制
- 设计可扩展的测试体系

#### 这不是小 bug。你要做成它，必须理解：

1. **推理输出生成机制**
   - Pipeline + seed 如何工作（你修过 diffusion pipeline）
   - 如何保证输出一致性
   - **结合你的经验**：你修过 T5 config、stage 对齐，现在需要理解整个 pipeline 的一致性保证

2. **Nondeterministic Source**
   - CUDA、kernel、parallelism 中的不确定性来源
   - **结合你的经验**：你做过 SP support，需要理解 SP 如何影响确定性
   - 如何控制这些不确定性

3. **性能回归标准**
   - **结合你的经验**：你做过 TTFT/ITL/p99/VRAM benchmark
   - 什么叫"可接受的性能回归"（±2% 阈值？）
   - 如何定义回归标准

4. **Regression Gating**
   - 如何做 regression gating（避免性能回退）
   - **结合你的经验**：你定位过 deadlocks、memory regressions，现在需要自动化检测
   - 如何建立自动化测试流程

#### 最终你会得到

- ✅ 对整个系统"从入口到输出"的可控理解
- ✅ 对"工程质量体系"的设计思维（这就是大厂系统能力）
- ✅ 把你已有的 benchmark 经验系统化、标准化
- ✅ PR 容易 merge，且影响面大

#### 相关 Issue

- 查找 SGLang 中与 testing、CI、accuracy 相关的 issue
- 可能涉及：`#15236` 或类似编号（需要确认）
