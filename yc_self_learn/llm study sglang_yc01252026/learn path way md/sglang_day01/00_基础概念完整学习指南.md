## 📖 第一部分：LLM推理基础

### 1.1 Transformer架构

**概念**: Transformer是LLM的基础架构

**知识点分解**:

#### 1.1.1 Self-Attention机制
- [ ] **Attention的定义**: 计算序列中每个位置对其他位置的关注度
- [ ] **Self-Attention的含义**: 序列中的每个位置关注序列中的所有位置（包括自己）
- [ ] **Q, K, V矩阵的来源**: 
  - [ ] Q (Query): 来自输入序列的线性变换
  - [ ] K (Key): 来自输入序列的线性变换
  - [ ] V (Value): 来自输入序列的线性变换
- [ ] **Attention计算公式**: `Attention(Q,K,V) = softmax(QK^T/√d_k)V`
- [ ] **Scaled Dot-Product**: 为什么要除以√d_k（防止softmax饱和）
- [ ] **Attention的输出**: 每个位置的加权value向量

**官方文档**:
- [Attention Is All You Need - Section 3.2.1](https://arxiv.org/abs/1706.03762)
- [The Illustrated Transformer - Attention](https://jalammar.github.io/illustrated-transformer/)

#### 1.1.2 Multi-Head Attention
- [ ] **Multi-Head的定义**: 并行计算多个attention head
- [ ] **Head的数量**: 通常8, 16, 32等（num_heads）
- [ ] **每个Head的维度**: `head_dim = hidden_size / num_heads`
- [ ] **Multi-Head的过程**:
  - [ ] 将Q, K, V分割成num_heads份
  - [ ] 每个head独立计算attention
  - [ ] 将所有head的结果拼接（concatenate）
  - [ ] 通过线性层输出
- [ ] **Multi-Head的优势**: 捕获不同类型的关系（语法、语义等）

**详细解释**: 参考 [00_A3_Attention_Head_详解.md](./00_A3_Attention_Head_详解.md) ⭐ **深入理解**

**官方文档**:
- [Attention Is All You Need - Section 3.2.2](https://arxiv.org/abs/1706.03762)

#### 1.1.3 Feed-Forward Network (FFN)
- [ ] **FFN的定义**: 两层全连接网络
- [ ] **FFN的结构**:
  - [ ] 第一层: `[hidden_size, intermediate_size]`，通常intermediate_size = 4 * hidden_size
  - [ ] 激活函数: GELU或ReLU
  - [ ] 第二层: `[intermediate_size, hidden_size]`
- [ ] **FFN的公式**: `FFN(x) = max(0, xW1 + b1)W2 + b2` (ReLU) 或 `GELU(xW1 + b1)W2 + b2`
- [ ] **FFN的作用**: 对每个位置独立进行非线性变换
- [ ] **FFN的位置**: 在Attention之后，每个Transformer layer中

**详细解释**: 参考 [00_Z8_Feed_Forward_Network_详解.md](./00_Z8_Feed_Forward_Network_详解.md) ⭐ **深入理解**

**官方文档**:
- [Attention Is All You Need - Section 3.3](https://arxiv.org/abs/1706.03762)

#### 1.1.4 Layer Normalization
- [ ] **Layer Norm的定义**: 对每个样本的特征维度进行归一化
- [ ] **Layer Norm的公式**: `LayerNorm(x) = γ * (x - μ) / √(σ² + ε) + β`
  - [ ] μ: 特征的均值
  - [ ] σ²: 特征的方差
  - [ ] γ, β: 可学习的缩放和偏移参数
  - [ ] ε: 防止除零的小常数
- [ ] **Layer Norm的位置**: 
  - [ ] Pre-norm: 在Attention和FFN之前
  - [ ] Post-norm: 在Attention和FFN之后
- [ ] **Layer Norm的作用**: 稳定训练，加速收敛

**官方文档**:
- [Layer Normalization论文](https://arxiv.org/abs/1607.06450)
- [Understanding Layer Normalization](https://pytorch.org/docs/stable/generated/torch.nn.LayerNorm.html)

#### 1.1.5 Positional Encoding
- [ ] **位置编码的作用**: 为序列添加位置信息（因为Self-Attention是位置无关的）
- [ ] **Sinusoidal Positional Encoding**: 
  - [ ] 使用sin和cos函数生成位置编码
  - [ ] 公式: `PE(pos, 2i) = sin(pos / 10000^(2i/d_model))`
  - [ ] 公式: `PE(pos, 2i+1) = cos(pos / 10000^(2i/d_model))`
- [ ] **Learned Positional Embedding**: 可学习的位置嵌入（更常用）
- [ ] **位置编码的添加方式**: 与token embedding相加
- [ ] **相对位置编码**: 关注相对位置而非绝对位置

**官方文档**:
- [Attention Is All You Need - Section 3.5](https://arxiv.org/abs/1706.03762)
- [Positional Encoding详解](https://kazemnejad.com/blog/transformer_architecture_positional_encoding/)

#### 1.1.6 Transformer Layer完整结构
- [ ] **一个Transformer Layer包含**:
  1. [ ] Multi-Head Attention
  2. [ ] Add & Norm (残差连接 + Layer Norm)
  3. [ ] Feed-Forward Network
  4. [ ] Add & Norm (残差连接 + Layer Norm)
- [ ] **残差连接的作用**: 缓解梯度消失，使深层网络可训练
- [ ] **Layer的堆叠**: 通常12, 24, 32, 48层等
- [ ] **输入输出形状**: `[batch_size, seq_len, hidden_size]`

**官方文档**:
- [Attention Is All You Need - Section 3.1](https://arxiv.org/abs/1706.03762)
- [Transformer Architecture Explained](https://huggingface.co/docs/transformers/model_doc/transformer)

**学习时间**: 4-6小时（需要深入理解每个组件）

### 1.2 LLM推理流程

**概念**: LLM如何生成token

**知识点分解**:

#### 1.2.1 Tokenization（分词）
- [ ] **什么是Token**: Token是文本的最小处理单元
- [ ] **Tokenizer的作用**: 将文本转换为token ID序列
- [ ] **常见Tokenizer**: BPE, WordPiece, SentencePiece
- [ ] **Token ID**: 每个token对应一个整数ID
- [ ] **特殊Token**: `<bos>`, `<eos>`, `<pad>`, `<unk>`的含义

**官方文档**:
- [Hugging Face Tokenizers](https://huggingface.co/docs/tokenizers/)
- [BPE算法详解](https://huggingface.co/learn/nlp-course/chapter6/5)

#### 1.2.2 Embedding（嵌入）
- [ ] **Embedding的作用**: 将token ID转换为向量表示
- [ ] **Embedding矩阵**: `[vocab_size, hidden_size]`的权重矩阵
- [ ] **Embedding查找**: 通过token ID索引获取对应的embedding向量
- [ ] **Positional Embedding**: 位置编码的作用和计算方式
- [ ] **输入Embedding**: Token embedding + Positional embedding

**官方文档**:
- [Word Embeddings Explained](https://pytorch.org/tutorials/beginner/nlp/word_embeddings_tutorial.html)
- [Positional Encoding](https://kazemnejad.com/blog/transformer_architecture_positional_encoding/)

#### 1.2.3 Prefill阶段（预填充阶段）
- [ ] **Prefill的定义**: 处理输入prompt的阶段
- [ ] **Prefill的输入**: 完整的prompt token序列
- [ ] **Prefill的过程**: 
  - [ ] 所有token并行处理（可以并行计算）
  - [ ] 计算所有位置的attention
  - [ ] 生成所有位置的hidden states
- [ ] **Prefill的输出**: 最后一层的hidden states（用于生成第一个token）
- [ ] **Prefill的特点**: 计算量大，但可以并行

**官方文档**:
- [LLM Inference Explained - Prefill](https://lilianweng.github.io/posts/2023-01-10-inference-optimization/#prefill-phase)

#### 1.2.4 KV Cache（键值缓存）
- [ ] **KV Cache的定义**: 缓存Attention计算中的K和V矩阵
- [ ] **为什么需要KV Cache**: 
  - [ ] 避免重复计算已处理token的K和V
  - [ ] 在decode阶段，只需要计算新token的K和V
- [ ] **KV Cache的存储**: 
  - [ ] 形状: `[batch_size, num_layers, seq_len, num_heads, head_dim]`
  - [ ] 每个layer都有自己的KV Cache
- [ ] **KV Cache的更新**: 
  - [ ] Prefill阶段: 填充所有位置的K和V
  - [ ] Decode阶段: 追加新token的K和V
- [ ] **KV Cache的内存占用**: 是LLM推理的主要内存消耗

**官方文档**:
- [KV Cache Explained](https://lilianweng.github.io/posts/2023-01-10-inference-optimization/#kv-cache)
- [Understanding KV Cache](https://www.anyscale.com/blog/understanding-kv-cache)

#### 1.2.5 Decode阶段（解码阶段）
- [ ] **Decode的定义**: 逐个生成新token的阶段
- [ ] **Decode的输入**: 
  - [ ] 当前token的embedding（单个token）
  - [ ] 之前所有token的KV Cache
- [ ] **Decode的过程**:
  - [ ] 计算当前token的Q, K, V
  - [ ] 使用KV Cache和当前K, V计算attention
  - [ ] 通过FFN生成hidden state
  - [ ] 通过LM Head生成logits
  - [ ] 采样下一个token
- [ ] **Decode的特点**: 
  - [ ] 串行处理（必须等上一个token生成完）
  - [ ] 计算量相对较小
  - [ ] 延迟敏感

**官方文档**:
- [LLM Inference Explained - Decode](https://lilianweng.github.io/posts/2023-01-10-inference-optimization/#decode-phase)

#### 1.2.6 Attention计算（在Decode阶段）
- [ ] **Q, K, V的计算**: 
  - [ ] Q: Query矩阵，来自当前token
  - [ ] K: Key矩阵，来自KV Cache + 当前token
  - [ ] V: Value矩阵，来自KV Cache + 当前token
- [ ] **Attention计算步骤**:
  - [ ] `scores = Q @ K^T / sqrt(d_k)`
  - [ ] `attention_weights = softmax(scores)`
  - [ ] `output = attention_weights @ V`
- [ ] **Causal Mask**: 确保只能看到之前的token
- [ ] **Attention的输出**: 当前token的context-aware表示

**官方文档**:
- [Attention Mechanism](https://lilianweng.github.io/posts/2018-06-24-attention/)

#### 1.2.7 LM Head和Logits
- [ ] **LM Head的定义**: 最后的线性层，将hidden state映射到vocab空间
- [ ] **LM Head的形状**: `[hidden_size, vocab_size]`
- [ ] **Logits的计算**: `logits = hidden_state @ LM_Head_weight`
- [ ] **Logits的含义**: 每个token的未归一化概率分数
- [ ] **Logits的形状**: `[batch_size, vocab_size]`

**官方文档**:
- [Language Model Head](https://huggingface.co/docs/transformers/model_doc/auto#transformers.AutoModelForCausalLM)

#### 1.2.8 Token采样（Sampling）
- [ ] **Sampling的作用**: 从logits中选择下一个token
- [ ] **Greedy Sampling**: 选择概率最高的token
- [ ] **Temperature Sampling**: 使用temperature参数控制随机性
- [ ] **Top-K Sampling**: 只从top-k个token中选择
- [ ] **Top-P (Nucleus) Sampling**: 只从累积概率达到p的token中选择
- [ ] **Sampling的输出**: 下一个token ID

**官方文档**:
- [Text Generation Strategies](https://huggingface.co/docs/transformers/generation_strategies)
- [SGLang Sampling Params](https://docs.sglang.ai/basic_usage/sampling_params.html)

#### 1.2.9 Token生成循环
- [ ] **循环的起点**: Prefill阶段完成后
- [ ] **循环的步骤**:
  1. [ ] 使用当前token ID生成embedding
  2. [ ] 计算Q, K, V（K和V追加到KV Cache）
  3. [ ] 计算attention（使用所有KV Cache）
  4. [ ] 通过FFN和LM Head生成logits
  5. [ ] 采样下一个token ID
  6. [ ] 如果达到max_tokens或遇到eos_token，停止
  7. [ ] 否则，回到步骤1
- [ ] **循环的终止条件**:
  - [ ] 达到最大生成长度（max_new_tokens）
  - [ ] 生成eos_token（结束token）
  - [ ] 用户停止请求

**官方文档**:
- [LLM Inference Loop](https://lilianweng.github.io/posts/2023-01-10-inference-optimization/)

#### 1.2.10 完整推理流程总结
- [ ] **步骤1**: Tokenization - 文本 → token IDs
- [ ] **步骤2**: Embedding - token IDs → embeddings
- [ ] **步骤3**: Prefill - 处理所有prompt tokens，生成KV Cache
- [ ] **步骤4**: Decode循环:
  - [ ] 当前token → Q, K, V
  - [ ] Attention计算（使用KV Cache）
  - [ ] FFN处理
  - [ ] LM Head生成logits
  - [ ] 采样下一个token
  - [ ] 更新KV Cache
  - [ ] 重复直到结束
- [ ] **步骤5**: Detokenization - token IDs → 文本

**官方文档**:
- [LLM Inference Explained - Complete Flow](https://lilianweng.github.io/posts/2023-01-10-inference-optimization/)

**学习时间**: 3-4小时（需要深入理解每个步骤）

### 1.3 Attention机制详解

**概念**: Attention如何计算

**知识点分解**:

#### 1.3.1 Q, K, V矩阵
- [ ] **Q (Query)的含义**: "我要找什么"
- [ ] **K (Key)的含义**: "我是什么"
- [ ] **V (Value)的含义**: "我的实际内容"
- [ ] **Q, K, V的计算**: 
  - [ ] `Q = X @ W_q`，其中W_q是query权重矩阵
  - [ ] `K = X @ W_k`，其中W_k是key权重矩阵
  - [ ] `V = X @ W_v`，其中W_v是value权重矩阵
- [ ] **Q, K, V的形状**: `[batch_size, seq_len, head_dim]`（单个head）
- [ ] **权重矩阵的形状**: `[hidden_size, head_dim]`（每个head）

**官方文档**:
- [Attention Is All You Need - Section 3.2.1](https://arxiv.org/abs/1706.03762)

#### 1.3.2 Attention计算公式详解
- [ ] **步骤1: 计算相似度**: `scores = Q @ K^T`
  - [ ] Q形状: `[batch_size, seq_len_q, head_dim]`
  - [ ] K^T形状: `[batch_size, head_dim, seq_len_k]`
  - [ ] scores形状: `[batch_size, seq_len_q, seq_len_k]`
- [ ] **步骤2: 缩放**: `scores = scores / √d_k`
  - [ ] d_k是head_dim
  - [ ] 为什么要缩放: 防止点积值过大导致softmax饱和
- [ ] **步骤3: 应用Mask（可选）**: 
  - [ ] Causal mask: 下三角矩阵，防止看到未来token
  - [ ] Padding mask: 忽略padding位置
- [ ] **步骤4: Softmax**: `attention_weights = softmax(scores)`
  - [ ] 对最后一个维度做softmax
  - [ ] 每行的权重和为1
- [ ] **步骤5: 加权求和**: `output = attention_weights @ V`
  - [ ] attention_weights形状: `[batch_size, seq_len_q, seq_len_k]`
  - [ ] V形状: `[batch_size, seq_len_k, head_dim]`
  - [ ] output形状: `[batch_size, seq_len_q, head_dim]`

**官方文档**:
- [Attention Is All You Need - Section 3.2.1](https://arxiv.org/abs/1706.03762)
- [Attention机制详解](https://lilianweng.github.io/posts/2018-06-24-attention/)

#### 1.3.3 Scaled Dot-Product Attention
- [ ] **为什么叫Scaled**: 因为有除以√d_k的缩放步骤
- [ ] **为什么需要缩放**: 
  - [ ] 当d_k很大时，QK^T的点积值会很大
  - [ ] 大的值会导致softmax的梯度很小（饱和）
  - [ ] 缩放后保持梯度流动
- [ ] **缩放因子的选择**: √d_k是最常用的选择

**详细解释**: 参考 [00_A1_Scaled_Dot_Product_Attention_详解.md](./00_A1_Scaled_Dot_Product_Attention_详解.md) ⭐ **深入理解**

**官方文档**:
- [Attention Is All You Need - Section 3.2.1](https://arxiv.org/abs/1706.03762)

#### 1.3.4 Flash Attention原理
- [ ] **Flash Attention的目标**: 减少内存使用和加速计算
- [ ] **标准Attention的问题**: 
  - [ ] 需要存储完整的attention matrix（O(n²)内存）
  - [ ] 内存访问不高效
- [ ] **Flash Attention的解决方案**:
  - [ ] Tiling: 将Q, K, V分块处理
  - [ ] Online softmax: 在线计算softmax，不需要存储完整矩阵
  - [ ] 重计算: 在backward时重新计算，而不是存储中间结果
- [ ] **Flash Attention的优势**:
  - [ ] 内存复杂度从O(n²)降到O(n)
  - [ ] 更高效的内存访问模式
  - [ ] 更快的计算速度

**官方文档**:
- [Flash Attention论文](https://arxiv.org/abs/2205.14135) ⭐ **性能优化必读**
- [Flash Attention官方实现](https://github.com/Dao-AILab/flash-attention) ⭐ **代码**

**学习时间**: 4-5小时（需要深入理解计算过程）

## 🔄 第四部分：并行计算

### 4.1 Tensor Parallelism (TP)

**概念**: 在多个GPU上分片模型权重

**知识点分解**:

#### 4.1.1 Tensor Parallelism基础
- [ ] **TP的定义**: 将模型的权重矩阵分割到多个GPU上
- [ ] **TP的目的**: 
  - [ ] 减少单个GPU的内存占用
  - [ ] 支持更大的模型
- [ ] **TP的粒度**: 在矩阵的某个维度上分片
- [ ] **TP的通信**: 需要GPU之间的通信来组合结果

**官方文档**:
- [Megatron-LM论文](https://arxiv.org/abs/1909.08053) ⭐ **TP经典论文**

#### 4.1.2 Column Parallelism（列并行）
- [ ] **列并行的定义**: 在输出维度（列）上分片权重矩阵
- [ ] **权重矩阵形状**: `W: [input_size, output_size]`
- [ ] **TP=2时的分片**:
  - [ ] GPU0: `W_0: [input_size, output_size/2]`（前一半列）
  - [ ] GPU1: `W_1: [input_size, output_size/2]`（后一半列）
- [ ] **计算过程**:
  - [ ] 输入X广播到所有GPU（每个GPU都有完整的输入）
  - [ ] GPU0计算: `Y_0 = X @ W_0`
  - [ ] GPU1计算: `Y_1 = X @ W_1`
  - [ ] 输出拼接: `Y = concat([Y_0, Y_1], dim=1)`
- [ ] **通信模式**: AllGather（收集所有GPU的输出）
- [ ] **适用场景**: QKV投影、MLP的gate/up投影

**官方文档**:
- [Megatron-LM - Column Parallel](https://arxiv.org/abs/1909.08053)

#### 4.1.3 Row Parallelism（行并行）
- [ ] **行并行的定义**: 在输入维度（行）上分片权重矩阵
- [ ] **权重矩阵形状**: `W: [input_size, output_size]`
- [ ] **TP=2时的分片**:
  - [ ] GPU0: `W_0: [input_size/2, output_size]`（前一半行）
  - [ ] GPU1: `W_1: [input_size/2, output_size]`（后一半行）
- [ ] **计算过程**:
  - [ ] 输入X分片: `X_0 = X[:, 0:input_size/2]`, `X_1 = X[:, input_size/2:]`
  - [ ] GPU0计算: `Y_0 = X_0 @ W_0`
  - [ ] GPU1计算: `Y_1 = X_1 @ W_1`
  - [ ] 输出求和: `Y = Y_0 + Y_1`（通过AllReduce）
- [ ] **通信模式**: AllReduce（求和所有GPU的输出）
- [ ] **适用场景**: MLP的down投影、输出投影

**官方文档**:
- [Megatron-LM - Row Parallel](https://arxiv.org/abs/1909.08053)
- Issue #17680代码分析（RowParallel在MoE中的应用）

#### 4.1.4 AllReduce操作
- [ ] **AllReduce的定义**: 所有GPU上的数据求和，然后广播结果
- [ ] **AllReduce的过程**:
  - [ ] 每个GPU有部分数据
  - [ ] 所有GPU的数据求和
  - [ ] 每个GPU都得到完整的求和结果
- [ ] **AllReduce的实现**: 
  - [ ] Ring AllReduce: 环形拓扑
  - [ ] Tree AllReduce: 树形拓扑
  - [ ] NCCL AllReduce: NVIDIA的优化实现
- [ ] **AllReduce的通信开销**: O(n) where n是数据大小

**官方文档**:
- [NCCL AllReduce](https://docs.nvidia.com/deeplearning/nccl/user-guide/docs/usage/collectives.html#allreduce)

#### 4.1.5 TP的通信开销
- [ ] **通信频率**: 每个Transformer layer都需要通信
- [ ] **通信量**: 
  - [ ] ColumnParallel: AllGather输出（output_size）
  - [ ] RowParallel: AllReduce输出（output_size）
- [ ] **通信延迟**: 取决于网络带宽和延迟
- [ ] **优化方法**: 
  - [ ] 使用高速互连（NVLink, InfiniBand）
  - [ ] 通信和计算重叠（Pipeline）
  - [ ] 减少通信次数

**官方文档**:
- [Megatron-LM - Communication Analysis](https://arxiv.org/abs/1909.08053)

#### 4.1.6 TP在MoE中的应用（Issue #17680相关）
- [ ] **MoE中的TP**: 每个expert的权重也需要TP分片
- [ ] **w1, w3 (gate/up)**: ColumnParallel
  - [ ] 在输出维度分片
  - [ ] 需要AllGather
- [ ] **w2 (down)**: RowParallel
  - [ ] 在输入维度分片
  - [ ] 需要AllReduce
  - [ ] **这是Issue #17680的问题所在**
- [ ] **权重加载时的分片**: 
  - [ ] 每个TP rank加载不同的权重切片
  - [ ] 需要正确处理边界情况

**官方文档**:
- [SGLang MoE Weight Loading](https://github.com/sgl-project/sglang/blob/main/python/sglang/srt/layers/moe/fused_moe_triton/layer.py)
- Issue #17680代码分析

**学习时间**: 5-6小时（需要深入理解TP原理和Issue #17680的问题）
