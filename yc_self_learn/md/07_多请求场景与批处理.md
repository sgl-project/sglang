# 多请求场景与批处理

## 🤔 什么时候会有多个请求？

在 LLM 服务中，**多个请求同时到达**是非常常见的场景。理解这些场景对于优化性能和资源利用至关重要。

## 📊 常见的多请求场景

### 1. **并发用户访问** 👥

**场景描述**:
```
时间点 T0: 用户A 发送请求 "介绍一下人工智能"
时间点 T0+10ms: 用户B 发送请求 "介绍一下机器学习"
时间点 T0+20ms: 用户C 发送请求 "介绍一下深度学习"
```

**实际情况**:
- Web 应用有多个用户同时使用
- API 服务被多个客户端调用
- 聊天机器人同时处理多个对话

**示例代码**:
```python
# 多个客户端同时发送请求
import asyncio
import aiohttp

async def send_request(session, prompt):
    async with session.post(
        "http://localhost:30000/v1/chat/completions",
        json={"messages": [{"role": "user", "content": prompt}]}
    ) as response:
        return await response.json()

async def main():
    async with aiohttp.ClientSession() as session:
        # 同时发送 10 个请求
        tasks = [
            send_request(session, f"问题 {i}")
            for i in range(10)
        ]
        results = await asyncio.gather(*tasks)
```

### 2. **批量处理任务** 📦

**场景描述**:
```
任务: 处理 1000 条用户评论的情感分析
方式: 将 1000 条评论分批发送给 LLM
```

**实际情况**:
- 数据分析任务
- 批量文本处理
- 批量翻译任务
- 批量内容生成

**示例代码**:
```python
# 批量处理评论
comments = ["评论1", "评论2", "评论3", ..., "评论1000"]

# 每批处理 32 条
batch_size = 32
for i in range(0, len(comments), batch_size):
    batch = comments[i:i+batch_size]
    # 同时发送 32 个请求
    results = await process_batch(batch)
```

### 3. **多轮对话中的并发** 💬

**场景描述**:
```
用户A 第1轮: "你好"
用户A 第2轮: "介绍一下AI"  # 与第1轮共享前缀
用户B 第1轮: "你好"         # 与用户A共享前缀
用户B 第2轮: "介绍一下ML"   # 与用户A第2轮共享前缀
```

**实际情况**:
- 多个用户同时进行多轮对话
- 每个用户的多轮对话可能共享前缀
- RadixAttention 可以大幅加速

### 4. **流式响应中的新请求** 🌊

**场景描述**:
```
T0: 请求1 开始生成（流式输出）
T0+100ms: 请求2 到达（请求1还在生成中）
T0+200ms: 请求3 到达（请求1和2都在处理中）
```

**实际情况**:
- 第一个请求还在生成时，新请求到达
- 需要同时处理多个处于不同阶段的请求
- 这就是**连续批处理（Continuous Batching）**的核心场景

### 5. **API 网关/负载均衡器** ⚖️

**场景描述**:
```
负载均衡器收到多个请求
    ↓
分发到同一个 SGLang 实例
    ↓
SGLang 需要同时处理多个请求
```

**实际情况**:
- 生产环境中的常见架构
- 多个客户端 → 负载均衡器 → SGLang 实例
- 需要高效的批处理能力

## 🔧 SGLang 如何处理多请求？

### 1. **等待队列（Waiting Queue）**

当多个请求到达时，SGLang 使用**等待队列**管理：

```python
# 在 schedule_policy.py 中
class SchedulePolicy:
    def _compute_prefix_matches(
        self, waiting_queue: List[Req], policy: CacheAwarePolicy
    ) -> Set[int]:
        """
        计算等待队列中请求的前缀匹配
        """
        for r in waiting_queue:
            # 为每个请求匹配前缀
            r.prefix_indices, r.last_node = (
                self.tree_cache.match_prefix(...)
            )
```

**工作流程**:
```
请求到达 → 进入等待队列 → 等待调度 → 批处理执行
```

### 2. **批处理（Batching）**

SGLang 将多个请求**组合成批次**一起处理：

#### a) 静态批处理（Static Batching）

```
批次1: [请求1, 请求2, 请求3, ..., 请求32]
批次2: [请求33, 请求34, ..., 请求64]
```

**特点**:
- 批次大小固定
- 等待批次填满才执行
- 简单但可能延迟

#### b) 连续批处理（Continuous Batching）

```
时间 T0: [请求1, 请求2] → 开始执行
时间 T1: 请求3 到达 → 加入批次
时间 T2: 请求1 完成 → 移除，请求4 加入
时间 T3: [请求2, 请求3, 请求4] → 继续执行
```

**特点**:
- 动态调整批次
- 请求可以随时加入/退出
- 更高的 GPU 利用率
- **SGLang 使用这种方式**

### 3. **前缀匹配优化**

当多个请求有相同前缀时，SGLang 会优化处理：

```python
# 在 schedule_policy.py 中
# 批内前缀缓存逻辑
if len(r.prefix_indices) <= IN_BATCH_PREFIX_CACHING_CHECK_THRESHOLD:
    in_batch_matching_prefixes = (
        self.waiting_queue_radix_tree.match_prefix(...)
    )
    # 如果多个请求共享前缀，优先调度
```

**优化效果**:
```
请求1: "你好，介绍一下AI"
请求2: "你好，介绍一下ML"
请求3: "你好，介绍一下DL"

优化前: 每个请求都计算 "你好，介绍一下"
优化后: 计算一次，三个请求共享
```

### 4. **调度策略**

SGLang 使用智能调度策略处理多请求：

#### a) 缓存感知策略（Cache-Aware Policy）

```python
class CacheAwarePolicy:
    """优先调度有缓存命中的请求"""
    
    def select(self, waiting_queue):
        # 1. 查找有前缀匹配的请求
        # 2. 优先调度这些请求
        # 3. 提高缓存命中率
```

**优势**:
- 提高缓存利用率
- 减少计算量
- 降低延迟

#### b) 缓存无关策略（Cache-Agnostic Policy）

```python
class CacheAgnosticPolicy:
    """不考虑缓存，按其他策略调度"""
    
    def select(self, waiting_queue):
        # FIFO, Priority, etc.
```

## 📈 性能影响

### 1. **吞吐量提升**

```
单个请求: 100 tokens/s
批处理 32 个请求: 2000+ tokens/s

提升: 20x+
```

### 2. **GPU 利用率**

```
单个请求: GPU 利用率 10-20%
批处理: GPU 利用率 80-90%

提升: 4-5x
```

### 3. **延迟影响**

```
单个请求: TTFT = 50ms

增加: 2-4x（但吞吐量大幅提升）
```

## 🔍 代码中的实现

### 1. 等待队列管理

```python
# 在 scheduler.py 中
class Scheduler:
    def __init__(self):
        self.waiting_queue: List[Req] = []  # 等待队列
        self.running_queue: List[Req] = []   # 运行队列
    
    def schedule(self):
        # 1. 从等待队列选择请求
        # 2. 组成批次
        # 3. 执行
        batch = self.select_batch(self.waiting_queue)
        self.execute_batch(batch)
```

### 2. 批处理执行

```python
# 在 forward_batch_info.py 中
class ForwardBatch:
    """批处理信息"""
    
    def __init__(self):
        self.reqs: List[Req] = []  # 批次中的请求
        self.num_tokens: int = 0    # 总 token 数
    
    def forward(self):
        # 同时处理所有请求
        outputs = model.forward(self.reqs)
        return outputs
```

### 3. 前缀匹配

```python
# 在 schedule_policy.py 中
def _compute_prefix_matches(self, waiting_queue):
    """为等待队列中的请求匹配前缀"""
    
    for r in waiting_queue:
        # 匹配前缀
        r.prefix_indices, r.last_node = (
            self.tree_cache.match_prefix(r.key)
        )
        
        # 批内前缀缓存

                self.waiting_queue_radix_tree.match_prefix(r.key)
```

## ⚙️ 配置和优化

### 1. 批次大小

```python
# 启动参数
--max-num-batched-tokens 8192  # 最大批次 token 数
```

## 🎓 总结

### 什么时候会有多个请求？

1. ✅ **并发用户访问**: 多个用户同时使用服务
2. ✅ **批量处理任务**: 需要处理大量数据
3. ✅ **多轮对话**: 多个用户的多轮对话
4. ✅ **流式响应**: 新请求在旧请求处理时到达
5. ✅ **负载均衡**: 通过负载均衡器分发请求

### SGLang 如何优化？

1. ✅ **等待队列**: 管理到达的请求
2. ✅ **连续批处理**: 动态组合批次
3. ✅ **前缀匹配**: 识别可共享的前缀
4. ✅ **智能调度**: 优化请求执行顺序
5. ✅ **RadixAttention**: 缓存和重用前缀

### 关键理解

> **多个请求是常态，不是例外！**
> 
> SGLang 的设计就是为了高效处理多请求场景：
> - 通过批处理提高吞吐量
> - 通过前缀缓存减少计算
> - 通过智能调度优化资源利用
