# 业界多项目管理实践 - 大厂和实际工作者的做法

## 📋 概述

**目的**：了解大厂和其他实际工作者如何管理多个项目的代码和文档

---

## 🏢 大厂的常见做法

### 方案 1：Monorepo（单仓库）- Google、Facebook、Microsoft 等

**核心思路**：
- 所有相关项目放在一个大的 Git 仓库中
- 使用工具（如 Bazel、Buck）管理构建和依赖
- 文档统一管理在一个仓库

**实际例子**：
```
google/
├── projects/
│   ├── search/           # 搜索项目
│   ├── ads/              # 广告项目
│   ├── cloud/            # 云服务项目
│   └── ai/               # AI 项目
├── docs/                 # 统一文档目录
│   ├── search/
│   ├── ads/
│   └── common/          # 通用文档
└── shared/               # 共享代码库
    ├── utils/
    └── libs/
```

**优点**：
- ✅ 代码和文档统一管理
- ✅ 容易共享代码和文档
- ✅ 原子性提交（一次提交可以跨多个项目）
- ✅ 统一的版本控制

**缺点**：
- ❌ 仓库可能非常大（GB 级别）
- ❌ Clone 和操作可能很慢
- ❌ 需要专门的工具管理

**适用场景**：
- 大厂内部项目
- 紧密相关的项目
- 需要频繁共享代码的项目

---

### 方案 2：Multi-repo（多仓库）+ 组织账户 - GitHub、GitLab 等

**核心思路**：
- 每个项目一个独立的 Git 仓库
- 使用 GitHub/GitLab 的组织账户统一管理
- 文档可以放在独立仓库或每个项目仓库中

**实际例子**：
```
GitHub Organization: "mycompany"
├── project-a/           # 独立仓库
│   ├── src/
│   └── docs/
├── project-b/           # 独立仓库
│   ├── src/
│   └── docs/
└── shared-docs/         # 共享文档仓库
    ├── common/
    └── guides/
```

**优点**：
- ✅ 项目独立，权限清晰
- ✅ 仓库小，操作快
- ✅ 可以独立发布和版本控制
- ✅ 适合开源项目

**缺点**：
- ❌ 跨项目共享代码需要额外工具（submodule、package 等）
- ❌ 跨项目文档管理需要额外工作

**适用场景**：
- 开源项目
- 独立的产品线
- 需要不同权限控制的项目

---

### 方案 3：Git Submodule/Subtree - 共享代码和文档

**核心思路**：
- 使用 Git Submodule 或 Subtree 共享代码和文档
- 主项目引用共享库

**实际例子**：
```
company-main-project/
├── src/
├── docs/
└── shared-libs/          # Submodule
    ├── common-utils/     # 共享工具库
    └── common-docs/     # 共享文档库

other-project/
├── src/
└── shared-libs/          # 同样的 Submodule
    ├── common-utils/
    └── common-docs/
```

**优点**：
- ✅ 代码和文档可以共享
- ✅ 保持项目独立性
- ✅ 可以独立更新共享库

**缺点**：
- ❌ Submodule 管理复杂
- ❌ 需要团队成员了解 Submodule
- ❌ 更新需要额外步骤

**适用场景**：
- 有共享代码库的项目
- 需要版本控制的共享文档
- 中型团队

---

### 方案 4：独立的文档系统 - Confluence、Notion、GitBook 等

**核心思路**：
- 代码在 Git 仓库中
- 文档在独立的文档系统中
- 通过链接关联

**实际例子**：
```
代码仓库（GitHub）:
├── project-a/
│   └── README.md        # 指向 Confluence 文档
└── project-b/
    └── README.md        # 指向 Confluence 文档

文档系统（Confluence）:
├── Project A/
│   ├── Architecture
│   ├── API Docs
│   └── Guides
├── Project B/
│   ├── Architecture
│   └── Guides
└── Common/
    ├── Best Practices
    └── Shared Knowledge
```

**优点**：
- ✅ 文档系统功能强大（搜索、版本、权限等）
- ✅ 代码和文档分离，互不影响
- ✅ 适合大型团队协作

**缺点**：
- ❌ 需要额外的文档系统
- ❌ 可能产生费用
- ❌ 文档和代码可能不同步

**适用场景**：
- 大厂内部项目
- 需要复杂文档管理的项目
- 多团队协作

## 👨‍💻 实际工作者的常见做法

### 个人开发者 / 小团队

**方案 A：简单粗暴 - 每个项目独立**
```
I:\projects\
├── project-a\
│   ├── code\
│   └── docs\
└── project-b\
    ├── code\
    └── docs\
```
- ✅ 简单直接
- ❌ 文档可能重复
- ❌ 难以共享知识

**方案 B：共享文档目录**
```
I:\projects\
├── project-a\
│   └── code\
├── project-b\
│   └── code\
└── shared-docs\          # 共享文档
    ├── common\
    └── guides\
```
- ✅ 文档可以共享
- ✅ 简单易用
- ❌ 需要手动管理

**方案 C：使用 Git Submodule（你现在的方案）**
```
sglang-repo/
├── code/
├── radixark/             # Submodule
└── docs/                 # 统一文档
```
- ✅ 代码和文档统一管理
- ✅ 可以对比两个项目
- ⚠️ 需要学习 Submodule

---

### 中型团队 / 创业公司

**常见做法**：
1. **GitHub/GitLab 组织账户**
   - 所有项目在一个组织下
   - 使用团队管理权限
   - 文档放在每个项目的 `docs/` 目录

2. **共享文档仓库**
   - 创建一个 `company-docs` 仓库
   - 所有项目文档统一管理
   - 使用 Git Submodule 或直接引用

3. **Wiki 或文档系统**
   - 使用 GitHub Wiki、GitLab Wiki
   - 或使用 Notion、Confluence
   - 文档和代码分离

## 📊 业界实践总结

### 大厂做法

| 公司 | 做法 | 原因 |
|------|------|------|
| **Google** | Monorepo + Bazel | 所有代码统一管理，容易共享 |
| **Facebook/Meta** | Monorepo + Buck | 类似 Google，统一管理 |
| **Microsoft** | Multi-repo + Azure DevOps | 项目独立，但使用统一平台 |
| **Netflix** | Multi-repo + 独立文档系统 | 项目独立，文档集中管理 |
| **Uber** | Monorepo + 内部工具 | 统一管理，提高效率 |

### 实际工作者做法

| 类型 | 常见做法 | 原因 |
|------|---------|------|
| **个人开发者** | 共享文档目录 | 简单、实用 |
| **小团队（<10人）** | Multi-repo + GitHub 组织 | 项目独立，权限清晰 |
| **中型团队（10-50人）** | Multi-repo + 文档系统 | 需要更好的文档管理 |
| **大团队（>50人）** | Monorepo 或 文档系统 | 需要统一管理和协作 |

## 💡 对你的建议

### 推荐方案

**方案：项目标记 + 通用文档 + Git Submodule（可选）**

**理由**：
1. **简单实用**：不需要复杂的工具
2. **符合实际**：大多数个人开发者都这样做
3. **灵活**：可以根据需要调整
4. **可扩展**：如果以后有更多项目，可以继续使用

**具体操作**：
1. **保持现有文档结构**，但添加项目标记
2. **创建通用文档目录**（`common/`），移动通用文档
3. **Pull RadixArk 代码**（使用 submodule 或直接 clone）
4. **在文档中补充 RadixArk 信息**（当研究代码后）

**不需要**：
- ❌ 复杂的文档系统（Confluence、Notion 等）
- ❌ Monorepo 工具（Bazel、Buck 等）
- ❌ 大厂的复杂流程

**需要**：
- ✅ 清晰的文档分类
- ✅ 项目标记
- ✅ 文档索引

## 🔍 实际案例

### 案例 1：个人开发者（类似你的情况）

**做法**：
```
I:\research\
├── sglang\
│   ├── code\
│   └── docs\              # 项目特定文档
└── radixark\
    ├── code\
    └── docs\              # 项目特定文档

I:\research\shared-docs\  # 共享文档
├── RadixAttention.md

```

**特点**：
- 简单直接
- 文档可以共享
- 不需要 Git Submodule

### 案例 2：小团队（2-5人）

**做法**：
```
GitHub Organization: "our-team"
├── project-a/            # 独立仓库
│   └── docs/
├── project-b/            # 独立仓库
│   └── docs/
└── shared-docs/          # 共享文档仓库
    └── common/
```

**特点**：
- 使用 GitHub 组织账户
- 每个项目独立仓库
- 共享文档单独仓库

### 案例 3：中型团队（10-20人）

**做法**：
```
GitHub Organization + Confluence
├── GitHub（代码）
│   ├── project-a/
│   └── project-b/
└── Confluence（文档）
    ├── Project A/
    ├── Project B/
    └── Common/
```

**特点**：
- 代码在 GitHub
- 文档在 Confluence
- 通过链接关联

## 📝 总结

**大厂做法**：
- Monorepo（Google、Facebook）
- Multi-repo + 文档系统（Microsoft、Netflix）
- 复杂的工具和流程

**实际工作者做法**：
- 个人开发者：共享文档目录（简单实用）
- 小团队：Multi-repo + GitHub 组织
- 中型团队：Multi-repo + 文档系统

**对你的建议**：
- ✅ 使用"项目标记 + 通用文档"（方案 1）
- ✅ 可选：Git Submodule（如果想在一个仓库中管理）
- ✅ 保持简单，不要过度设计
- ✅ 根据实际需要调整

**关键点**：
- 大厂的复杂方案不适合个人开发者
- 简单实用的方案往往最好
- 文档管理的关键是"清晰"和"可维护"，不是"复杂"

---

**记住**：最好的方案是你能坚持使用的方案，而不是最完美的方案。
