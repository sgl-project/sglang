### Router 的分配作用

**Router 是请求分配的第一层**，在请求到达 Worker 之前就决定了请求应该发送到哪个 Worker。

#### Router 的分配策略

**1. Cache-Aware Routing（缓存感知路由）**
```
目标: 提高缓存命中率，减少重复计算

流程:
  1. Router 维护每个 Worker 的近似 Radix Tree
  2. 新请求到达时，Router 计算与每个 Worker 的前缀匹配率
  3. 如果匹配率 > cache_threshold:
     → 路由到匹配率最高的 Worker（利用缓存）
  4. 如果匹配率 ≤ cache_threshold:
     → 路由到树大小最小的 Worker（最多可用缓存空间）
```

**2. Load Balancing（负载均衡）**
```
目标: 平衡各个 Worker 的负载

流程:
  1. Router 跟踪每个 Worker 的负载（pending requests）
  2. 检查系统是否不平衡:
     - (max_load - min_load) > abs_threshold
     - max_load > min_load * rel_threshold
  3. 如果不平衡:
     → 路由到负载最轻的 Worker
  4. 如果平衡:
     → 使用 Cache-Aware Routing
```

**3. 健康检查和故障转移**
```
目标: 确保请求只发送到健康的 Worker

流程:
  1. Router 定期检查 Worker 的健康状态
  2. 如果 Worker 不健康:
     → 从候选列表中移除
     → 路由到其他健康的 Worker
  3. 如果所有 Worker 都不健康:
     → 返回错误（SERVICE_UNAVAILABLE）
```

#### 代码示例（SGLang Router）

```rust
// sgl-router/src/policies/cache_aware.rs

impl LoadBalancingPolicy for CacheAwarePolicy {
    fn select_worker(
        &self,
        workers: &[Arc<dyn Worker>],
        request_text: Option<&str>,
    ) -> Option<usize> {
        // 1. 获取健康的 Worker
        let healthy_indices = get_healthy_worker_indices(workers);
        
        // 2. 检查负载是否不平衡
        let loads: Vec<usize> = workers.iter().map(|w| w.load()).collect();
        let max_load = *loads.iter().max().unwrap_or(&0);
        let min_load = *loads.iter().min().unwrap_or(&0);
        
        let is_imbalanced = max_load.saturating_sub(min_load) > self.config.balance_abs_threshold
            && (max_load as f32) > (min_load as f32 * self.config.balance_rel_threshold);
        
        if is_imbalanced {
            // 3. 负载不平衡 → 使用负载均衡
            let min_load_idx = healthy_indices
                .iter()
                .min_by_key(|&&idx| workers[idx].load())
                .copied()?;
            return Some(min_load_idx);
        }
        
        // 4. 负载平衡 → 使用缓存感知路由
        let text = request_text.unwrap_or("");
        let tree = self.trees.get(model_id)?;
        let (matched_text, matched_worker) = tree.prefix_match(text);
        let match_rate = matched_text.chars().count() as f32 / text.chars().count() as f32;
        
        if match_rate > self.config.cache_threshold {
            // 匹配率高 → 路由到匹配的 Worker
            return find_worker_index(workers, matched_worker);
        } else {
            // 匹配率低 → 路由到缓存空间最大的 Worker
            let smallest_tenant = tree.get_smallest_tenant();
            return find_worker_index(workers, smallest_tenant);
        }
    }
}
```

#### 为什么需要 Router？

**场景 1: 多个 Worker 部署**
```
没有 Router:
  Client → 随机选择一个 Worker
  问题:
    - 负载可能不均衡
    - 无法利用缓存
    - 故障时无法自动转移

有 Router:
  Client → Router → 选择最优 Worker
  优势:
    - 负载均衡
    - 缓存感知路由
    - 自动故障转移
```

**场景 2: 缓存优化**
```
没有 Router:
  每个 Worker 独立处理请求
  问题:
    - 相同前缀的请求可能被分配到不同 Worker
    - 无法共享缓存
    - 重复计算

有 Router:
  Router 维护全局缓存状态
  优势:
    - 相同前缀的请求路由到同一 Worker
    - 共享缓存，减少计算
    - 提高吞吐量
```

### 关键时间点

**T0: 请求到达**
- Client 发送请求到 API Server
- API Server 接收并解析请求
- 创建 `Req` 对象

**T1: 进入 Waiting Queue**
- `Req` 对象被添加到 `waiting_queue`
- 记录进入队列的时间：`req.time_stats.wait_queue_entry_time`
- 请求开始等待调度

**T2: 批处理决策**
- 调度器检查是否可以创建新的 batch
- 决定哪些请求应该加入 batch
- 决定 batch 大小

**T3: 从 Waiting Queue 移除**
- 选中的请求从 `waiting_queue` 移除
- 添加到 `running_batch`
- 开始处理

## Waiting Queue 详解

### 什么是 Waiting Queue？

**Waiting Queue（等待队列）** 是存储**新到达但尚未被处理**的请求的队列。

#### 核心作用

```
1. 缓冲请求
   - 当 GPU 正在处理其他请求时，新请求在 waiting queue 中等待
   - 避免请求丢失

2. 批处理准备
   - 积累多个请求，准备组成 batch
   - 提高 GPU 利用率

3. 调度决策
   - 调度器从 waiting queue 中选择请求
   - 根据策略决定处理顺序
```

### Waiting Queue 的数据结构

#### 代码实现（SGLang）

```python
# python/sglang/srt/managers/scheduler.py

class Scheduler:
    def __init__(self, ...):
        # 等待队列：存储等待被调度的请求
        self.waiting_queue: List[Req] = []
```

**类型**: `List[Req]` - Python 列表，存储 `Req` 对象

#### Req 对象的关键字段

```python
# python/sglang/srt/managers/schedule_batch.py

class Req:
    def __init__(self, ...):
        # 请求标识
        self.rid: str                    # 请求唯一 ID
        
        # 输入数据
        self.origin_input_ids: List[int] # Token IDs: [101, 202, 303, ...]
        self.origin_input_text: str      # 原始文本
        
        # 输出数据（初始为空）
        self.output_ids: List[int] = []  # 生成的 token IDs
        
        # 采样参数
        self.sampling_params: SamplingParams
        # 包含:
        #   - max_new_tokens: int      # 最大生成 token 数
        #   - temperature: float       # 温度参数
        #   - top_p: float             # Top-p 采样
        #   - stop: List[str]          # 停止词
        
        # 优先级（用于优先级调度）
        self.priority: Optional[int] = None
        
        # 前缀缓存相关
        self.prefix_indices: List[int] = []  # 缓存的 token 索引
        self.last_node = None                 # Radix Tree 节点
        
        # 时间统计
        self.time_stats = TimeStats()
        self.time_stats.wait_queue_entry_time = time.perf_counter()  # 进入队列时间
```

### Waiting Queue 的操作

#### 1. 添加请求到 Waiting Queue

**时机**: 请求刚到达时

```python
# python/sglang/srt/managers/scheduler.py

def process_input_requests(self, recv_reqs: List[Req]):
    """处理新到达的请求"""
    for req in recv_reqs:
        # 1. 初始化请求
        req.init_next_round_input(self.tree_cache)
        
        # 2. 添加到 waiting queue
        self.waiting_queue.append(req)
        
        # 3. 记录进入队列时间
        req.time_stats.wait_queue_entry_time = time.perf_counter()
```

#### 2. 从 Waiting Queue 选择请求

**时机**: 调度器决定创建新 batch 时

```python
# python/sglang/srt/managers/scheduler.py

def get_new_batch_prefill(self):
    """从 waiting queue 中选择请求创建新的 prefill batch"""
    
    # 遍历 waiting queue
    for req in self.waiting_queue:
        # 检查是否可以加入 batch
        if can_add_to_batch(req):
            # 从 waiting queue 移除
            self.waiting_queue.remove(req)
            
            # 添加到 batch
            batch.add_request(req)
```

#### 3. Waiting Queue 的状态

```
┌─────────────────────────────────────┐
│      Waiting Queue 状态              │
├─────────────────────────────────────┤
│  空队列 (len = 0)                    │
│  - 没有等待的请求                     │
│  - 调度器可以跳过批处理检查            │
│                                      │
│  有请求 (len > 0)                    │
│  - 有请求等待处理                     │
│  - 调度器需要检查是否可以创建 batch    │
│                                      │
│  队列满 (len 很大)                    │
│  - 可能表示系统负载高                 │
│  - 需要增加 GPU 资源或优化调度        │
└─────────────────────────────────────┘
```

## 批处理决策的时机

### 什么时候做批处理决策？

**批处理决策发生在请求刚到达时**，具体时机如下：

#### 时机 1: 新请求到达时

```
Client → API Server → 创建 Req → 添加到 Waiting Queue
                                              │
                                              ▼
                                    [批处理决策开始]
```

#### 时机 2: 调度器循环检查时

```
调度器主循环:
  while True:
      # 1. 检查是否有新请求
      if len(waiting_queue) > 0:
          # 2. 批处理决策
          batch = get_new_batch_prefill()
          
      # 3. 处理当前 batch
      if running_batch:
          process_batch(running_batch)
```

### 批处理决策需要回答的问题

#### 问题 1: 何时创建新的 batch？

**决策因素**：

```
1. Waiting Queue 是否有请求？
   - len(waiting_queue) > 0

2. 当前 Running Batch 是否已满？
   - running_batch.batch_is_full == False
   - 或者 running_batch 为 None

3. 是否有足够的资源？
   - GPU 内存是否足够？
   - KV Cache 内存池是否有空间？
   - 是否达到最大并发请求数？

4. 是否应该等待更多请求？
   - 如果等待，可能组成更大的 batch（提高吞吐量）
   - 如果不等待，可以更快处理（降低延迟）
```

**代码示例**：

```python
def should_create_new_batch(self) -> bool:
    """判断是否应该创建新的 batch"""
    
    # 1. 检查 waiting queue
    if len(self.waiting_queue) == 0:
        return False
    
    # 2. 检查当前 batch 是否已满
    if self.running_batch and self.running_batch.batch_is_full:
        return False
    
    # 3. 检查资源
    if not self.has_enough_memory():
        return False
    
    # 4. 检查是否达到最大并发数
    if self.get_current_batch_size() >= self.max_running_requests:
        return False
    
    return True
```

#### 问题 2: 哪些请求应该加入 batch？

**决策因素**：

```
1. 调度策略
   - FCFS: 按到达时间顺序
   - LPM: 按前缀匹配长度
   - Priority: 按优先级

2. 资源限制
   - 请求的输入长度
   - 请求的预期输出长度
   - 总的内存需求

3. 批处理大小限制
   - 最大 batch size
   - 当前 batch 的剩余空间
```

**代码示例**：

```python
def select_requests_for_batch(self) -> List[Req]:
    """从 waiting queue 中选择请求加入 batch"""
    
    selected = []
    remaining_memory = self.get_available_memory()
    
    # 根据调度策略排序
    sorted_queue = self.sort_waiting_queue()  # FCFS/LPM/Priority
    
    for req in sorted_queue:
        # 估算请求的内存需求
        memory_needed = self.estimate_memory(req)
        
        # 检查是否可以加入
        if memory_needed <= remaining_memory:
            selected.append(req)
            remaining_memory -= memory_needed
            
            # 检查是否达到 batch 大小限制
            if len(selected) >= self.max_batch_size:
                break
    
    return selected
```

#### 问题 3: Batch 大小应该是多少？

**决策因素**：

```
1. 吞吐量 vs 延迟权衡
   - 更大的 batch → 更高的吞吐量，但延迟更高
   - 更小的 batch → 更低的延迟，但吞吐量更低

2. GPU 资源
   - GPU 内存限制
   - GPU 计算能力

3. 请求特征
   - 输入长度分布
   - 输出长度分布
   - 请求优先级

4. 系统负载
   - 当前等待的请求数
   - 当前运行的请求数
```

**动态调整策略**：

```python
def calculate_optimal_batch_size(self) -> int:
    """计算最优的 batch 大小"""
    
    # 1. 基础限制
    max_batch_size = min(
        self.max_running_requests,
        self.get_memory_limited_batch_size()
    )
    
    # 2. 根据等待队列长度调整
    waiting_count = len(self.waiting_queue)
    if waiting_count > 100:
        # 队列很长，增加 batch size（提高吞吐量）
        target_batch_size = max_batch_size
    elif waiting_count < 10:
        # 队列很短，减小 batch size（降低延迟）
        target_batch_size = min(8, max_batch_size)
    else:
        # 中等负载，平衡
        target_batch_size = max_batch_size // 2
    
    return target_batch_size
```

### 关键代码流程

#### 1. 请求到达并添加到 Waiting Queue

```python
# python/sglang/srt/managers/scheduler.py

def process_input_requests(self, recv_reqs: List[Req]):
    """处理新到达的请求"""
    for req in recv_reqs:
        # 1. 初始化请求（检查前缀缓存等）
        req.init_next_round_input(self.tree_cache)
        
        # 2. 添加到 waiting queue
        self._add_request_to_queue(req, is_retracted=False)
        
def _add_request_to_queue(self, req: Req, is_retracted: bool):
    """添加请求到 waiting queue"""
    # 记录进入队列时间
    req.time_stats.wait_queue_entry_time = time.perf_counter()
    
    # 添加到 waiting queue
    self.waiting_queue.append(req)
    
    # 如果是重试的请求，标记
    if is_retracted:
        req.is_retracted = True
```

#### 2. 从 Waiting Queue 选择请求创建 Batch

```python
# python/sglang/srt/managers/scheduler.py

def get_new_batch_prefill(self) -> Optional[ScheduleBatch]:
    """从 waiting queue 创建新的 prefill batch"""
    
    # 1. 检查 waiting queue
    if len(self.waiting_queue) == 0:
        return None
    
    # 2. 创建 PrefillAdder（用于管理批处理决策）
    adder = PrefillAdder(
        self.running_batch,
        self.req_to_token_pool,
        self.token_to_kv_pool_allocator,
        ...
    )
    
    # 3. 遍历 waiting queue，选择请求
    for req in self.waiting_queue:
        # 检查资源是否足够
        if not self.has_enough_resources(req):
            break
        
        # 检查 batch 是否已满
        if self.running_batch.batch_is_full:
            break
        
        # 初始化请求
        req.init_next_round_input(self.tree_cache)
        
        # 尝试添加到 batch
        res = adder.add_one_req(req, ...)
        
        if res == AddReqStatus.DELAYED:
            # 请求被延迟（需要等待）
            continue
        elif res == AddReqStatus.SUCCESS:
            # 请求成功加入 batch
            # 注意：此时 req 还在 waiting_queue 中
            continue
        else:
            # 无法加入，停止
            break
    
    # 4. 从 waiting queue 移除已选中的请求
    selected_reqs = adder.can_run_list
    for req in selected_reqs:
        self.waiting_queue.remove(req)
    
    # 5. 创建 ScheduleBatch
    if len(selected_reqs) > 0:
        new_batch = ScheduleBatch.init_new(
            selected_reqs,
            self.req_to_token_pool,
            self.token_to_kv_pool_allocator,
            self.tree_cache,
            ...
        )
        return new_batch
    
    return None
```

#### 3. PrefillAdder 的核心逻辑

```python
# python/sglang/srt/managers/schedule_policy.py

class PrefillAdder:
    def __init__(self, ...):
        self.can_run_list: List[Req] = []  # 可以运行的请求列表
        
    def add_one_req(self, req: Req, ...) -> AddReqStatus:
        """尝试添加一个请求到 batch"""
        
        # 1. 检查内存是否足够
        input_tokens = len(req.origin_input_ids)
        estimated_output_tokens = req.sampling_params.max_new_tokens
        
        total_tokens = input_tokens + estimated_output_tokens
        
        if not self.has_enough_memory(total_tokens):
            return AddReqStatus.NO_MEMORY
        
        # 2. 检查 batch 大小限制
        if len(self.can_run_list) >= self.max_batch_size:
            return AddReqStatus.BATCH_FULL
        
        # 3. 添加到 can_run_list
        self.can_run_list.append(req)
        
        # 4. 更新资源使用
        self.allocated_memory += total_tokens
        
        return AddReqStatus.SUCCESS
```

## 📝 总结

### 关键要点

1. **请求到达时的处理**：
   - 请求到达 → 创建 Req 对象 → 添加到 Waiting Queue
   - **这是批处理决策的开始**

2. **Waiting Queue 的作用**：
   - 缓冲新到达的请求
   - 为批处理积累请求
   - 支持调度策略（FCFS/LPM/Priority）

3. **批处理决策的时机**：
   - 请求刚到达时（添加到 waiting queue 后）
   - 调度器循环检查时
   - 需要回答：何时创建 batch？哪些请求加入？batch 大小？

4. **从 Waiting Queue 到 Running Batch**：
   - 调度器从 waiting_queue 选择请求
   - 创建 ScheduleBatch
   - 请求从 waiting_queue 移除，添加到 running_batch
   - 开始处理

### 设计考虑

1. **延迟 vs 吞吐量**：
   - 立即处理 → 低延迟，但吞吐量低
   - 等待积累 → 高吞吐量，但延迟高

2. **资源管理**：
   - 检查 GPU 内存
   - 检查 KV Cache 内存池
   - 避免 OOM

3. **调度策略**：
   - FCFS：公平，但可能不是最优
   - LPM：提高缓存命中率
   - Priority：支持 QoS

**通过理解请求队列管理和批处理决策，你将能够设计出高效的 LLM 推理调度系统！** 🚀
