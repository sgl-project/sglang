# 面试视角：云端推理（Fireworks / MiniMax 类）常见系统与高频问题

## 🎯 你现在的学习如何对应“云端推理岗位”

你在本地学 SGLang 的这些点，其实就是云端推理的核心，只是云端多了几层“平台化”：

- **入口层（API Gateway / Router）**：鉴权、限流、路由、重试、熔断、多租户、计费打点
- **推理运行时（Worker）**：Tokenizer → Scheduler → GPU 推理 → Detokenize → 返回
- **集群层（多机多卡）**：DP/TP/PP、NCCL 通信、跨节点调度、弹性扩缩
- **可观测与可靠性**：metrics/trace/log、SLO、故障隔离、回滚、灰度

你已经掌握的关键词：**TTFT、continuous batching、KV cache / prefix cache（Radix）、cache-aware routing、调度等待**，都非常对口。

## 🧭 一张图：云端推理系统“最常见形态”

```
Client SDK / App
  ↓  (OpenAI-compatible HTTP)
API Gateway / Router
  - auth / rate limit / request-id / retries / routing policy
  ↓
Inference Workers (N replicas)
  - FastAPI/HTTP server (or gRPC)
  - TokenizerManager
  - Scheduler (batching + memory + admission)
  - GPU kernels (attention / matmul)
  - KV cache (paged + prefix)
  ↓
Response (streaming SSE or non-streaming JSON)
```

## ✅ 面试高频问题清单（按模块）

### 1) 延迟指标：TTFT / TPOT / 端到端延迟

- **面试在问什么**：你会不会把延迟拆成可优化的子项，并知道每项在系统哪一层。
- **回答骨架**：
  - **TTFT** = 入站 → 排队/调度 → prefill 首 token 出来（含 tokenization、batching、GPU prefill）
  - **TPOT**（time per output token）= decode 循环每步的开销（更受 KV cache 带宽/内核影响）
  - 端到端还包含 **HTTP/序列化/网络/streaming flush** 等
- **你可对照的 SGLang 文档**：`yc_self_learn/md/03_TTFT_为什么重要.md`、`yc_self_learn/md/22_SGLang完整请求流程详解_纠正版.md`

### 2) 为什么 batching 能提吞吐，但会伤 TTFT？

- **面试在问什么**：你是否理解“吞吐 vs 延迟”的取舍与调度策略。
- **回答骨架**：
  - batching 提升 GPU 利用率（prefill 大 GEMM、更像吞吐型）
  - 但为了攒 batch 会引入排队等待 → TTFT 增加
  - 连续批处理（continuous batching）通过“边跑边加”降低等待
  - 用 admission 控制（max_batch_tokens、max_running_reqs、budget）稳定尾延迟
- **对照文档**：`yc_self_learn/md/12_调度器waiting_queue与get_new_batch详解.md`

### 3) KV cache 是什么？为什么 decode 阶段更吃带宽？

- **面试在问什么**：你是否理解 Transformer 推理的瓶颈从“算力”变为“显存带宽/容量”。
- **回答骨架**：
  - prefill：一次性算全序列 attention，更重算力；decode：每步都要读全历史 KV → **带宽型**
  - KV cache 规模 ~ O(layers × heads × head_dim × seq_len)，长上下文爆显存
  - 优化方向：paged KV、KV 量化、压缩、重排、减少 copy、融合 kernel
- **对照文档**：`yc_self_learn/md/06_RadixAttention详解.md`

### 4) Prefix cache / Radix cache 在解决什么？命中后省的是什么？

- **面试在问什么**：你能否清晰区分“路由层 cache-aware”与“worker 内 prefix cache”。
- **回答骨架**：
  - **命中后省的是 prefill 的重复计算**（共享前缀的注意力/投影）
  - Worker 内是 **token 级精确匹配 → 返回 KV 索引复用**
  - Router 层是 **文本级近似匹配 → 把请求送去更可能命中缓存的 worker**
- **对照文档**：`yc_self_learn/md/18_RadixAttention与Prefix_Matching关系详解.md`、`yc_self_learn/md/21_Router硬件与网络角度理解_应用层vs网络层.md`

### 5) PagedAttention / 分页 KV：为什么能撑长上下文？

- **面试在问什么**：你对“显存碎片、分配开销、索引结构”的理解。
- **回答骨架**：
  - 传统 contiguous KV：分配大块连续显存，容易碎片/失败
  - paged：KV 分块+页表（block table），按需分配/回收，管理更稳定
  - 代价：额外 indirection + 更复杂的 kernel（但能换来稳定性与吞吐）

### 6) Scheduler 在做什么？为什么会有“调度等待 0-10ms”？

- **面试在问什么**：你是否理解请求从 waiting→running 的决策点，以及预算约束。
- **回答骨架**：
  - Scheduler 维护 waiting_queue / running_set
  - 在预算内（tokens、KV、并发）挑选一批 req 进入 batch
  - 等待来自：攒 batch、等资源（KV/slot）、避免抖动（conservativeness）
- **对照文档**：`yc_self_learn/md/12_调度器waiting_queue与get_new_batch详解.md`

### 7) Attention kernel 在哪一层？你如何解释 FlashAttention 的价值？

- **面试在问什么**：你是否知道内核优化的目标是“减少 HBM 读写、减少临时张量”。
- **回答骨架**：
  - Attention 是 memory-bound；FlashAttention 通过 tiling/融合减少读写与中间激活
  - Kernel 选择会影响 TPOT、长上下文稳定性
  - 系统上要配合：合理 batch、KV layout、cuda graph（可选）
- **在流程里**：发生在 **GPU 推理阶段（Step 9）**

### 8) 并行（TP/PP/DP/PD）分别解决什么？瓶颈在哪里？

- **面试在问什么**：你是否理解“算力拆分”和“通信开销”的 trade-off。
- **回答骨架**：
  - **DP**：多副本分流请求（吞吐扩展），需要 Router/负载均衡
  - **TP**：层内张量切分（单请求加速/适配大模型），通信（NCCL all-reduce）成为瓶颈
  - **PP**：层间流水（吞吐），需要 micro-batch 和 bubble 控制
  - **PD（prefill/decode 拆分）**：优化资源利用（prefill heavy vs decode long tail），但引入跨节点/跨服务协调与状态（KV/请求）传递成本

### 9) 通信是不是都在 FastAPI/HTTP？

- **面试在问什么**：你能否区分应用层通信与推理内部通信。
- **回答骨架**：
  - HTTP/FastAPI：**入口/出口**（client↔gateway↔worker）
  - Worker 内部：IPC（例如 ZMQ）在 Tokenizer/Scheduler/Detokenizer 之间
  - 多 GPU/多机：NCCL/gRPC/自定义通道，主要发生在 GPU 推理阶段或并行同步点

### 10) 可靠性：重试/熔断/超时/幂等怎么设计？

- **面试在问什么**：你能否在“流式输出、长请求”场景里正确处理失败。
- **回答骨架**：
  - 重试要基于状态码/错误类型；流式已输出部分通常不可简单重试
  - 熔断保护下游（GPU worker）避免雪崩
  - request-id 用于 trace/排障，幂等依赖 request-id + 缓存/去重（可选）

### 11) 观测：你会看哪些指标定位性能回退？

- **面试在问什么**：你有没有系统化的 debug 方式。
- **回答骨架**：
  - 入口：RPS、排队深度、429、payload size
  - 推理：TTFT、TPOT、batch tokens、cache hit rate、GPU util、HBM BW、OOM 次数
  - 尾延迟：p50/p95/p99 + 分项 breakdown（queueing vs compute）

### 12) 成本：你如何把“更快”转化成“更便宜”？

- **面试在问什么**：你是否理解推理商业化最终目标是 $/token。
- **回答骨架**：
  - 吞吐↑ → 单位时间输出 token↑ → $/token↓
  - KV cache 命中↑ → prefill 省算力/省带宽 → QPS↑ 或 GPU 数↓
  - 合理并行与路由减少热点与空转
