## 1. Transformer 是什么？

### 1.1 历史背景

**Transformer** 是 Google 在 2017 年提出的神经网络架构，论文标题："Attention is All You Need"。

**革命性意义**：
- ✅ **首次完全基于注意力机制**，不需要 RNN 或 CNN
- ✅ **并行计算友好**，训练速度快
- ✅ **长距离依赖建模能力强**
- ✅ 成为现代 LLM（大语言模型）的基础架构

### 1.2 核心思想

**用一句话概括**：
> Transformer 通过**注意力机制**让模型能够"关注"输入序列中的任何位置，从而理解上下文关系。

**类比理解**：
```
想象你在读一篇文章：

传统 RNN: 像逐字阅读，必须按顺序，前面的信息会"遗忘"
  - 读第 10 个字时，第 1 个字的信息已经衰减很多

Transformer: 像"全局视野"阅读，可以同时看所有字
  - 读第 10 个字时，可以同时"关注"第 1 个字、第 5 个字等
  - 通过注意力机制，决定"关注"哪些位置
```

## 2. 整体架构

### 2.1 两种主要架构

Transformer 有两种主要架构：

#### a) **Encoder-Decoder 架构**（原始论文）
- 用于**序列到序列**任务（如翻译）
- 包含 Encoder 和 Decoder 两部分

#### b) **Decoder-Only 架构**（现代 LLM）
- 用于**生成式任务**（如 GPT、LLaMA）
- 只有 Decoder 部分
- **这是你学习的 SGLang 等推理系统主要处理的架构**

### 2.2 Encoder-Decoder 架构（完整版）

```
输入序列: "你好，世界"
         ↓
┌─────────────────────────────────────┐
│         Encoder Stack               │
│  ┌───────────────────────────────┐ │
│  │  Multi-Head Self-Attention    │ │
│  │  Feed-Forward Network         │ │
│  │  (重复 N 层)                   │ │
│  └───────────────────────────────┘ │
└─────────────────────────────────────┘
         ↓ (编码后的表示)
┌─────────────────────────────────────┐
│         Decoder Stack               │
│  ┌───────────────────────────────┐ │
│  │  Masked Self-Attention        │ │
│  │  Cross-Attention              │ │
│  │  Feed-Forward Network         │ │
│  │  (重复 N 层)                   │ │
│  └───────────────────────────────┘ │
└─────────────────────────────────────┘
         ↓
输出序列: "Hello, World"
```

### 2.3 Decoder-Only 架构（现代 LLM）

```
输入序列: "你好，世界"
         ↓
┌─────────────────────────────────────┐
│      Decoder-Only Stack             │
│  ┌───────────────────────────────┐ │
│  │  Masked Multi-Head Attention  │ │
│  │  Feed-Forward Network         │ │
│  │  (重复 N 层，如 32, 40, 80)    │ │
│  └───────────────────────────────┘ │
└─────────────────────────────────────┘
         ↓
输出序列: "Hello, World"
```

**关键区别**：
- Decoder-Only 没有 Encoder
- 使用 **Masked Attention**（只能看前面的 token，不能看后面的）
- 这是 GPT、LLaMA、ChatGLM 等模型的架构

#### 2.3.1 为什么不需要 Encoder？关键理解

**核心问题**：为什么 Decoder-Only 架构没有 Encoder 就能直接输出？

**答案**：因为 Decoder-Only 架构是**自回归生成**的，输入和输出在**同一个序列**中，不需要分开的 Encoder 来"理解"输入。

**用生活例子理解**：

```
═══════════════════════════════════════════════════════
Encoder-Decoder 架构（翻译任务）
═══════════════════════════════════════════════════════

场景：翻译 "你好，世界" → "Hello, World"

步骤 1: Encoder 理解输入
  - 输入: "你好，世界"（中文）
  - Encoder: 理解中文的含义，编码成隐藏表示
  - 输出: 隐藏表示（不是文字，是向量）

步骤 2: Decoder 生成输出
  - 输入: Encoder 的隐藏表示
  - Decoder: 根据隐藏表示，生成英文
  - 输出: "Hello, World"

关键: 输入和输出是**不同的语言**，需要 Encoder 先"理解"输入

═══════════════════════════════════════════════════════
Decoder-Only 架构（生成任务）
═══════════════════════════════════════════════════════

场景：续写 "你好，世界" → "你好，世界！今天天气真好"

步骤 1: 处理输入（Prefill）
  - 输入: "你好，世界"
  - Decoder: 直接处理这个序列，理解上下文
  - 不需要单独的 Encoder！

步骤 2: 生成输出（Decode）
  - 基于 "你好，世界" 的上下文
  - 逐个生成: "！" → "今天" → "天气" → "真好"
  - 输出: "你好，世界！今天天气真好"

关键: 输入和输出是**同一个序列的延续**，不需要分开理解
```

#### 2.3.2 Decoder-Only 的工作原理（详细版）

**关键理解**：Decoder-Only 架构把**输入和输出都当作同一个序列**的一部分。

**训练时的过程**：

```
训练数据: "你好，世界！今天天气真好"

训练方式（Teacher Forcing）:
  - 输入: ["<start>", "你好", "，", "世界", "！", "今天", "天气", "真好"]
  - 目标: 预测下一个 token

第 1 步: 输入 "<start>"
  - 模型预测: "你好"（目标）
  - 学习: 给定 "<start>"，应该输出 "你好"

第 2 步: 输入 ["<start>", "你好"]
  - 模型预测: "，"（目标）
  - 学习: 给定 "<start>你好"，应该输出 "，"

第 3 步: 输入 ["<start>", "你好", "，"]
  - 模型预测: "世界"（目标）
  - 学习: 给定 "<start>你好，"，应该输出 "世界"

... 以此类推

关键: 模型学习的是"给定前面的序列，预测下一个 token"
```

**推理时的过程**：

```
用户输入: "你好，世界"

步骤 1: Prefill（处理输入）
  - 输入: ["你", "好", "，", "世界"]
  - 模型处理整个输入序列
  - 生成 KV Cache（存储所有 token 的 K, V）

步骤 2: Decode（生成输出）
  - 基于 "你好，世界" 的上下文
  - 生成第 1 个 token: "！"
  - 生成第 2 个 token: "今天"
  - 生成第 3 个 token: "天气"
  - ... 逐个生成

关键: 模型根据输入序列的上下文，自回归地生成后续内容
```

#### 2.3.3 为什么不需要 Encoder？

**原因 1: 输入和输出是同一个序列**

```
Encoder-Decoder:
  输入: "你好，世界"（中文）
  输出: "Hello, World"（英文）
  → 输入和输出是**不同的表示**，需要 Encoder 转换

Decoder-Only:
  输入: "你好，世界"
  输出: "你好，世界！今天天气真好"
  → 输入和输出是**同一个序列的延续**，不需要转换
```

**原因 2: Decoder 自己就能"理解"输入**

```
Decoder-Only 的 Decoder:
  - 通过 Masked Self-Attention，可以"看到"所有输入 token
  - 通过多层 Transformer，可以理解输入的语义
  - 不需要单独的 Encoder 来"理解"输入

类比:
  - Encoder-Decoder: 像翻译，需要先理解中文，再生成英文
  - Decoder-Only: 像续写，直接基于前面的文字继续写
```

**原因 3: 训练目标就是"续写"**

```
训练目标:
  - 给定前面的序列，预测下一个 token
  - 模型学习的是"如何基于上下文生成下一个词"

推理时:
  - 给定用户输入，模型基于上下文生成后续内容
  - 本质上就是"续写"任务

结论: 不需要 Encoder，因为任务本身就是"续写"，不是"理解+生成"
```

#### 2.3.4 Encoder-Decoder vs Decoder-Only 对比

| 方面 | Encoder-Decoder | Decoder-Only |
|------|----------------|--------------|
| **架构** | Encoder + Decoder | 只有 Decoder |
| **任务类型** | 序列到序列（翻译、摘要） | 生成任务（续写、对话） |
| **输入输出关系** | 不同表示（如中→英） | 同一序列的延续 |
| **Encoder 作用** | 理解输入，编码成隐藏表示 | 不需要 |
| **Decoder 作用** | 基于 Encoder 输出生成 | 直接处理输入并生成 |
| **Cross-Attention** | 有（Decoder 关注 Encoder） | 没有 |
| **代表模型** | T5, BART | GPT, LLaMA, ChatGLM |

**详细对比**：

```
═══════════════════════════════════════════════════════
Encoder-Decoder（翻译任务）
═══════════════════════════════════════════════════════

输入: "你好，世界"（中文）
     ↓
Encoder: 理解中文，编码成隐藏表示
     ↓
隐藏表示: [0.1, 0.3, ..., 0.7]（向量，不是文字）
     ↓
Decoder: 基于隐藏表示，生成英文
     ↓
输出: "Hello, World"（英文）

关键: Encoder 和 Decoder 处理的是**不同的表示**

═══════════════════════════════════════════════════════
Decoder-Only（续写任务）
═══════════════════════════════════════════════════════

输入: "你好，世界"
     ↓
Decoder: 直接处理输入序列，理解上下文
     ↓
上下文理解: 基于 "你好，世界" 的语义
     ↓
Decoder: 基于上下文，生成后续内容
     ↓
输出: "你好，世界！今天天气真好"

关键: Decoder 处理的是**同一个序列**，不需要转换
```

#### 2.3.5 为什么 Decoder-Only 成为主流？

**优势 1: 架构简单**
- 只有一种组件（Decoder），更容易设计和优化
- 不需要协调 Encoder 和 Decoder 的交互

**优势 2: 训练统一**
- 所有任务都可以转化为"续写"任务（通过 prompt）
- 一个模型可以处理多种任务

**优势 3: 效果更好**
- 在生成任务上表现更好
- 可以学习更长的上下文依赖

**优势 4: 推理高效**
- 不需要维护两套参数（Encoder + Decoder）
- 推理时只需要 Decoder

**例子：如何用 Decoder-Only 做翻译？**

```
传统 Encoder-Decoder:
  输入: "你好，世界"
  → Encoder 理解 → Decoder 生成 → "Hello, World"

Decoder-Only（通过 Prompt）:
  输入: "Translate Chinese to English: 你好，世界"
  → Decoder 理解任务（通过 prompt）→ 生成 → "Hello, World"

关键: 通过设计 prompt，让 Decoder-Only 模型理解任务类型
```

#### 2.3.6 关键总结

**Decoder-Only 为什么不需要 Encoder？**

1. **输入和输出是同一序列的延续**
   - 不需要转换表示
   - 直接基于输入生成后续内容

2. **Decoder 自己就能理解输入**
   - 通过 Masked Self-Attention 理解上下文
   - 通过多层 Transformer 提取语义

3. **训练目标就是"续写"**
   - 学习"给定前面，预测后面"
   - 不需要分开的"理解"和"生成"阶段

4. **模型已经训练好了**
   - 训练时学习的是续写能力
   - 推理时直接应用这个能力

**类比理解**：
```
Encoder-Decoder: 像翻译员
  - 先理解中文（Encoder）
  - 再翻译成英文（Decoder）

Decoder-Only: 像作家
  - 看到前面的文字，直接继续写
  - 不需要先"理解"再"生成"，直接"续写"
```

## 3. 核心组件详解

### 3.1 注意力机制（Attention Mechanism）

#### a) 基本概念

**注意力机制的核心**：让模型决定"关注"输入序列中的哪些位置。

**用生活例子理解**：
```
你在读句子："The cat sat on the mat"

当处理 "sat" 这个词时：
  - 注意力机制会"关注" "cat"（主语）
  - 也会"关注" "mat"（宾语）
  - 不太关注 "The"（冠词）

注意力分数：表示每个词对当前词的重要性
```

#### b) 数学公式（简化理解）

**注意力计算**：
```
Attention(Q, K, V) = softmax(QK^T / √d) V
```

**参数解释**：
- **Q (Query)**: "我想找什么？"（查询向量）
- **K (Key)**: "我有什么？"（键向量）
- **V (Value)**: "我的内容是什么？"（值向量）
- **d**: 向量维度（用于缩放）

**计算步骤**（用例子理解）：
```
假设输入序列: ["你", "好", "世界"]
每个词都有 Q, K, V 向量

处理 "好" 这个词时：
  1. 计算 Q("好") 和所有 K 的相似度
     - Q("好") · K("你") = 0.8  (相似度高)
     - Q("好") · K("好") = 1.0  (自己和自己最相似)
     - Q("好") · K("世界") = 0.2 (相似度低)

  2. 通过 softmax 归一化，得到注意力分数
     - 注意力("你") = 0.4
     - 注意力("好") = 0.5
     - 注意力("世界") = 0.1

  3. 用注意力分数加权 V，得到输出
     - 输出 = 0.4 × V("你") + 0.5 × V("好") + 0.1 × V("世界")
```

**关键理解**：
- Q 和 K 用于计算"相似度"（注意力分数）
- V 是实际要"提取"的内容
- 注意力分数决定从每个位置"提取"多少信息

#### c) 自注意力（Self-Attention）

**定义**：Q、K、V 都来自同一个输入序列。

**例子**：
```
输入: ["你", "好", "世界"]

自注意力过程：
  - 每个词计算自己的 Q, K, V
  - "好" 的 Q 和所有词的 K 匹配（包括自己的 K）
  - 得到 "好" 的注意力输出
```

**为什么叫"自"注意力**：
- 因为序列中的每个位置都在"关注"序列中的其他位置（包括自己）

### 3.2 多头注意力（Multi-Head Attention）

#### a) 为什么需要多头？

**单头注意力的局限**：
- 只能学习一种"关注模式"
- 就像只有一双眼睛，只能从一个角度看

**多头注意力的优势**：
- 多个"头"可以学习不同的关注模式
- 就像有多双眼睛，从不同角度看

**例子**：
```
输入: "The cat sat on the mat"

头 1: 关注语法关系（"cat" 关注 "sat"）
头 2: 关注语义关系（"cat" 关注 "mat"）
头 3: 关注位置关系（"sat" 关注 "on"）
...
头 N: 关注其他模式

最后把所有头的输出拼接起来
```

#### b) 实现方式

```
输入: X (序列长度 × 模型维度)

对每个头 h:
  Q_h = X × W_Q_h  (查询投影)
  K_h = X × W_K_h  (键投影)
  V_h = X × W_V_h  (值投影)
  
  Attention_h = Attention(Q_h, K_h, V_h)

拼接所有头:
  MultiHead(X) = Concat(Attention_1, ..., Attention_H) × W_O
```

**参数**：
- **num_heads**: 头的数量（如 32, 64, 128）
- **head_dim**: 每个头的维度（通常是 model_dim / num_heads）

### 3.3 位置编码（Positional Encoding）

#### a) 为什么需要位置编码？

**问题**：注意力机制本身是**位置无关**的（permutation invariant）。

**例子**：
```
输入 1: ["你", "好", "世界"]
输入 2: ["世界", "好", "你"]

如果没有位置编码，模型会认为这两个序列是一样的！
（因为注意力只看内容，不看顺序）
```

**解决方案**：给每个位置添加位置编码（Positional Encoding）。

#### b) 两种位置编码方式

**1. 绝对位置编码（Absolute Positional Encoding）**
```
位置 0: [0.0, 0.0, 0.0, ...]
位置 1: [0.1, 0.2, 0.3, ...]
位置 2: [0.2, 0.4, 0.6, ...]
...

每个位置有固定的编码向量
```

**2. 相对位置编码（Relative Positional Encoding）**
```
关注位置之间的相对距离，而不是绝对位置
- 更灵活
- 可以处理更长的序列
```

**现代 LLM 常用**：
- **RoPE (Rotary Position Embedding)**: LLaMA、ChatGLM 使用
- **ALiBi (Attention with Linear Biases)**: 一些模型使用

#### c) 如何添加到输入？

```
输入嵌入: E = [embedding("你"), embedding("好"), embedding("世界")]
位置编码: P = [pos_encoding(0), pos_encoding(1), pos_encoding(2)]

最终输入: X = E + P  (逐元素相加)
```

### 3.4 前馈网络（Feed-Forward Network, FFN）

#### a) 结构

**每个 Transformer 层都包含 FFN**：

```
FFN(x) = ReLU(xW1 + b1)W2 + b2
```

**典型配置**：
- 输入维度: d_model (如 4096)
- 中间维度: d_ff (如 11008，通常是 d_model 的 2-4 倍)
- 输出维度: d_model

**用图理解**：
```
输入: x (d_model 维)
     ↓
线性层 1: x → 4×d_model (扩展)
     ↓
ReLU 激活
     ↓
线性层 2: 4×d_model → d_model (压缩)
     ↓
输出: FFN(x) (d_model 维)
```

#### b) 作用

- **非线性变换**：增加模型的表达能力
- **特征提取**：从注意力输出中提取更高级的特征

### 3.5 残差连接（Residual Connection）

#### a) 什么是残差连接？

**定义**：将输入直接加到输出上。

**公式**：
```
output = Layer(input) + input
```

**用图理解**：
```
输入: x
     ↓
    ┌─────────────┐
    │  Attention  │
    └─────────────┘
     ↓
    ┌─────────────┐
    │  Add & Norm │ ← 输入 x 直接加到这里
    └─────────────┘
     ↓
输出: Attention(x) + x
```

#### b) 为什么需要残差连接？

**问题**：深层网络容易出现梯度消失。

**解决方案**：残差连接提供"高速公路"，让梯度可以直接传播。

**类比**：
```
没有残差连接: 信息必须"爬楼梯"（逐层传递）
  - 深层时，信息容易丢失

有残差连接: 信息可以"坐电梯"（直接传递）
  - 即使深层，信息也能保留
```

### 3.6 层归一化（Layer Normalization）

#### a) 什么是层归一化？

**定义**：对每个样本的特征进行归一化。

**公式**：
```
LayerNorm(x) = γ * (x - μ) / √(σ² + ε) + β
```

其中：
- μ: 均值
- σ²: 方差
- γ, β: 可学习参数
- ε: 小常数（防止除零）

#### b) 为什么需要层归一化？

**作用**：
- **稳定训练**：防止激活值过大或过小
- **加速收敛**：让每层的输入分布更稳定
- **提高性能**：通常能提升模型效果

#### c) 位置：Pre-Norm vs Post-Norm

**Pre-Norm**（现代常用）：
```
x → LayerNorm → Attention → Add → LayerNorm → FFN → Add
```

**Post-Norm**（原始论文）：
```
x → Attention → Add → LayerNorm → FFN → Add → LayerNorm
```

**现代 LLM 多用 Pre-Norm**（训练更稳定）

### 3.7 完整的 Transformer 层

#### Decoder-Only 架构的完整层：

```
输入: x
     ↓
┌─────────────────────────────┐
│  LayerNorm (Pre-Norm)       │
│         ↓                    │
│  Masked Multi-Head Attention│
│         ↓                    │
│  Add (残差连接)              │
└─────────────────────────────┘
     ↓
┌─────────────────────────────┐
│  LayerNorm (Pre-Norm)       │
│         ↓                    │
│  Feed-Forward Network       │
│         ↓                    │
│  Add (残差连接)              │
└─────────────────────────────┘
     ↓
输出: 下一层的输入
```

**重复 N 次**（N = 层数，如 32, 40, 80）

## 4. 训练 vs 推理

### 4.1 训练阶段（Training）

#### a) 并行计算

**特点**：
- 所有 token **同时处理**
- 使用 **Teacher Forcing**：用真实标签作为输入

**例子**：
```
输入序列: ["你", "好", "世界"]
目标序列: ["Hello", "World"]

训练时:
  - 同时处理所有输入 token
  - 并行计算所有位置的输出
  - 计算损失，反向传播
```

**优势**：
- ✅ 充分利用 GPU 并行能力
- ✅ 训练速度快

#### b) 注意力掩码（Masking）

**训练时使用 Causal Mask**（因果掩码）：
```
输入: ["你", "好", "世界"]

注意力矩阵（训练时）:
     你    好    世界
你    ✓    ✗    ✗    (只能看自己)
好    ✓    ✓    ✗    (只能看自己和前面的)
世界  ✓    ✓    ✓    (可以看所有)
```

**为什么需要掩码**：
- 防止模型"作弊"（看到未来的 token）
- 保证推理时的一致性

### 4.2 推理阶段（Inference）

#### a) 自回归生成

**特点**：
- **逐个生成** token
- 每次生成一个，然后作为下一次的输入

**例子**：
```
输入: "你好"

步骤 1: 处理 "你好" → 生成 "世界"
步骤 2: 处理 "你好世界" → 生成 "！"
步骤 3: 处理 "你好世界！" → 生成 "</eos>"
...
```

#### b) KV Cache 的作用

**问题**：每次生成新 token 时，需要重新计算所有历史 token 的注意力。

**解决方案**：KV Cache（你已经学过！）

**回顾**：
- 每个 token 的 K 和 V 计算后**缓存**起来
- 生成新 token 时，只需要：
  1. 计算新 token 的 Q, K, V
  2. 从 KV Cache 读取所有历史 K, V
  3. 计算注意力

**详细内容**：参考 `25_KV_Cache详解与Decode带宽瓶颈.md`

### 4.3 关键区别总结

| 方面 | 训练 | 推理 |
|------|------|------|
| **计算模式** | 并行（所有 token 同时） | 串行（逐个生成） |
| **输入** | 完整序列 | 逐步累积 |
| **注意力** | 所有位置同时计算 | 每次只计算新 token |
| **KV Cache** | 不需要 | 必需（优化性能） |
| **瓶颈** | 计算量（Compute-Bound） | 内存带宽（Memory-Bound） |

## 4.4 为什么 Decode 阶段智能涌现得这么强烈？

### 4.4.1 什么是"智能涌现"？

**涌现（Emergence）** 是指系统在整体层面表现出单个组件所没有的复杂行为。

**在 LLM 中的表现**：
- 模型在训练时只学习"预测下一个 token"
- 但在 Decode 阶段，却表现出：
  - ✅ **推理能力**（逻辑推理、数学计算）
  - ✅ **理解能力**（理解复杂语义、上下文）
  - ✅ **创造能力**（写诗、编程、创作）
  - ✅ **规划能力**（多步推理、任务分解）

**关键问题**：为什么简单的"预测下一个 token"能产生如此强大的智能？

### 4.4.2 核心原因 1: 上下文累积效应

**Decode 过程是上下文不断累积的过程**：

```
生成第 1 个 token:
  上下文: [用户输入]
  智能水平: 基础理解

生成第 10 个 token:
  上下文: [用户输入 + 前 9 个生成的 token]
  智能水平: 开始展现推理

生成第 100 个 token:
  上下文: [用户输入 + 前 99 个生成的 token]
  智能水平: 复杂推理、多步思考

生成第 1000 个 token:
  上下文: [用户输入 + 前 999 个生成的 token]
  智能水平: 深度分析、创造性思维
```

**用生活例子理解**：

```
写文章的过程:

写第 1 句: "人工智能是..."
  - 上下文: 只有题目
  - 能力: 基础表达

写第 10 句: "...因此，我们可以得出结论..."
  - 上下文: 前面 9 句已经建立了逻辑框架
  - 能力: 开始展现推理和论证

写第 100 句: "...综上所述，这个问题的本质在于..."
  - 上下文: 前面 99 句已经构建了完整的论述体系
  - 能力: 深度分析、综合判断

关键: 每生成一个 token，上下文就丰富一分，智能就提升一分
```

**技术解释**：

```
每个新生成的 token:
  1. 被添加到 KV Cache（成为历史的一部分）
  2. 在后续生成时，所有历史 token 的 K, V 都被读取
  3. 新 token 的 Q 和所有历史 K 做注意力匹配
  4. 模型"看到"了整个上下文，包括自己刚生成的内容

结果:
  - 上下文越来越丰富
  - 模型可以"回顾"自己生成的内容
  - 可以基于前面的内容进行推理和规划
```

### 4.4.3 核心原因 2: 注意力机制的"全局视野"

**注意力机制让模型能够"同时看到"所有历史信息**：

```
生成第 N 个 token 时:

注意力计算:
  Q_new 和 [K1, K2, ..., KN-1] 匹配
  → 得到注意力分数: [score1, score2, ..., scoreN-1]
  → 加权 V: score1×V1 + score2×V2 + ... + scoreN-1×VN-1

关键: 模型可以"同时关注"所有历史 token，而不是只看最近的
```

**类比理解**：

```
传统 RNN（只能看最近的）:
  生成第 100 个 token 时:
    - 只能"记住"最近几个 token
    - 前面的信息已经"遗忘"
    - 像"健忘的人"，只能基于最近的内容生成

Transformer（全局视野）:
  生成第 100 个 token 时:
    - 可以"同时看到"所有前 99 个 token
    - 通过注意力机制，选择性地"关注"相关的内容
    - 像"过目不忘的人"，可以基于整个上下文生成

结果: Transformer 可以建立长距离依赖，进行复杂推理
```

**为什么这很重要？**

```
复杂推理需要多步思考:

例子: 解决数学问题
  "如果 A > B，B > C，那么 A 和 C 的关系是什么？"

生成过程:
  Token 1: "如果" → 开始建立逻辑框架
  Token 2: "A" → 引入变量 A
  Token 3: ">" → 建立关系
  Token 4: "B" → 引入变量 B
  ...
  Token 20: "那么" → 开始推理
  Token 21: "A" → 回顾前面的 A
  Token 22: "和" → 连接
  Token 23: "C" → 回顾前面的 C
  Token 24: "的" → 继续
  Token 25: "关系" → 推理关系
  Token 26: "是" → 准备结论
  Token 27: "A" → 再次回顾 A
  Token 28: ">" → 建立关系
  Token 29: "C" → 得出结论

关键: 生成 Token 29 时，模型需要"同时看到"Token 3（A > B）和 Token 7（B > C）
  → 只有注意力机制能做到这一点
  → 从而进行逻辑推理
```

### 4.4.4 核心原因 3: 自回归生成的"思考链"

**自回归生成让模型可以"逐步思考"**：

```
生成过程 = 思考过程:

生成 Token 1: "首先" → 开始思考
生成 Token 2: "我们" → 继续思考
生成 Token 3: "需要" → 规划步骤
生成 Token 4: "分析" → 确定方法
生成 Token 5: "问题" → 明确目标
生成 Token 6: "的" → 连接
生成 Token 7: "本质" → 深入思考
...

关键: 每个 token 都是"思考的一步"
  - 模型可以"边想边说"
  - 前面的思考为后面的思考提供基础
  - 形成"思考链"（Chain of Thought）
```

**例子：Chain of Thought 推理**：

```
问题: "如果今天是星期三，那么 3 天后是星期几？"

生成过程（思考链）:
  Token 1-5: "今天是星期三"
  Token 6-10: "，3 天后就是"
  Token 11-15: "星期三 + 3 = 星期六"
  Token 16-20: "，所以答案是星期六"

关键: 
  - Token 11-15 是"中间推理步骤"
  - 这些步骤不是直接答案，而是"思考过程"
  - 模型通过生成这些中间步骤，展现了推理能力
```

**为什么这能产生智能涌现？**

```
1. 模型可以"显式地"进行多步推理
   - 不是直接给出答案
   - 而是展示思考过程
   - 这需要"规划"和"推理"能力

2. 前面的推理步骤影响后面的生成
   - Token 11-15 的推理结果影响 Token 16-20 的结论
   - 模型需要"记住"自己的推理过程
   - 这需要"工作记忆"能力

3. 可以处理复杂任务
   - 简单任务: 直接生成答案
   - 复杂任务: 生成思考链，然后生成答案
   - 这需要"任务分解"能力
```

### 4.4.5 核心原因 4: 多层 Transformer 的"抽象层次"

**多层 Transformer 让模型能够建立多层次的抽象**：

```
第 1 层: 理解词汇和语法
  - "你好" → 理解这是问候语
  - "世界" → 理解这是名词

第 10 层: 理解语义和关系
  - "你好，世界" → 理解这是程序员的经典输出
  - 建立词汇之间的关系

第 20 层: 理解上下文和意图
  - 根据前面的对话，理解用户的意图
  - 理解当前生成的内容应该符合什么风格

第 30 层: 高级推理和规划
  - 进行逻辑推理
  - 规划多步任务
  - 创造性思维

关键: 每层都在"抽象"上一层的信息
  - 低层: 具体细节
  - 高层: 抽象概念
  - 最终: 智能行为
```

**在 Decode 过程中的体现**：

```
生成每个 token 时，都要经过所有层:

Token "你好":
  第 1 层: 理解 "你" 和 "好" 的词汇
  第 10 层: 理解这是问候语
  第 20 层: 理解这是对话的开始
  第 30 层: 规划后续应该生成什么

Token "世界":
  第 1 层: 理解 "世界" 的词汇
  第 10 层: 理解 "你好，世界" 是经典程序输出
  第 20 层: 理解用户可能是程序员
  第 30 层: 规划后续应该生成技术相关内容

关键: 每层都在"理解"和"规划"
  - 低层理解具体内容
  - 高层进行抽象推理
  - 最终产生智能行为
```

### 4.4.6 核心原因 5: 训练数据的"知识压缩"

**模型在训练时学习了海量知识**：

```
训练数据包含:
  - 百科全书（事实知识）
  - 代码库（编程知识）
  - 论文（科学知识）
  - 对话（语言知识）
  - 数学题（推理知识）
  - ...

模型学习:
  - 不是"记住"所有数据
  - 而是"压缩"成参数（权重）
  - 在 Decode 时"解压"这些知识
```

**为什么 Decode 时能"解压"知识？**

```
生成过程 = 知识检索 + 知识组合:

生成 "人工智能" 时:
  - 检索: 从参数中"检索"关于 AI 的知识
  - 组合: 将检索到的知识组合成文本

生成 "机器学习算法" 时:
  - 检索: 检索 AI、ML、算法等相关知识
  - 组合: 将这些知识组合成连贯的文本

关键: 
  - 每个 token 的生成都"激活"了相关的知识
  - 多个 token 的组合形成了"知识网络"
  - 最终展现出"智能"
```

### 4.4.7 为什么 Prefill 阶段智能不明显？

**Prefill 阶段的特点**：

```
Prefill 阶段:
  - 处理用户输入（已有的文本）
  - 并行计算所有 token
  - 主要是"理解"输入，不是"生成"内容

为什么智能不明显？
  1. 没有"生成"过程，只是"理解"
  2. 没有"思考链"，只是"编码"
  3. 没有"上下文累积"，输入是固定的
```

**Decode 阶段的特点**：

```
Decode 阶段:
  - 生成新内容（从无到有）
  - 逐个生成 token
  - 上下文不断累积
  - 可以展现"思考过程"

为什么智能明显？
  1. 有"生成"过程，展现创造性
  2. 有"思考链"，展现推理能力
  3. 有"上下文累积"，智能逐步提升
```

### 4.4.8 关键总结

**为什么 Decode 阶段智能涌现得这么强烈？**

1. **上下文累积效应**
   - 每个新 token 都丰富上下文
   - 上下文越丰富，智能越强

2. **注意力机制的全局视野**
   - 可以同时看到所有历史信息
   - 建立长距离依赖，进行复杂推理

3. **自回归生成的思考链**
   - 可以"边想边说"
   - 展现推理过程，而不是只给答案

4. **多层 Transformer 的抽象层次**
   - 低层理解细节，高层进行抽象推理
   - 每层都在"理解"和"规划"

5. **训练数据的知识压缩**
   - 模型参数中压缩了海量知识
   - Decode 时"解压"这些知识，展现智能

**类比理解**：

```
Prefill = 阅读（理解已有内容）
  - 智能不明显，因为只是"理解"

Decode = 写作（创造新内容）
  - 智能明显，因为需要"创造"和"推理"
  - 就像作家写作时展现的智慧
  - 每个词都是思考的结果
  - 整篇文章展现了完整的思维过程
```

**关键洞察**：

> **智能不是存储在模型中的，而是在 Decode 过程中"涌现"出来的。**
> 
> - 模型参数 = 压缩的知识
> - Decode 过程 = 知识解压 + 推理 + 创造
> - 智能 = 这个过程的结果

## 5. 与 KV Cache 的关系

### 5.1 为什么需要 KV Cache？

**从 Transformer 架构角度理解**：

```
生成第 N+1 个 token 时：

没有 KV Cache:
  - 需要重新计算所有历史 token 的 Q, K, V
  - 计算量: O(N²)（重复计算）

有 KV Cache:
  - 只计算新 token 的 Q, K, V
  - 从缓存读取历史 K, V
  - 计算量: O(N)（线性增长）
```

### 5.2 KV Cache 存储什么？

**存储内容**：
- 每个 token 在每个层的 **K** 和 **V** 向量
- **不存储 Q**（因为 Q 是查询，每次都要重新计算）

**存储位置**：
- GPU 显存（HBM - High Bandwidth Memory）

**大小计算**（你已经学过）：
```
KV_size_per_token = 2 × num_layers × num_heads × head_dim × dtype_size
```

### 5.3 Prefill vs Decode

**Prefill 阶段**：
- 处理**输入序列**（用户提供的 prompt）
- 并行计算所有输入 token 的 K, V
- 存储到 KV Cache

**Decode 阶段**：
- 逐个生成新 token
- 每次从 KV Cache 读取所有历史 K, V
- 计算注意力，生成下一个 token

**详细内容**：参考 `25_KV_Cache详解与Decode带宽瓶颈.md`

## 6. 常见变体

### 6.1 GPT 系列（Decoder-Only）

**特点**：
- 只有 Decoder 部分
- 使用 Masked Self-Attention
- 自回归生成

**代表模型**：
- GPT-1, GPT-2, GPT-3, GPT-4
- LLaMA, ChatGLM, Qwen

### 6.2 BERT（Encoder-Only）

**特点**：
- 只有 Encoder 部分
- 使用双向注意力（可以看到前后文）
- 用于理解任务（分类、NER 等）

**与 Decoder-Only 的区别**：
- BERT: 可以看到整个序列（双向）
- GPT: 只能看到前面的序列（单向）

### 6.3 T5（Encoder-Decoder）

**特点**：
- 完整的 Encoder-Decoder 架构
- 用于序列到序列任务（翻译、摘要等）

### 6.4 现代优化变体

#### a) FlashAttention
- **优化内存访问**：减少 HBM 访问
- **融合操作**：将多个操作融合成一个 kernel

#### b) Swin Transformer
- **层次化设计**：类似 CNN 的层次结构
- **用于视觉任务**

#### c) Longformer, BigBird
- **稀疏注意力**：减少计算量
- **处理长序列**

#### d) Grouped-Query Attention (GQA) - 标准方案

**什么是 GQA？**

GQA 是一种**减少 KV Cache 内存占用**的注意力机制变体。

**GQA 的设计背景与设计者：**

GQA (Grouped-Query Attention) 最初由 **Google 的研究团队**在 2023 年提出，主要作者包括 Joshua Ainslie 等人。这个设计是为了解决大模型推理时 KV Cache 内存占用过大的问题。

**设计动机：**
- 在自回归生成（如 GPT）中，需要缓存所有历史 token 的 K 和 V
- 对于长序列，KV Cache 会占用大量 GPU 内存
- 传统 MHA 中，每个头都有独立的 K、V，内存占用 = `2 × num_heads × head_dim × seq_len`
- 对于 70B 模型，如果 `num_heads=64`，KV Cache 可能达到几十 GB

**设计思路：**

GQA 的核心洞察是：**多个 Query 头可以共享同一组 Key-Value 对**，而不会显著影响模型性能。

设计过程：
1. **观察发现**：在 MHA 中，不同头的 K、V 往往学习到相似的模式
2. **实验验证**：通过消融实验发现，减少 KV 头数对性能影响很小
3. **设计权衡**：
   - `num_groups = 1` → MQA (Multi-Query Attention)，所有 Q 共享一个 K、V（最激进）
   - `num_groups = num_heads` → MHA (Multi-Head Attention)，每个 Q 有独立的 K、V（传统）
   - `num_groups` 在两者之间 → GQA（平衡方案）

**LLaMA-2 70B 中 num_groups = 8 的设计选择：**

```
设计考虑因素：
1. 内存效率：64 / 8 = 8 倍 KV Cache 减少
2. 性能保持：实验表明 8 个 KV 头足以保持模型性能
3. 计算效率：8 是 64 的因子，便于并行计算和硬件优化
4. 经验调优：通过大量实验发现 8 是一个好的平衡点

具体配置：
- num_heads = 64 (Query 头数)
- num_kv_heads = 8 (Key-Value 头数，即 num_groups)
- 每个 KV 头被 64/8 = 8 个 Q 头共享
```

**实现细节：**

在代码实现中（如 sglang 的 `GptOssAttention`）：
```python
# 关键参数
self.total_num_heads = 64      # Q 头数
self.total_num_kv_heads = 8    # KV 头数（即 num_groups）

# 线性层设计
self.qkv_proj = QKVParallelLinear(
    hidden_size,
    head_dim,
    total_num_heads=64,        # Q 投影到 64 个头
    total_num_kv_heads=8,      # K, V 只投影到 8 个头
    ...
)

# 注意力计算时
# 每个 Q 头会与对应的 KV 头计算注意力
# Q[0:8] 共享 K[0], V[0]
# Q[8:16] 共享 K[1], V[1]
# ...
# Q[56:64] 共享 K[7], V[7]
```

**设计优势：**
- ✅ **内存效率**：大幅减少 KV Cache（8 倍减少）
- ✅ **性能保持**：实验表明性能损失很小（通常 < 1%）
- ✅ **推理加速**：减少内存访问，提升推理速度
- ✅ **硬件友好**：分组设计便于并行计算

**传统 Multi-Head Attention (MHA)**：
```
每个头都有独立的 Q, K, V:
  - Q: num_heads 个（如 32 个）
  - K: num_heads 个（如 32 个）
  - V: num_heads 个（如 32 个）

KV Cache 大小: 2 × num_heads × head_dim × seq_len
```

**GQA 的核心思想**：
```
多个 Query 头共享同一个 Key 和 Value:
  - Q: num_heads 个（如 32 个）
  - K: num_groups 个（如 8 个，比 num_heads 少）
  - V: num_groups 个（如 8 个）

KV Cache 大小: 2 × num_groups × head_dim × seq_len
  → 减少了 (num_heads / num_groups) 倍
```

**为什么 Q 保持独立，而 K、V 要减少？**

这是 GQA 设计的核心问题。原因如下：

**1. Q 需要保持多样性（为什么 Q 不减少？）**
```
Q (Query) 的作用：
- Q 代表"我要查询什么"、"我关注什么"
- 不同的 Q 头学习不同的查询模式：
  * Q[0] 可能关注语法关系
  * Q[1] 可能关注语义相似性
  * Q[2] 可能关注位置关系
  * ...
- 如果减少 Q 头数，会损失模型的表达能力
- Q 只在当前 token 计算，不需要缓存，不占用额外内存
```

**2. K、V 可以共享（为什么 K、V 可以减少？）**
```
K (Key) 和 V (Value) 的作用：
- K 代表"我是什么"、"我的特征"
- V 代表"我的内容"、"我的值"
- 多个不同的 Q 可以查询同一个 K/V，得到不同的注意力模式

实验观察：
- 研究发现，不同头的 K、V 往往学习到相似的模式
- 多个 Q 头共享一组 K、V 时，每个 Q 仍然能产生不同的注意力权重
- 因为注意力 = Q·K^T，即使 K 相同，不同的 Q 也会产生不同的注意力分布

类比理解：
- 想象一个图书馆（K/V 是书籍）：
  * 不同的读者（Q）可以查询同一本书（K/V）
  * 但每个读者关注的重点不同（不同的 Q 产生不同的注意力）
  * 不需要为每个读者准备不同的书
```

**3. 内存效率考虑（关键原因）**
```
在自回归生成中：
- Q: 只计算当前 token，不缓存 → 内存占用 = 0（缓存）
- K, V: 需要缓存所有历史 token → 内存占用 = 2 × num_heads × head_dim × seq_len

减少 K、V 头数的收益：
- KV Cache 减少: (num_heads / num_groups) 倍
- 对于 LLaMA-2 70B: 64/8 = 8 倍减少
- 对于 2048 长度的序列，可能节省几十 GB 内存

而减少 Q 头数：
- 不会减少 KV Cache（因为 Q 本来就不缓存）
- 但会损失模型表达能力
- 得不偿失！
```

**4. 计算流程示例**
```
假设 num_heads=64, num_groups=8:

传统 MHA:
  Q[0] → K[0], V[0] → 注意力输出[0]
  Q[1] → K[1], V[1] → 注意力输出[1]
  ...
  Q[63] → K[63], V[63] → 注意力输出[63]
  需要缓存: 64 个 K + 64 个 V

GQA:
  Q[0:8]   → K[0], V[0] → 注意力输出[0:8]   (8个Q共享1个K/V)
  Q[8:16]  → K[1], V[1] → 注意力输出[8:16]  (8个Q共享1个K/V)
  ...
  Q[56:64] → K[7], V[7] → 注意力输出[56:64] (8个Q共享1个K/V)
  需要缓存: 8 个 K + 8 个 V  (减少 8 倍！)

每个 Q 仍然产生不同的注意力：
  - Q[0]·K[0]^T ≠ Q[1]·K[0]^T ≠ Q[2]·K[0]^T
  - 因为 Q[0], Q[1], Q[2] 是不同的向量
  - 所以即使共享 K，注意力分布也不同
```

**总结：**
- ✅ **Q 保持独立**：维持模型表达能力，且不占用缓存内存
- ✅ **K、V 减少**：基于实验观察（相似模式）+ 内存效率（减少缓存）
- ✅ **设计权衡**：在性能和效率之间找到最佳平衡点

**例子**：
```
LLaMA-2 70B:
  - num_heads = 64
  - num_groups = 8 (GQA 配置)
  - KV Cache 减少: 64 / 8 = 8 倍

优势:
  ✅ 大幅减少 KV Cache 内存占用
  ✅ 推理速度提升
  ⚠️ 可能略微降低性能（但通常可接受）
```

**采用 GQA 的模型**：
- LLaMA-2, LLaMA-3
- Mistral
- 大多数现代开源 LLM

#### e) Multi-Head Latent Attention (MLA) - DeepSeek V3 的创新

**什么是 MLA？**

MLA 是 DeepSeek V3 (2024) 引入的**更激进的 KV Cache 压缩方案**，通过**低秩压缩**进一步减少内存占用。

**GQA vs MLA 对比**：

```
═══════════════════════════════════════════════════════
GQA (Grouped-Query Attention)
═══════════════════════════════════════════════════════

方法: 减少 K, V 的头数
  - 多个 Q 头共享 K, V
  - 直接减少 KV Cache 大小

KV Cache: 2 × num_groups × head_dim × seq_len
  - 例如: 2 × 8 × 128 × 2048 = 4 MB (对于 2048 tokens)

优势:
  ✅ 实现简单
  ✅ 内存减少明显
  ⚠️ 可能略微降低性能

═══════════════════════════════════════════════════════
MLA (Multi-Head Latent Attention)
═══════════════════════════════════════════════════════

方法: 低秩压缩 K, V
  - 将 K, V 压缩到低维空间（latent space）
  - 生成压缩向量 c_KV ∈ R^d_c (d_c 很小，如 512)
  - 只缓存压缩后的向量

KV Cache: 2 × d_c × seq_len (d_c << head_dim)
  - 例如: 2 × 512 × 2048 = 2 MB (对于 2048 tokens)
  - 比 GQA 更小！

优势:
  ✅ 内存占用更小（比 GQA 还小）
  ✅ 性能损失更小（甚至没有损失）
  ✅ 更灵活（可以调整压缩维度）

DeepSeek V3 配置:
  - KV 压缩维度 d_c = 512
  - Query 压缩维度 d'_c = 1536 (训练时)
```

**MLA 的工作原理**：

```
传统方式:
  K, V ∈ R^(head_dim) → 直接缓存
  KV Cache 大小: 2 × head_dim × seq_len

MLA 方式:
  1. 计算 K, V
  2. 压缩: K, V → c_KV ∈ R^d_c (低维潜在向量)
  3. 缓存: 只缓存 c_KV
  4. 使用时: 从 c_KV 恢复（如果需要）

KV Cache 大小: 2 × d_c × seq_len (d_c << head_dim)
  → 大幅减少内存占用
```

**为什么 MLA 性能损失更小？**

```
GQA:
  - 直接减少 K, V 的头数
  - 信息损失: 硬性减少（不可逆）

MLA:
  - 通过低秩压缩保留关键信息
  - 信息损失: 智能压缩（可恢复）
  - 压缩维度可以调整，平衡内存和性能
```

**MLA 的优势总结**：

1. **内存占用更小**
   - 比 GQA 进一步减少 KV Cache
   - 对于长序列，节省更明显

2. **性能损失更小**
   - 通过智能压缩保留关键信息
   - DeepSeek V3 实验显示几乎没有性能损失

3. **更灵活**
   - 可以调整压缩维度 d_c
   - 根据需求平衡内存和性能

**采用 MLA 的模型**：
- DeepSeek V3 (2024)

**未来趋势**：
- MLA 可能成为新的标准（替代 GQA）
- 特别是在长上下文场景中

#### f) Mixture-of-Experts (MoE) - 不再是可选的！

**重要澄清**：MoE **并没有过时**，相反，它已经成为**主流架构**！

**什么是 MoE？**

MoE 是一种**稀疏激活**的模型架构，通过只激活部分"专家"（Expert）来减少计算量。

**传统 Dense 模型**：
```
每个 token 都经过所有参数:
  - 总参数: 70B
  - 每个 token 激活: 70B 参数
  - 计算量: 大
```

**MoE 模型**：
```
每个 token 只经过部分专家:
  - 总参数: 671B (DeepSeek V3)
  - 每个 token 激活: 37B 参数（只激活部分专家）
  - 计算量: 小（只计算激活的专家）
```

**MoE 的工作原理**：

```
输入 token → Router（路由器）
  ↓
Router 选择 Top-K 个专家（如 Top-2）
  ↓
只激活选中的专家
  ↓
专家处理输入 → 输出
  ↓
加权组合所有专家的输出
```

**例子：DeepSeek V3**：
```
总参数: 671B
专家数量: 多个（具体数量未公开）
每个 token 激活: 37B 参数
激活比例: 37B / 671B ≈ 5.5%

优势:
  ✅ 模型容量大（671B 参数）
  ✅ 计算量小（只激活 37B）
  ✅ 性能强（大模型的能力）
  ✅ 推理效率高（稀疏激活）
```

**为什么 MoE 成为主流？**

**1. 前十大最智能的开源模型都使用 MoE**（NVIDIA 报告）：
- DeepSeek-R1
- Kimi K2 Thinking
- gpt-oss-120B
- Mistral Large 3
- ...

**2. 效率优势明显**：
```
Dense 模型:
  - 70B 参数 → 需要 70B 的计算
  - 内存: 大
  - 推理速度: 慢

MoE 模型:
  - 671B 参数 → 只需要 37B 的计算
  - 内存: 相对较小（只激活部分）
  - 推理速度: 快（稀疏计算）
```

**3. 性能优势**：
- 更大的模型容量 → 更强的能力
- 稀疏激活 → 更高效的计算
- 最佳平衡点

**MoE 的挑战**：

1. **负载均衡**
   - 需要确保所有专家都被使用
   - 避免某些专家过载，某些空闲

2. **通信开销**
   - 专家之间需要通信
   - 分布式训练和推理的挑战

3. **实现复杂度**
   - 比 Dense 模型更复杂
   - 需要专门的优化

**采用 MoE 的模型**：
- DeepSeek V3 (671B 总参数，37B 激活)
- GPT-4 (推测)
- Claude (推测)
- Mixtral (Mistral)
- 大多数前沿模型

**关键理解**：

> **"MoE is No Longer Optional"** 的意思是：
> - MoE 不再是"可选的"架构
> - 而是**必需的/主流的**架构
> - 如果你想达到 SOTA 性能，MoE 几乎是必须的

**MoE 并没有过时，而是成为了标准！** ✅

## 8. 常见问题 FAQ

### Q1: 为什么 Transformer 比 RNN 好？

**A**: 
- ✅ **并行计算**：RNN 必须串行，Transformer 可以并行
- ✅ **长距离依赖**：RNN 有梯度消失问题，Transformer 通过注意力直接建模
- ✅ **训练速度**：Transformer 训练更快

### Q2: 注意力机制的计算复杂度是多少？

**A**:
- **时间复杂度**: O(N²)，其中 N 是序列长度
- **空间复杂度**: O(N²)，需要存储注意力矩阵

**为什么是 O(N²)**：
- 每个 token 需要和所有 token 计算注意力
- N 个 token × N 个 token = N²

**优化方法**：
- FlashAttention: 减少内存访问
- 稀疏注意力: 减少计算量
- 线性注意力: 降低复杂度到 O(N)

### Q3: 为什么 Decoder-Only 架构成为主流？

**A**:
- ✅ **生成任务**：LLM 主要是生成任务，不需要 Encoder
- ✅ **架构简单**：更容易训练和优化
- ✅ **效果更好**：在生成任务上表现更好
- ✅ **统一架构**：一个架构解决多种任务（通过 prompt）

### Q4: 位置编码为什么用加法而不是拼接？

**A**:
- **加法更简单**：只需要逐元素相加
- **维度不变**：保持输入维度
- **实验效果好**：实际效果更好

**拼接的问题**：
- 维度翻倍（需要更多参数）
- 效果不一定更好

### Q5: 残差连接为什么能解决梯度消失？

**A**:
- **直接路径**：梯度可以通过残差连接直接传播
- **数学上**：如果 Layer(x) ≈ 0，那么 output ≈ x，梯度 ≈ 1
- **深层网络**：即使很多层，梯度也能传播
