## 1. 什么是 KV Cache？

### 1.1 基本概念

**KV Cache（Key-Value Cache）** 是 Transformer 模型在推理时用来缓存**注意力机制中间结果**的内存区域。

### 1.2 Transformer 注意力机制回顾（简化版）

**不用手推公式！用直觉理解：**

在 Transformer 的注意力机制中，每个 token 需要"看"其他所有 token：

```
Attention(Q, K, V) = softmax(QK^T / √d) V
```

**用生活例子理解**：
- **Q (Query)**: "我想找什么？"（查询）
- **K (Key)**: "我有什么？"（键）
- **V (Value)**: "我的内容是什么？"（值）

**注意力过程**：
1. Q 和 K 匹配 → 找到"相关度"
2. 用相关度加权 V → 得到"关注的内容"

**关键点**：每个 token 的 K 和 V 计算出来后，**可以重复使用**，不需要每次都重新算！

### 1.3 KV Cache 的作用

**在推理过程中**：
- 每个 token 经过模型后，都会生成对应的 **K** 和 **V** 向量
- 这些 K 和 V 向量需要**保存下来**，供后续 token 计算注意力时使用
- **KV Cache 就是存储这些 K 和 V 向量的内存区域**

### 1.4 直观理解（不用手推！）

**用"图书馆"的比喻**：

```
想象你在图书馆写论文：

第 1 天: 读了"人工智能"这本书
  → 做了笔记（K1, V1）
  → 把笔记存到"我的笔记库"（KV Cache）

第 2 天: 读"机器学习"这本书
  → 做了新笔记（K2, V2）
  → 存到笔记库
  → 写论文时，需要参考昨天的笔记（从笔记库读取 K1, V1）

第 3 天: 读"深度学习"这本书
  → 做了新笔记（K3, V3）
  → 存到笔记库
  → 写论文时，需要参考前两天的笔记（从笔记库读取 K1, V1, K2, V2）

... 以此类推

关键：笔记（KV）只做一次，以后直接查笔记库，不用重新读书！
```

**对应到 Transformer**：
```
输入序列: "你好，世界"

Token 1: "你"
  → 计算并存储 K1, V1 到 KV Cache（做笔记）

Token 2: "好"
  → 计算并存储 K2, V2 到 KV Cache（做新笔记）
  → 计算注意力时，从 KV Cache 读取 K1, V1（查旧笔记）

Token 3: "，"
  → 计算并存储 K3, V3 到 KV Cache（做新笔记）
  → 计算注意力时，从 KV Cache 读取 K1, V1, K2, V2（查旧笔记）

... 以此类推
```

**核心思想**：**一次计算，多次使用**！不用每次都重新算！

## 2. 为什么需要 KV Cache？

### 2.1 没有 KV Cache 的问题

**如果不缓存 KV**：
- 每次生成新 token 时，都需要**重新计算**所有历史 token 的 K 和 V
- 计算量：O(n²)，其中 n 是序列长度
- **非常低效！**

### 2.2 使用 KV Cache 的优势

**缓存 KV 后**：
- 每个 token 的 K 和 V **只计算一次**
- 后续生成时，直接**从缓存读取**
- 计算量：O(n)，线性增长
- **大幅提升效率！**

### 2.3 计算量对比（用例子理解，不用手推公式）

**用"做菜"的比喻**：

```
场景：要做 100 道菜，每道菜都需要用到前面所有菜的"调料包"

没有 KV Cache（每次都重新做调料包）:
  - 第 1 道菜: 做 1 个调料包
  - 第 2 道菜: 重新做 2 个调料包（浪费！）
  - 第 3 道菜: 重新做 3 个调料包（浪费！）
  - ...
  - 第 100 道菜: 重新做 100 个调料包（超级浪费！）
  - 总工作量: 1 + 2 + 3 + ... + 100 = 5050 次做调料包

有 KV Cache（调料包只做一次，存起来）:
  - 第 1 道菜: 做 1 个调料包，存到冰箱（KV Cache）
  - 第 2 道菜: 做 1 个新调料包，从冰箱拿 1 个旧的
  - 第 3 道菜: 做 1 个新调料包，从冰箱拿 2 个旧的
  - ...
  - 第 100 道菜: 做 1 个新调料包，从冰箱拿 99 个旧的
  - 总工作量: 100 次做调料包（只做新的，旧的从冰箱拿）
```

**对应到 Transformer**：
```
序列长度 = 100 tokens

没有 KV Cache:
  - 总计算量: 5050 次 KV 计算（每次都重新算）

有 KV Cache:
  - 总计算量: 100 次 KV 计算（只算新的，旧的从缓存读）
```

**加速比**: 5050 / 100 = **50.5x**（对于 100 token 序列）

**关键理解**：不用手推公式，记住"**一次计算，多次使用**"就行！

## 3. KV Cache 的内存占用

### 3.1 计算公式

**单个 token 的 KV Cache 大小**：

```
KV_size_per_token = 2 × num_layers × num_heads × head_dim × dtype_size
```

其中：
- **2**: K 和 V 各一份
- **num_layers**: Transformer 层数（如 32, 40, 80）
- **num_heads**: 注意力头数（如 32, 40, 128）
- **head_dim**: 每个头的维度（如 128, 256）
- **dtype_size**: 数据类型大小（FP16 = 2 bytes, BF16 = 2 bytes, FP32 = 4 bytes）

### 3.2 实际例子

**Llama-2 70B 模型**：
```
num_layers = 80
num_heads = 64
head_dim = 128
dtype = FP16 (2 bytes)

KV_size_per_token = 2 × 80 × 64 × 128 × 2
                  = 2,621,440 bytes
                  ≈ 2.5 MB per token
```

**对于 2048 token 的序列**：
```
总 KV Cache = 2.5 MB × 2048 = 5,120 MB ≈ 5 GB
```

**对于 32K token 的序列**：
```
总 KV Cache = 2.5 MB × 32768 = 81,920 MB ≈ 80 GB
```

### 3.3 内存占用特点

- ✅ **线性增长**: 序列长度每增加 1，内存增加 `KV_size_per_token`
- ✅ **累积性**: 所有历史 token 的 KV 都需要保存
- ✅ **长上下文挑战**: 长序列会占用大量显存

## 4. Prefill vs Decode 的区别

### 4.1 Prefill 阶段（首次处理输入）

**特点**：
- 一次性处理**整个输入序列**
- 并行计算所有 token 的注意力
- **计算密集型**（Compute-Bound）

**用"批量做菜"的比喻**：
```
场景：有 100 道菜要做，所有食材都准备好了

Prefill = 批量做菜
  - 所有菜一起做（并行）
  - 所有调料包一起做（一次性计算所有 KV）
  - GPU 算力被充分利用（像大厨房，所有炉子一起开）
  - 内存访问：一次性读取所有食材，一次性存储所有调料包
```

**计算模式（简化理解）**：
```
输入: [token1, token2, token3, ..., tokenN]

一次性并行计算:
  - 所有 token 的 Q, K, V 一起算
  - 所有 token 之间的注意力一起算
  - 像"大矩阵乘法"，GPU 最喜欢这种！

内存访问:
  - 读取输入: 一次性
  - 写入 KV Cache: 一次性
  - 访问量: 相对较小
```

**关键**：**并行计算，GPU 算力是瓶颈**，不是内存带宽！

### 4.2 Decode 阶段（生成新 token）

#### 4.2.1 什么是 Decode？

**Decode（解码/生成）** 是指模型根据已有的输入，**逐个生成新 token** 的过程。

**用生活例子理解**：
```
场景：你在和 ChatGPT 聊天

你输入: "请介绍一下人工智能"

Prefill 阶段（模型处理你的输入）:
  - 模型理解你的问题
  - 准备好回答的"上下文"

Decode 阶段（模型生成回答）:
  - 第 1 步: 生成 "人工"
  - 第 2 步: 生成 "智能"
  - 第 3 步: 生成 "是"
  - 第 4 步: 生成 "一门"
  - 第 5 步: 生成 "学科"
  - ... 逐个生成，直到回答完成
```

**为什么叫 "Decode"（解码）？**
- 模型内部用**隐藏表示**（hidden states）来"编码"信息
- 生成时，需要把这些隐藏表示"解码"成我们能看到的**文字**（token）
- 所以叫 Decode（解码）

**关键理解**：
- Decode = **生成新内容**的过程
- 每次只生成 **1 个 token**（一个字或词）
- 必须**按顺序生成**（不能跳着生成）

#### 4.2.2 Decode 的完整过程（详细版）

**用具体例子理解**：

```
假设你已经输入: "你好"
模型要生成回答: "世界"

═══════════════════════════════════════════════════════
Decode 步骤 1: 生成第一个 token "世界"
═══════════════════════════════════════════════════════

当前状态:
  - 输入序列: ["你", "好"] (已经在 KV Cache 中)
  - 要生成: 第 1 个新 token

步骤 1.1: 计算新 token 的 Q, K, V
  - 输入: 上一个 token（或特殊起始符）
  - 输出: Q_new, K_new, V_new（只算 1 个 token，很快！）

步骤 1.2: 从 KV Cache 读取历史 KV
  - 读取: K("你"), V("你")  ← 从内存读取
  - 读取: K("好"), V("好")  ← 从内存读取
  - 读取量: 2 × KV_size_per_token（相对较小）

步骤 1.3: 计算注意力
  - Q_new 和 [K("你"), K("好")] 匹配
  - 得到注意力分数: [0.3, 0.7]（"好"更重要）
  - 加权 V: 0.3 × V("你") + 0.7 × V("好")
  - 得到上下文向量

步骤 1.4: 通过模型层
  - 上下文向量 → FFN → 层归一化 → ...
  - 经过所有 Transformer 层

步骤 1.5: 生成 token
  - 最后一层输出 → 词汇表概率分布
  - 选择概率最高的 token: "世界"
  - 将 "世界" 的 K, V 存入 KV Cache

结果: 生成 "世界"，KV Cache = [("你", K1, V1), ("好", K2, V2), ("世界", K3, V3)]

═══════════════════════════════════════════════════════
Decode 步骤 2: 生成第二个 token "！"
═══════════════════════════════════════════════════════

当前状态:
  - 已生成序列: ["你", "好", "世界"]
  - 要生成: 第 2 个新 token

步骤 2.1: 计算新 token 的 Q, K, V
  - 输入: "世界"（上一个生成的 token）
  - 输出: Q_new, K_new, V_new（只算 1 个，很快！）

步骤 2.2: 从 KV Cache 读取历史 KV（注意：现在有 3 个了！）
  - 读取: K("你"), V("你")  ← 从内存读取
  - 读取: K("好"), V("好")  ← 从内存读取
  - 读取: K("世界"), V("世界")  ← 从内存读取
  - 读取量: 3 × KV_size_per_token（比之前多了！）

步骤 2.3: 计算注意力
  - Q_new 和 [K("你"), K("好"), K("世界")] 匹配
  - 得到注意力分数: [0.1, 0.2, 0.7]（"世界"最重要）
  - 加权 V: 0.1 × V("你") + 0.2 × V("好") + 0.7 × V("世界")
  - 得到上下文向量

步骤 2.4: 通过模型层 → 生成 "！"
  - 将 "！" 的 K, V 存入 KV Cache

结果: 生成 "！"，KV Cache = [("你", K1, V1), ("好", K2, V2), ("世界", K3, V3), ("！", K4, V4)]

... 以此类推，直到生成结束符或达到最大长度
```

#### 4.2.3 Decode 的关键特点

**特点 1: 逐个生成（自回归）**
```
不能并行生成:
  ❌ 错误: 同时生成 "世界" 和 "！"（不可能，因为 "！" 依赖 "世界"）
  ✅ 正确: 先生成 "世界"，再生成 "！"（必须按顺序）
```

**特点 2: 每次都要读取所有历史**
```
生成第 100 个 token 时:
  - 需要读取前 99 个 token 的 KV Cache
  - 内存读取量: 99 × KV_size_per_token

生成第 1000 个 token 时:
  - 需要读取前 999 个 token 的 KV Cache
  - 内存读取量: 999 × KV_size_per_token（非常大！）
```

**特点 3: 计算量小，但内存访问量大**
```
计算量:
  - 只计算 1 个新 token 的 Q, K, V
  - 计算量: O(1)（很小）

内存访问量:
  - 读取 N 个历史 token 的 KV Cache
  - 内存访问量: O(N)（很大，且随序列长度增长）

比例: 内存访问量 >> 计算量
```

#### 4.2.4 用"逐道做菜"的比喻（详细版）

```
场景：已经做了 99 道菜，现在要做第 100 道

═══════════════════════════════════════════════════════
Decode = 逐道做菜（不能批量）
═══════════════════════════════════════════════════════

为什么不能批量？
  - 第 100 道菜需要知道第 99 道菜的味道
  - 第 99 道菜需要知道第 98 道菜的味道
  - ... 必须按顺序做！

第 100 道菜的制作过程:

步骤 1: 准备新菜的食材（计算新 token 的 Q, K, V）
  - 时间: 很快（只准备 1 道菜的食材）
  - 类比: GPU 计算很快

步骤 2: 从冰箱拿所有旧菜的调料包（读取 KV Cache）
  - 需要拿: 前 99 道菜的调料包（K1, V1, K2, V2, ..., K99, V99）
  - 时间: 很慢！（要打开冰箱 99 次，拿 99 个调料包）
  - 类比: 内存读取很慢（内存带宽是瓶颈）

步骤 3: 混合调料包（计算注意力）
  - 时间: 很快（只是混合，计算量小）
  - 但必须等步骤 2 完成（等所有调料包都拿出来）

步骤 4: 做菜（通过模型层）
  - 时间: 很快

步骤 5: 把新菜的调料包存到冰箱（写入 KV Cache）
  - 时间: 很快（只存 1 个调料包）

总时间 = 步骤 2 的时间（最慢！）
瓶颈 = 从冰箱拿东西的速度（内存带宽）
```

#### 4.2.5 计算模式（详细版）

```
已生成序列: [token1, token2, ..., tokenN] (KV Cache 中)
要生成: token(N+1)

═══════════════════════════════════════════════════════
步骤 1: 计算新 token 的 Q, K, V
═══════════════════════════════════════════════════════
输入: tokenN（上一个生成的 token，或特殊起始符）
计算:
  Q_new = tokenN × W_Q  (查询矩阵)
  K_new = tokenN × W_K  (键矩阵)
  V_new = tokenN × W_V  (值矩阵)

特点:
  - 只计算 1 个 token
  - 计算量: 很小（O(1)）
  - 时间: 很快（< 0.1 ms）

═══════════════════════════════════════════════════════
步骤 2: 从 KV Cache 读取所有历史 KV（关键步骤！）
═══════════════════════════════════════════════════════
需要读取:
  - K1, K2, ..., KN  (N 个 K 向量)
  - V1, V2, ..., VN  (N 个 V 向量)

读取量:
  - 总大小: N × KV_size_per_token
  - 例子: 1000 tokens × 2.5 MB = 2.5 GB（很大！）

特点:
  - 必须读取所有历史（不能跳过）
  - 内存访问量: O(N)（随序列长度线性增长）
  - 时间: 很慢（1-10 ms，取决于序列长度）
  - 瓶颈: 内存带宽（Memory Bandwidth）

═══════════════════════════════════════════════════════
步骤 3: 计算注意力
═══════════════════════════════════════════════════════
计算:
  - Q_new 和 [K1, K2, ..., KN] 做点积
  - 得到注意力分数: [score1, score2, ..., scoreN]
  - Softmax 归一化
  - 用分数加权 [V1, V2, ..., VN]
  - 得到上下文向量: context = Σ(score_i × V_i)

特点:
  - 计算量: 中等（O(N)）
  - 但必须等步骤 2 完成（等 KV Cache 读取完成）
  - 时间: 相对较快（但受内存读取限制）

═══════════════════════════════════════════════════════
步骤 4: 通过 Transformer 层
═══════════════════════════════════════════════════════
context → LayerNorm → FFN → LayerNorm → ...
  (经过所有 Transformer 层，如 32, 40, 80 层)

特点:
  - 计算量: 中等
  - 时间: 较快

═══════════════════════════════════════════════════════
步骤 5: 生成 token 并更新 KV Cache
═══════════════════════════════════════════════════════
最后一层输出 → 词汇表概率分布 → 选择 token(N+1)
将 token(N+1) 的 K_new, V_new 写入 KV Cache

特点:
  - 写入量: 1 × KV_size_per_token（很小）
  - 时间: 很快
```

#### 4.2.6 内存访问模式（详细分析）

**内存访问的三个操作**：

```
1. 读取历史 KV Cache（最大！）
   - 读取量: N × KV_size_per_token
   - 操作: 从 HBM（高带宽内存）读取
   - 时间: 最长（瓶颈）

2. 写入新 token 的 KV（中等）
   - 写入量: 1 × KV_size_per_token
   - 操作: 写入 HBM
   - 时间: 中等

3. 读取模型权重（较小）
   - 读取量: 模型参数大小（但可以缓存）
   - 操作: 从 HBM 读取
   - 时间: 相对较小（因为权重可以复用）
```

**内存访问量对比**：
```
假设序列长度 = 1000 tokens
KV_size_per_token = 2.5 MB

读取历史 KV: 1000 × 2.5 MB = 2.5 GB  ← 最大！
写入新 KV: 1 × 2.5 MB = 2.5 MB      ← 很小
读取权重: ~140 GB（Llama-70B），但可以缓存

比例: 读取历史 KV >> 写入新 KV
```

**关键理解**：
- **内存读取量 >> 写入量 >> 计算量**
- **瓶颈是内存带宽**，不是 GPU 算力
- **序列越长，内存读取量越大**，瓶颈越明显

#### 4.2.7 为什么 Decode 必须串行？

**技术原因**：
```
生成 token(N+1) 需要:
  - tokenN 的输出（作为输入）
  - 所有历史 token 的 KV Cache（用于注意力）

生成 token(N+2) 需要:
  - token(N+1) 的输出（依赖 token(N+1)）
  - 所有历史 token 的 KV Cache（包括 token(N+1)）

结论: token(N+2) 必须等 token(N+1) 生成完成
→ 必须串行，不能并行
```

**类比理解**：
```
写文章:
  - 第 2 句话需要知道第 1 句话写的是什么
  - 第 3 句话需要知道前 2 句话写的是什么
  - 不能同时写第 2 句和第 3 句（必须按顺序）

生成文本:
  - 第 2 个 token 需要知道第 1 个 token 是什么
  - 第 3 个 token 需要知道前 2 个 token 是什么
  - 不能同时生成第 2 个和第 3 个 token（必须按顺序）
```

#### 4.2.8 关键总结

**Decode 阶段的本质**：
1. **逐个生成**：一次只生成 1 个 token，必须按顺序
2. **读取所有历史**：每次都要从 KV Cache 读取所有历史 token 的 K 和 V
3. **内存带宽瓶颈**：计算很快，但被内存读取速度限制
4. **序列越长越慢**：序列长度每增加 1，内存读取量就增加 KV_size_per_token

**与 Prefill 的对比**：
| 方面 | Prefill | Decode |
|------|---------|--------|
| **处理方式** | 并行处理所有 token | 串行处理，逐个生成 |
| **计算量** | 大（O(N²)） | 小（O(1)） |
| **内存访问** | 相对较小 | 很大（O(N)） |
| **瓶颈** | GPU 算力 | 内存带宽 |
| **GPU 利用率** | 高（并行计算） | 低（等待内存） |

**关键**：**计算很快，但被内存读取速度限制**！

---

## 5. 为什么 Decode 阶段更吃带宽？

### 5.1 核心原因

**Decode 阶段是内存带宽瓶颈**，原因如下：

#### 原因 1: 每次都要读取所有历史 KV

```
生成第 100 个 token 时:
  - 需要读取: 前 99 个 token 的 KV Cache
  - 内存读取量: 99 × KV_size_per_token

生成第 1000 个 token 时:
  - 需要读取: 前 999 个 token 的 KV Cache
  - 内存读取量: 999 × KV_size_per_token
```

**内存读取量随序列长度线性增长！**

#### 原因 2: 计算量小，但内存访问量大

```
Decode 阶段的计算:
  - 计算量: O(1)（只计算 1 个新 token）
  - 内存读取: O(N)（读取 N 个历史 token 的 KV）
  - 内存写入: O(1)（写入 1 个新 token 的 KV）

比例: 内存访问量 >> 计算量
```

#### 原因 3: GPU 算力 vs 内存带宽

**现代 GPU 的特点**：
- ✅ **算力强大**: 每秒可以执行大量计算（TFLOPS）
- ❌ **内存带宽有限**: 每秒只能传输有限的数据（TB/s）

**Decode 阶段的瓶颈**：
```
GPU 算力: 1000 TFLOPS（每秒 1000 万亿次浮点运算）
内存带宽: 2 TB/s（每秒 2 TB 数据传输）

Decode 计算需求:
  - 计算量: 很小（只计算 1 个 token）
  - 内存读取: 很大（读取所有历史 KV）

结果: GPU 算力有剩余，但被内存带宽限制！
```

### 5.2 定量分析

**以 Llama-2 70B 为例**：

```
KV_size_per_token = 2.5 MB

生成第 1000 个 token:
  - 需要读取: 999 × 2.5 MB = 2.5 GB
  - 需要写入: 1 × 2.5 MB = 2.5 MB
  - 总内存访问: 2.5 GB（主要是读取）

假设内存带宽: 2 TB/s
  - 读取时间: 2.5 GB / 2 TB/s = 1.25 ms

假设 GPU 算力: 1000 TFLOPS
  - 计算时间: 很小（< 0.1 ms）

瓶颈: 内存带宽！读取 KV Cache 的时间远大于计算时间
```

### 5.3 Prefill vs Decode 对比

| 阶段 | 计算模式 | 内存访问模式 | 瓶颈类型 |
|------|---------|-------------|---------|
| **Prefill** | 并行计算所有 token | 一次性读取输入，一次性写入 KV | **计算瓶颈**（Compute-Bound） |
| **Decode** | 逐个计算新 token | 每次读取所有历史 KV | **带宽瓶颈**（Memory-Bound） |

**Prefill 阶段**：
```
计算量: O(N²)（N×N 的注意力矩阵）
内存访问: O(N)（读取输入，写入 KV）
比例: 计算量 >> 内存访问量
→ GPU 算力是瓶颈
```

**Decode 阶段**：
```
计算量: O(1)（只计算 1 个新 token）
内存访问: O(N)（读取 N 个历史 token 的 KV）
比例: 内存访问量 >> 计算量
→ 内存带宽是瓶颈
```

### 5.4 为什么序列越长，带宽压力越大？

```
序列长度 = 100:
  - 每次读取: 100 × 2.5 MB = 250 MB
  - 内存带宽利用率: 250 MB / 2 TB/s = 0.0125%

序列长度 = 1000:
  - 每次读取: 1000 × 2.5 MB = 2.5 GB
  - 内存带宽利用率: 2.5 GB / 2 TB/s = 0.125%

序列长度 = 10000:
  - 每次读取: 10000 × 2.5 MB = 25 GB
  - 内存带宽利用率: 25 GB / 2 TB/s = 1.25%

序列长度 = 32000:
  - 每次读取: 32000 × 2.5 MB = 80 GB
  - 内存带宽利用率: 80 GB / 2 TB/s = 4%
```

**结论**: 序列越长，每次 decode 需要读取的 KV Cache 越多，内存带宽压力越大！

## 6. 优化方向

### 6.1 内存优化

#### a) Paged KV Cache（分页 KV 缓存）

**问题**: 传统连续内存分配容易产生碎片

**解决方案**: 将 KV Cache 分成固定大小的页（page），按需分配

**优势**:
- ✅ 减少内存碎片
- ✅ 支持更长的序列
- ✅ 更好的内存利用率

**SGLang 实现**: `python/sglang/srt/mem_cache/allocator.py`

#### b) KV Cache 量化

**方法**: 使用更低精度的数据类型存储 KV Cache

**例子**:
- FP16 → INT8: 内存占用减半
- FP16 → INT4: 内存占用减少 75%

**权衡**:
- ✅ 内存占用减少
- ❌ 可能影响精度（需要量化感知训练）

#### c) KV Cache 压缩

**方法**: 压缩存储 KV Cache，使用时解压

**权衡**:
- ✅ 内存占用减少
- ❌ 解压需要额外计算

### 6.2 带宽优化

#### a) 融合 Kernel（Fused Kernel）

**方法**: 将多个操作融合成一个 kernel，减少内存访问

**例子**: FlashAttention
- 传统方式: 计算 QK^T → 写入 HBM → 读取 → 计算 softmax → 写入 HBM → 读取 → 计算 attention
- FlashAttention: 所有操作融合，中间结果保留在 SRAM

**优势**:
- ✅ 减少 HBM（高带宽内存）访问
- ✅ 提高内存带宽利用率

#### b) 预取（Prefetching）

**方法**: 提前将下一个 token 需要的 KV Cache 加载到缓存

**优势**:
- ✅ 隐藏内存访问延迟
- ✅ 提高 GPU 利用率

#### c) 批处理优化

**方法**: 将多个请求的 decode 合并，提高内存访问效率

**优势**:
- ✅ 提高内存带宽利用率
- ✅ 减少内存访问次数

### 6.3 架构优化

#### a) Prefill-Decode 分离（PD Disaggregation）

**方法**: 将 Prefill 和 Decode 阶段分离到不同的 GPU

**原因**:
- Prefill: 需要高算力（Compute-Bound）
- Decode: 需要高内存带宽（Memory-Bound）

**优势**:
- ✅ 针对性地优化资源分配
- ✅ 提高整体资源利用率

**SGLang 实现**: `python/sglang/srt/disaggregation/`

#### b) 前缀缓存（Prefix Caching）

**方法**: 缓存相同前缀的 KV Cache，避免重复计算

**SGLang 实现**: RadixAttention

**优势**:
- ✅ 减少 Prefill 计算量
- ✅ 减少内存带宽压力（共享前缀）

## 7. 总结

### 7.1 关键要点

1. **KV Cache 是什么**:
   - 存储 Transformer 注意力机制中 K 和 V 向量的内存区域
   - 避免重复计算，大幅提升推理效率

2. **内存占用**:
   - 线性增长: O(layers × heads × head_dim × seq_len)
   - 长上下文会占用大量显存

3. **Prefill vs Decode**:
   - Prefill: 计算密集型，并行处理所有 token
   - Decode: 带宽密集型，每次读取所有历史 KV

4. **为什么 Decode 更吃带宽**:
   - 每次都要读取所有历史 token 的 KV Cache
   - 内存读取量随序列长度线性增长
   - GPU 算力有剩余，但被内存带宽限制
