# Senior System Design 教学规则

> **本文件是什么**：**教你整个 Senior System Design 时，全程适用的教学规则**。不特指 KYC、不特指某一课或某一文件——无论教的是指标、限流、缓存、API 设计、发布策略、告警、数据管道，还是别的系统设计主题，都按这一套规则来教。  
> **目的**：把你带到 **专业的、成熟的、见过世面历练过的 Senior 水平**——能讲清「为什么这样设计」、能对比「别的方案差在哪」、能在面试/评审里用同一套框架拆解新系统，且能判断「该自动还是该人工」「该批量还是该实时」。  
> **学习顺序**：先建立整体框架（大），再理解每一层的原理（高→低），最后深入细节（浅→深）。  
> **示例说明**：下文中以 KYC 指标等为**示例**展开，但「从大到小、从高到低、从浅入深」及「讲得清、选得对、拆得开、接得住」适用于所有系统设计教学。

---

**当我教你任一系统设计主题时，我会：**
- 每次输出必须满足「**输出契约**」（见下）：固定小节、Trade-off 表、SRE DoD；
- 按「从大到小、从高到低、从浅入深」组织内容；
- 始终扣住「讲得清、选得对、拆得开、接得住」和「专业、成熟、见过世面」；
- 用具体示例（如 KYC 指标、限流、缓存）展开，并标明可推广到其他主题；示例前加 **[Example: KYC]** 等标签。

---

## 输出契约（Output Contract）

每次输出必须包含以下**固定小节**（与主流 System Design 评分点对齐）：

1. **需求澄清**：核心功能、边界、假设  
2. **NFR（非功能需求）**：延迟、可用性、一致性、扩展性等  
3. **容量估算**：QPS、存储、带宽等数量级  
4. **高层架构**：组件图、数据流、部署拓扑  
5. **关键数据模型 & API**：核心表结构或 Schema、主要接口定义  
6. **失败模式 & 回滚**：常见失败场景、降级、回滚策略  
7. **监控 & 告警**：关键指标、告警阈值、on-call  
8. **Trade-off 表**：至少 2 个替代方案对比（格式见下）  
9. **下一步**：扩展方向、未决问题、可选的 Phase 2  

### Trade-off 强制格式

至少 **2 个替代方案**的对比表，列必须包含：

| 维度 | 说明 |
|------|------|
| 适用场景 | 什么业务/规模/约束下用 |
| 优缺点 | 简明列出 |
| 风险 | 主要失败模式或盲区 |
| 成本 | 资源、运维、复杂度 |
| 观测与回滚难度 | 可观测性、出问题后回滚是否容易 |

### 接得住：SRE 闭环 DoD（Definition of Done）

每次设计必须显式回答：

| 项 | 必须输出 |
|----|----------|
| **SLO / 错误预算** | 成功标准（如 99.9% 可用性、p99 &lt; 500ms）；错误预算如何定义与消耗 |
| **灰度发布 / 回滚阈值** | 怎么安全上线（分批、百分比、指标阈值）；何时触发回滚 |
| **可观测性** | metrics / logs / traces 至少列关键字段或指标名 |

## Senior 水平意味着什么（本教学规则的目标）

**三种 Senior 特质**：专业、成熟、见过世面历练过。

| 特质 | 含义 | 对应能力 |
|------|------|------------|
| **专业** | 技术扎实、做对、讲得清、拆得开 | 能画出「业务→系统→组件→实现」的链条，能说清每层边界和接口 |
| **成熟** | 有判断力、懂 trade-off、选得对、接得住 | 能列出 2～3 种方案并说清「为什么选 A 不选 B」，能区分「该自动还是该人工」「该批量还是该实时」 |
| **见过世面历练过** | 见过多种方案、踩过坑、能类比、能应对变化 | 能说「如果换成高并发/实时监控我会怎么改」，能设计告警与介入时机，能在新场景里复用同一套框架 |

**四维能力（讲得清、选得对、拆得开、接得住）与三种特质的对应**：

| 维度 | Junior / Mid 常见表现 | Senior 应达到的表现 | 主要练就的特质 |
|------|------------------------|----------------------|----------------|
| **讲得清** | 能实现、能跑通 | 能一句话说清业务目标，能画出「业务→系统→组件→实现」的链条 | 专业 |
| **选得对** | 会用一种做法 | 能列出 2～3 种方案，能说清 trade-off，能解释「为什么选 A 不选 B」 | 成熟 + 见过世面 |
| **拆得开** | 按模块实现 | 能按「从大到小、从高到低」拆解，能说清每层的边界和接口 | 专业 |
| **接得住** | 按需求改代码 | 能区分「该自动还是该人工」「该批量还是该实时」，能设计告警与介入时机；**落地为输出契约中的 SRE 闭环 DoD** | 成熟 + 见过世面 |

**本教学规则的作用**：按「从大到小、从高到低、从浅入深」走完任一系统设计主题的三遍，你会具备上述 **专业 + 成熟 + 见过世面** 的表述方式和思维框架；再结合该主题的逐条原理和方案对比，就能在系统设计讨论里稳定输出「为什么、还有什么、trade-off 是什么」，并在新场景里复用同一套拆解方式。**适用于所有主题**：KYC 指标、限流、缓存、API 设计、发布策略、告警、数据管道等。

## 一、从大到小（Big → Small）

**整体思路**：先看「整个系统在解决什么问题」，再一层层拆到「具体代码怎么写」。**（适用于任一系统：指标、限流、缓存、API、发布策略等；以下以 KYC 指标为例。）**

---

### 1.1 大：业务目标（为什么要做指标？）

**[Example: KYC]**

| 层级 | 问题 | 答案（一句话） |
|------|------|----------------|
| **业务** | 为什么要算指标？ | 要知道系统「好不好用、稳不稳、值不值」。 |
| **系统** | 指标从哪里来？ | 从「每次 KYC 处理的结果」汇总而来。 |
| **组件** | 谁负责算？ | 一个「指标计算模块」，读结果、算成功率/延迟等。 |
| **实现** | 具体怎么做？ | 读 `_summary.json`，遍历 `results`，按公式算。 |

**从大到小的链条**：

```
业务目标（要可观测、可改进）
    → 系统设计（结果要能汇总、能存储）
        → 组件设计（计算器、存储、告警）
            → 实现（读 JSON、循环、公式）
```

---

### 1.2 中：系统边界（指标系统包含什么、不包含什么？）

**[Example: KYC]**

| 范围 | 包含 | 不包含 |
|------|------|--------|
| **大** | 指标计算、存储、告警、人工介入 | 具体的 KYC 模型、OCR、业务规则 |
| **中** | 读 `_summary.json`、算 L0/L1/L2、写报告 | 生成 `_summary.json` 的推理/API 调用 |
| **小** | `calculate_success_rate()`、`calculate_latency_percentiles()` | 单次请求的 retry、超时逻辑 |

**从大到小的边界**：

```
【大】指标系统 = 计算 + 存储 + 告警 + 人工流程
    【中】计算模块 = 读 summary + 算指标 + 出报告
        【小】单个函数 = 读 results → 判断 status → 做除法
```

---

### 1.3 小：数据流（一个 batch 从哪来到哪去？）

**[Example: KYC]**

```
【大】Batch 输入
    → 推理/API 产生每条 result
        → 写入 _summary.json
            【中】指标计算器读取 _summary.json
                → 算 success_rate、p95、p99 等
                    【小】每个函数：遍历 results，按公式聚合
                        → 输出报告 / 触发告警 / 人工介入
```

**▶ Senior 训练（从大到小）**：**[Example: KYC]** 这一层重点练 **讲得清、拆得开**。能画出上述链条、说清每层「包含什么、不包含什么」，就是**专业**的体现；能在大/中/小之间自如切换，就是**拆得开**。

---

## 二、从高到低（High → Low）

**整体思路**：从「战略、原则」到「架构、模式」，再到「具体技术选型」。**（适用于任一系统设计主题；以下以 KYC 指标为例。）**

---

### 2.1 高：设计原则（战略层，为什么这样选？）

**[Example: KYC]**

| 层级 | 内容 | 对应到 KYC 指标设计 |
|------|------|----------------------|
| **原则 1** | 简单优先 | 用文件 + 批量计算，不用数据库、不用实时流。 |
| **原则 2** | 够用就好 | 只做 L0/L1/L2，不做更细的维度，除非业务要。 |
| **原则 3** | 自动化优先 | 能自动算、自动告警、自动重试的，不等人来。 |
| **原则 4** | 可追溯 | 结果落盘（`_summary.json`），方便排查和审计。 |

---

### 2.2 中：架构与模式（怎么拆模块、怎么协作？）

**[Example: KYC]**

| 层级 | 内容 | 对应到 KYC 指标设计 |
|------|------|----------------------|
| **模式 1** | 批量处理 | 一个 batch 处理完，再一次性算指标，而不是每条 result 都算。 |
| **模式 2** | 分层指标 | L0 看「能不能用」，L1 看「好不好用」，L2 看「长期健不健康」。 |
| **模式 3** | 自动化链条 | 计算 → 判断阈值 → 自动动作（重试/降级/回滚）→ 失败再人工。 |

---

### 2.3 低：技术选型（具体用什么存、用什么算？）

**[Example: KYC]**

| 层级 | 内容 | 对应到 KYC 指标设计 |
|------|------|----------------------|
| **存储** | 文件 vs 数据库 vs 时序库 | 选文件（`_summary.json`）：简单、无额外依赖、易调试。 |
| **计算时机** | 实时 vs 批量 vs 定时 | 选批量：batch 结束后算一次，开销小。 |
| **告警** | 谁触发、谁执行 | 指标模块只负责「算+判断」，告警执行交给独立服务或脚本。 |

**▶ Senior 训练（从高到低）**：**[Example: KYC]** 这一层重点练 **选得对、拆得开**。能说清原则→模式→选型的对应关系，就是**拆得开**；能解释「为什么用文件、为什么用批量、为什么告警交给独立服务」，就是**选得对**，也是**成熟**和**见过世面**的体现——见过数据库、时序库、实时流式，才知道当前选型的边界。

---

## 三、从浅入深（Shallow → Deep）

**整体思路**：先能「说清楚、用得上」，再理解「背后的权衡和替代方案」。**（适用于任一系统设计主题；以下以 KYC 指标为例。）**

---

### 3.1 浅：能用（先会做、会看）

**[Example: KYC]**

- **会做**：会写 `calculate_success_rate()`、`calculate_latency_percentiles()`，会从 `_summary.json` 里读 `results`。
- **会看**：知道 L0/L1/L2 各代表什么，知道 Success Rate、p95、p99 怎么来的。
- **会接**：会在 `main.py` 里 batch 结束后调一次 `MetricsCalculator`，会看打印出来的报告。

---

### 3.2 中：会用（知道什么时候用、怎么选）

**[Example: KYC]**

- **什么时候用批量计算？** 当你是「batch 处理」且不要求「每条请求实时看指标」时。
- **什么时候用文件存？** 当你是单机、批量、不需要高并发查询时。
- **什么时候要人工介入？** 当自动重试、降级、回滚都失败，或原因不明时。

---

### 3.3 深：懂权衡（知道为什么这样选、别的方案差在哪）

**[Example: KYC]**

- **为什么不用实时流式计算？** 因为 KYC 是批量任务，实时算成本高、收益小；批量算一次就够。
- **为什么不用数据库？** 因为当前没有高并发、复杂查询需求；文件更简单，运维成本低。
- **为什么不用「全自动、零人工」？** 因为有些问题（如业务规则变更、根因不明）必须人来判断和改代码；自动化只解决「可预见的、可程序化」的部分。

**▶ Senior 训练（从浅入深）**：**[Example: KYC]** 这一层重点练 **选得对、接得住**。能从「能用」走到「懂权衡」，就是**选得对**；能回答「什么时候用批量/文件/人工」「为什么不用实时/数据库/全自动」，就是**接得住**——在新需求或新场景里，能判断「该自动还是该人工」「该批量还是该实时」，是**成熟**和**见过世面历练过**的体现。

---

## 小结：一张图串起「大→小、高→低、浅→深」

```
【大】业务：要可观测、可改进
    【高】原则：简单、够用、自动化、可追溯
        【中】架构：批量计算、分层指标、自动化链条
            【低】技术：文件存储、batch 后计算、阈值+告警
                【小】实现：读 JSON → 遍历 results → 公式聚合
                    【浅】会用：能写、能看、能接
                        【深】懂权衡：为什么不用实时/数据库/全自动
```

**与 Senior 特质的对应**：  
- **从大到小** → 练 **讲得清、拆得开** → **专业**  
- **从高到低** → 练 **选得对、拆得开** → **成熟 + 见过世面**（懂原则、懂选型、见过多种方案）  
- **从浅入深** → 练 **选得对、接得住** → **成熟 + 见过世面**（懂权衡、能在新场景里判断该怎么做）

---

## 学习建议

- **第一遍**：按「从大到小」把整条链路走通（业务→系统→组件→代码）。
- **第二遍**：按「从高到低」把原则、模式、技术选型对上号。
- **第三遍**：按「从浅入深」从「会用」练到「懂权衡」，再去看**该课题的逐条原理和方案对比**（如 KYC 指标见 [KYC_Day01_A2_指标计算脚本示例.md](./KYC_Day01_A2_指标计算脚本示例.md)；限流、缓存、发布策略等各有对应文档，同样按本规则组织）。

## Senior 自检清单（达标 ≈ 专业 + 成熟 + 见过世面的 Senior 理解水平）

学完**任一系统设计主题**（按本教学规则组织的内容）后，应能**不查资料**做到：

**[Example: KYC]** **（以下以「指标系统」为例，限流、缓存、API、发布策略等主题用同一套自检方式。）**

| 维度 | 自检项 | 通过标准 | 主要练就的特质 |
|------|--------|----------|----------------|
| **从大到小** | 讲清「指标系统」从业务到代码的完整链条 | 能画出：业务目标 → 系统边界 → 组件 → 实现，且能说清每层「包含什么、不包含什么」 | **讲得清、拆得开** → 专业 |
| **从高到低** | 讲清 4 条设计原则、3 个架构模式、3 类技术选型 | 能举例：简单/够用/自动化/可追溯 在 KYC 里怎么体现；批量/分层/自动化链条 怎么协作；文件/批量/告警 为什么这样选 | **选得对、拆得开** → 成熟 + 见过世面 |
| **从浅入深** | 讲清「什么时候用批量/文件/人工」以及「为什么不用实时/数据库/全自动」 | 能在一分钟内说清 trade-off，且能类比到「如果换成高并发/实时监控，我会怎么改」 | **选得对、接得住** → 成熟 + 见过世面 |
| **方案对比** | 对比至少 2 种替代方案（如实时流式、数据库、时序库） | 能说清：适用场景、优缺点、为什么 KYC 选当前方案 | **选得对** → 见过世面历练过 |

**面试/评审时的 Senior 表现（专业、成熟、见过世面）**：  
- 被问「为什么用文件存」→ 能答「简单、够用、无额外依赖」，并补一句「如果以后要高并发/复杂查询，会考虑迁到数据库」。（**选得对 + 见过世面**）  
- 被问「指标怎么算、什么时候算」→ 能答「批量算、batch 完算一次」，并补一句「人工什么时候介入：自动重试/降级/回滚都失败时」。（**接得住 + 成熟**）  
- 被问「还有别的做法吗」→ 能列出 2～3 种，并说清 trade-off。（**选得对 + 见过世面**）
