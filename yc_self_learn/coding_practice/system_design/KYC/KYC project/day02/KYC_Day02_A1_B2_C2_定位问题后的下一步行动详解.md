## Definition（定义）

**核心问题**：**如果知道是 Schema Validation 这里出错了，Senior 工程师下一步去干嘛？**

**核心答案**：
- ✅ **步骤 1**：**查看 Logs（日志）**，获取详细的错误信息和上下文
- ✅ **步骤 2**：**分析根因**，了解为什么会失败（缺少字段、格式不对等）
- ✅ **步骤 3**：**快速止血**（如果紧急），临时方案降低错误率
- ✅ **步骤 4**：**制定解决方案**，长期修复方案（修改 Prompt、调整 Schema 等）

**关键理解**：
- ✅ **Traces 只是定位问题**：告诉你哪个环节失败了
- ✅ **Logs 才是分析根因**：告诉你为什么会失败
- ✅ **Senior 的价值**：快速定位 → 快速止血 → 长期修复

## 🎯 核心问题

### 定位问题后的下一步行动

**场景**：你在 Traces 中看到 Span 5（Schema Validation）失败了。

**下一步行动**：

```
1. 查看 Logs（获取详细错误信息）
   ↓
2. 分析根因（了解为什么会失败）
   ↓
3. 快速止血（如果紧急）
   ↓
4. 制定解决方案（长期修复）
```

---

## 📊 详细步骤

### 步骤 1：查看 Logs（获取详细错误信息）

**目的**：获取详细的错误信息和上下文，了解为什么会失败。

**操作步骤**：

1. **使用 trace_id 查询 Logs**：
   ```
   打开 Logs Dashboard（Kibana/Datadog Logs）
   输入 trace_id: trace-abc123
   或
   span_name: "Schema Validation" AND error_code: "SCHEMA_VALIDATION_FAILED"
   ```

2. **查看 Logs 详情**：
   ```json
   {
     "timestamp": "2024-01-15T10:30:00Z",
     "level": "ERROR",
     "trace_id": "trace-abc123",
     "span_name": "Schema Validation",
     "error_code": "SCHEMA_VALIDATION_FAILED",
     "error_message": "Schema validation failed: missing required field 'user_name'",
     "llm_output": {
       "status": "success",
       "result": {
         "user_id": "u123",
         "birth_date": "1990-01-01"
         // 缺少 "user_name" 字段
       }
     },
     "schema_expected": {
       "user_id": "string",
       "user_name": "string",  // 必需字段
       "birth_date": "string"
     },
     "schema_actual": {
       "user_id": "string",
       "birth_date": "string"
       // 缺少 "user_name" 字段
     },
     "prompt_hash": "abc123...",
     "model_version": "qwen2.5-vl-32b-v1.0"
   }
   ```

3. **提取关键信息**：
   - ✅ **错误消息**：`"missing required field 'user_name'"`
   - ✅ **LLM 输出**：缺少 `user_name` 字段
   - ✅ **Schema 要求**：`user_name` 是必需字段
   - ✅ **模型版本**：`qwen2.5-vl-32b-v1.0`
   - ✅ **Prompt 版本**：`prompt_hash: abc123...`

**关键点**：
- ✅ **Traces 告诉你"哪里"**：Schema Validation 失败了
- ✅ **Logs 告诉你"为什么"**：缺少 `user_name` 字段
- ✅ **Logs 包含上下文**：LLM 输出、Schema 要求、模型版本等

---

### 步骤 2：分析根因

**目的**：了解为什么会失败，找出根本原因。

**分析过程**：

**根因分析**：
```
1. LLM 推理成功（status: "success"）
   - LLM 输出了结果，但缺少 "user_name" 字段

2. Schema 验证失败
   - Schema 要求 "user_name" 是必需字段
   - LLM 输出缺少 "user_name" 字段
   - 导致验证失败

3. 可能的原因：
   - Prompt 没有明确要求输出 "user_name" 字段
   - 模型版本 qwen2.5-vl-32b-v1.0 可能对某些文档格式识别不准确
   - Schema 要求过于严格（将 user_name 设为必需字段）
```

**结论**：
- ✅ **根因**：LLM 输出缺少 `user_name` 字段
- ✅ **直接原因**：Prompt 没有明确要求输出 `user_name` 字段
- ✅ **深层原因**：Schema 要求过于严格，或者 LLM 对某些文档格式识别不准确

---

### 步骤 3：快速止血（如果紧急）

**目的**：临时降低错误率，保证系统可用性。

**场景**：如果 Error Rate 很高（> 5%），需要立即降低错误率。

**快速止血方案**：

**方案 1：降低 Schema 验证严格度**（最快）
```
临时方案：将 user_name 从必需字段改为可选字段

操作：
1. 修改 Schema 配置：
   - 将 "user_name" 从 required 改为 optional
   - 或降低 validator_strictness 从 "high" 到 "medium"

2. 立即生效（不需要重启服务）：
   - 使用 Feature Flag 或配置中心
   - 修改配置后，新请求立即生效

3. 观察错误率：
   - 监控 Error Rate 是否下降
   - 如果下降，说明方案有效
```

**方案 2：触发 Fallback**（保证可用性）
```
临时方案：Schema 验证失败时，触发 Fallback 或人工审核

操作：
1. 修改验证逻辑：
   - 如果 Schema 验证失败，不直接失败
   - 而是标记为 needs_review，转人工审核

2. 保证可用性：
   - 请求不会失败（只是转人工）
   - Error Rate 下降

3. 观察错误率：
   - 监控 Error Rate 是否下降
   - 监控 needs_review 的数量
```

**方案 3：回滚到上一个版本**（如果问题是由新版本引起的）
```
临时方案：回滚到上一个正常运行的版本

操作：
1. 检查是否是版本问题：
   - 查看 model_version 和 prompt_version
   - 如果是最新版本引起的，回滚

2. 回滚操作：
   - 切换到上一个正常运行的版本
   - 使用灰度发布或配置中心

3. 观察错误率：
   - 监控 Error Rate 是否下降
   - 如果下降，说明是版本问题
```

**关键点**：
- ✅ **快速止血 = 临时方案**：目标是快速降低错误率，不是彻底解决问题
- ✅ **观察效果**：监控错误率是否下降，验证方案是否有效
- ✅ **记录决策**：记录临时方案的决策和效果，为长期修复做准备

---

### 步骤 4：制定解决方案（长期修复）

**目的**：彻底解决问题，防止再次发生。

**长期修复方案**：

**方案 1：修改 LLM Prompt**（推荐）
```
长期方案：在 Prompt 中明确要求输出 "user_name" 字段

操作：
1. 修改 Prompt：
   - 在 Prompt 中添加 "必须包含 user_name 字段"
   - 明确要求输出的字段列表
   - 示例格式

2. 测试验证：
   - 在测试环境测试新 Prompt
   - 验证 LLM 输出是否包含所有必需字段
   - 如果有效，部署到生产环境

3. 监控效果：
   - 监控 Error Rate 是否下降
   - 监控 LLM 输出质量
   - 如果有效，说明问题解决

优点：
- 解决根本问题（从源头解决）
- 提高自动化率
- 不影响业务逻辑

缺点：
- 需要修改 Prompt（可能需要测试多个版本）
- 可能需要调整示例格式
```

**方案 2：调整 Schema**（如果业务允许）
```
长期方案：将 user_name 改为可选字段

操作：
1. 与业务团队确认：
   - 确认 user_name 是否真的必需
   - 如果业务允许，改为可选字段

2. 修改 Schema：
   - 将 "user_name" 从 required 改为 optional
   - 更新验证逻辑

3. 测试验证：
   - 在测试环境测试
   - 验证业务逻辑是否正确

优点：
- 简单直接
- 不需要修改 Prompt

缺点：
- 可能影响业务逻辑（如果 user_name 真的是必需的）
- 需要业务团队确认
```

**方案 3：优化模型**（如果模型问题）
```
长期方案：升级模型版本或优化模型

操作：
1. 分析模型性能：
   - 检查模型版本 qwen2.5-vl-32b-v1.0 的性能
   - 如果问题较多，考虑升级模型

2. 测试新模型：
   - 在测试环境测试新模型
   - 验证输出质量是否提升

3. 部署新模型：
   - 使用灰度发布部署新模型
   - 监控错误率和输出质量

优点：
- 解决模型问题（如果确实是模型问题）
- 提升整体输出质量

缺点：
- 成本高（可能需要更强的模型）
- 需要大量测试
```

**关键点**：
- ✅ **长期修复 = 彻底解决**：目标是彻底解决问题，防止再次发生
- ✅ **测试验证**：在测试环境测试，验证方案是否有效
- ✅ **监控效果**：部署后监控错误率，验证问题是否解决

## 💡 实际场景：照片反光导致 LLM 无法识别

### 场景描述

**问题**：用户上传的照片有很多反光，导致 LLM 读不出来 `user_name`，Schema Validation 失败。

**从 Traces 看到**：
```
Span 5: Schema Validation
├─ status: ERROR
└─ error_code: SCHEMA_VALIDATION_FAILED
```

---

### 步骤 1：查看 Logs（获取详细错误信息）

**操作**：
```json
{
  "trace_id": "trace-abc123",
  "span_name": "Schema Validation",
  "error_code": "SCHEMA_VALIDATION_FAILED",
  "error_message": "Schema validation failed: missing required field 'user_name'",
  "llm_output": {
    "status": "success",
    "result": {
      "user_id": "u123",
      "birth_date": "1990-01-01"
      // 缺少 "user_name" 字段
    }
  },
  "file_id": "doc_001.jpg",
  "file_url": "s3://bucket/doc_001.jpg"  // 可以查看原始图片
}
```

**关键信息**：
- ✅ **错误消息**：`"missing required field 'user_name'"`
- ✅ **LLM 输出**：缺少 `user_name` 字段
- ✅ **文件 ID**：`doc_001.jpg`（可以查看原始图片）

**查看原始图片**：
```
1. 下载图片：s3://bucket/doc_001.jpg
2. 查看图片：发现照片有大量反光，遮挡了姓名区域
3. 确认根因：照片质量问题，导致 LLM 无法识别 user_name
```

---

### 步骤 2：分析根因

**根因分析**：

**单个用户问题 vs 系统性问题**：
```
查看相关错误：
1. 统计：查询最近 1 小时的 SCHEMA_VALIDATION_FAILED 错误
2. 分析：
   - 只有 1 个用户的照片有反光问题 → 单个用户问题
   - 多个用户的照片都有反光问题 → 系统性问题（可能是上传流程、设备问题）

结果：
- 如果是单个用户 → 人工审核处理即可
- 如果是系统性问题 → 需要优化预处理或上传流程
```

**根因**：
```
1. **数据质量问题**：照片有反光，遮挡了姓名区域
2. **LLM 无法识别**：反光导致 LLM 无法识别 user_name
3. **Schema 验证失败**：缺少必需字段，验证失败

可能的深层原因：
- 用户上传时使用了有反光的设备（手机摄像头、扫描仪）
- 上传流程没有质量检查（没有检测反光、模糊等问题）
- 预处理步骤不够（没有去反光、增强对比度等）
```

---

### 步骤 3：快速止血（如果紧急）

**场景**：如果是系统性问题（多个用户都有反光问题），需要快速降低错误率。

**方案 1：触发 Fallback 或人工审核**（推荐）
```
临时方案：Schema 验证失败时，转人工审核

操作：
1. 修改验证逻辑：
   - 如果 Schema 验证失败，不直接失败
   - 而是标记为 needs_review，转人工审核
   - 人工审核员可以手动识别 user_name

2. 保证可用性：
   - 请求不会失败（只是转人工）
   - Error Rate 下降

3. 观察错误率：
   - 监控 Error Rate 是否下降
   - 监控 needs_review 的数量

时间：10 分钟
```

**方案 2：暂时降低 Schema 严格度**（临时）
```
临时方案：将 user_name 从必需字段改为可选字段

操作：
1. 修改 Schema 配置：
   - 将 "user_name" 从 required 改为 optional
   - 使用 Feature Flag 立即生效

2. 观察错误率：
   - 监控 Error Rate 是否下降
   - 注意：可能影响业务逻辑（如果 user_name 真的是必需的）

时间：5 分钟

注意：这只是临时方案，不能长期使用（如果 user_name 真的是必需的）
```

---

### 步骤 4：制定解决方案（长期修复）

**方案 1：优化图片预处理**（推荐）

**目的**：在 LLM 推理之前，对图片进行预处理，去除反光、增强对比度等。

**操作**：
```
1. 添加反光检测：
   - 使用图像处理算法检测反光区域
   - 如果反光严重，直接标记为需要人工审核

2. 图像增强：
   - 去反光处理（使用图像处理算法）
   - 增强对比度
   - 锐化处理

3. 质量检查：
   - 在上传时检查图片质量（反光、模糊、清晰度等）
   - 如果质量不达标，提示用户重新上传

4. 测试验证：
   - 在测试环境测试预处理效果
```

**优点**：

- ✅ **成本低**：只需要修改 Prompt，不需要额外开发

**缺点**：
- ⚠️ **效果有限**：如果反光太严重，LLM 仍然无法识别

**方案 3：多模态增强**（高级）

**目的**：使用 OCR + LLM 的组合，提高识别率。

**缺点**：

**优点**：

- ✅ **改善用户体验**：提示用户如何拍好照片

**缺点**：

## 📊 业界通用解决方案（照片反光问题）

### 业界通用方案

**场景**：用户符合大部分需求，但上传的照片有反光，导致 LLM 无法识别信息。

**业界通用解决方案**：

---

### 方案 1：用户端实时提示（最常见）

**做法**：
- ✅ **上传时实时检测**：在用户拍照或上传时，实时检测图片质量（反光、模糊、清晰度等）
- ✅ **即时反馈**：如果检测到反光或质量问题，立即提示用户重新拍照
- ✅ **拍照指南**：提供清晰的拍照指南（如何避免反光、如何拍清晰等）

**例子**（微信/支付宝的 KYC）：
```
1. 用户拍照时，实时显示反光检测结果
2. 如果检测到反光，提示："检测到反光，请避免强光直射"
3. 提供拍照指南："
   - 避免强光直射
   - 确保光线均匀
   - 保持摄像头与证件平行"
```

**Trade-off**：
- ✅ **优点**：
  - **从源头解决问题**：在用户端就避免质量问题
  - **用户体验好**：即时反馈，用户知道如何改进
  - **减少成本**：避免后续处理失败，减少 LLM 调用成本
- ⚠️ **缺点**：
  - **可能影响转化率**：如果检测太严格，用户可能放弃上传
  - **开发成本**：需要在客户端集成图像质量检测（可能需要第三方 SDK）
  - **需要平衡**：需要在质量检查和用户体验之间找到平衡

---

### 方案 2：图像预处理（服务端处理）

**做法**：
- ✅ **上传后预处理**：在服务端对图片进行预处理（去反光、增强对比度、锐化等）
- ✅ **自动修复**：尝试自动修复反光问题，提升图片质量
- ✅ **质量检查**：如果预处理后质量仍不达标，转人工审核

**例子**（支付宝/微信的 KYC）：
```
1. 用户上传图片后，服务端自动预处理
2. 图像增强：
   - 去反光处理（使用图像处理算法）
   - 增强对比度
   - 锐化处理
3. 如果预处理后质量仍不达标，转人工审核
```

**Trade-off**：
- ✅ **优点**：
  - **自动化处理**：自动修复反光问题，提升识别率
  - **用户无感知**：用户不需要重新拍照，体验好
  - **处理灵活**：可以结合多种图像处理算法
- ⚠️ **缺点**：
  - **处理时间**：预处理会增加处理时间（可能增加 1-2 秒）
  - **处理成本**：图像处理需要计算资源（CPU/GPU）
  - **效果有限**：如果反光太严重，预处理可能无法完全修复

---

### 方案 3：多模态识别（OCR + LLM 组合）

**做法**：
- ✅ **OCR 优先**：先用 OCR（光学字符识别）提取文本信息
- ✅ **LLM 辅助**：如果 OCR 能识别，直接使用；如果 OCR 无法识别，再用 LLM 推理
- ✅ **人工兜底**：两者都失败时，转人工审核

**例子**（银行 KYC、金融平台）：
```
1. 上传图片后，先用 OCR 提取文本
2. 如果 OCR 能识别 user_name，直接使用
3. 如果 OCR 无法识别（反光严重），再用 LLM 推理
4. 如果 LLM 也无法识别，转人工审核
```

**Trade-off**：
- ✅ **优点**：
  - **识别率高**：OCR + LLM 组合，识别率更高
  - **互补**：OCR 擅长清晰文本，LLM 擅长复杂场景
  - **成本优化**：OCR 便宜，LLM 贵，优先用 OCR 可以降低成本
- ⚠️ **缺点**：
  - **成本增加**：需要 OCR 服务（虽然比 LLM 便宜，但仍有成本）
  - **开发复杂**：需要集成 OCR 服务，处理两种结果
  - **处理时间**：OCR + LLM 可能增加处理时间

---

### 方案 4：人工审核兜底（传统方案）

**做法**：
- ✅ **自动识别失败时**：如果自动识别失败（包括反光、模糊等情况），转人工审核
- ✅ **人工识别**：人工审核员可以手动识别信息，或联系用户重新上传
- ✅ **持续优化**：收集人工审核的数据，优化自动识别模型

**例子**（传统 KYC 平台）：
```
1. 自动识别失败时（包括反光、模糊等），转人工审核
2. 人工审核员查看图片，手动识别信息
3. 如果图片质量太差，审核员联系用户重新上传
4. 收集数据，优化自动识别模型
```

**Trade-off**：
- ✅ **优点**：
  - **识别率高**：人工审核识别率高（几乎 100%）
  - **用户体验好**：用户不需要重新拍照，有人工兜底
  - **持续优化**：可以收集数据，优化自动识别模型
- ⚠️ **缺点**：
  - **成本高**：人工审核成本高（每人每分钟几元到几十元）
  - **处理时间长**：人工审核需要时间（可能几分钟到几小时）
  - **扩展性差**：人工审核无法快速扩展

---

### 方案 5：混合方案（业界最佳实践）

**做法**：
- ✅ **用户端检测**：上传时实时检测图片质量，如果质量不达标，提示用户重新拍照
- ✅ **服务端预处理**：上传后自动预处理（去反光、增强对比度等）
- ✅ **多模态识别**：OCR + LLM 组合识别
- ✅ **人工兜底**：自动识别失败时，转人工审核

**例子**（支付宝、微信、银行 KYC）：
```
1. 用户端：上传时实时检测，提示用户避免反光
2. 服务端：自动预处理（去反光、增强对比度）
3. 识别：OCR + LLM 组合识别
4. 兜底：自动识别失败时，转人工审核
```

**Trade-off**：
- ✅ **优点**：
  - **多层次保护**：从用户端到服务端，多个环节保障质量
  - **识别率高**：多模态识别 + 人工兜底，识别率高
  - **用户体验好**：用户端提示，服务端自动处理
- ⚠️ **缺点**：
  - **开发复杂**：需要多个环节配合开发
  - **成本较高**：用户端检测、服务端预处理、OCR + LLM、人工兜底，成本较高
  - **维护成本**：需要维护多个环节

```
推荐：混合方案（所有方案结合）
```

## 💡 实际案例：银行支票审查（快速反馈）

### 场景描述

**问题**：银行兑换支票的审查，反馈速度非常快（几秒到几分钟），他们是怎么做的？

**观察**：
- ✅ **反馈速度快**：几秒到几分钟就能给出结果
- ✅ **准确率高**：很少有错误
- ✅ **用户体验好**：不需要等待很久

---

### 银行支票审查的流程

**完整流程**：
```
1. 用户拍照上传支票
   ↓
2. OCR 识别支票信息（金额、账号、签名等）
   ↓
3. 规则引擎验证（金额、账号、签名匹配等）
   ↓
4. 风控系统检查（账户余额、风险评分等）
   ↓
5. 自动审核通过 or 转人工审核
   ↓
6. 返回结果（几秒到几分钟）
```

---

### 为什么银行支票审查这么快？

**关键原因**：

---

#### 1. OCR 优先（快速识别）

**做法**：
- ✅ **OCR 识别**：优先使用 OCR 识别支票信息（金额、账号、签名等）
- ✅ **结构化输出**：OCR 输出结构化的数据（JSON），不需要 LLM 推理
- ✅ **识别速度快**：OCR 识别速度快（几百毫秒到几秒）

**例子**：
```
用户上传支票照片
   ↓
OCR 识别（500ms）：
{
  "amount": "1000.00",
  "account_number": "1234567890",
  "signature": "John Doe",
  "date": "2024-01-15"
}
   ↓
结构化输出（不需要 LLM 推理）
```

**对比 KYC**：
```
KYC 项目：
- 使用 LLM 推理（5-10秒）→ 慢
- LLM 输出可能需要结构化处理

银行支票：
- 使用 OCR 识别（500ms-2秒）→ 快
- OCR 直接输出结构化数据
```

**Trade-off**：
- ✅ **优点**：识别速度快，成本低，输出结构化
- ⚠️ **缺点**：只能识别清晰的文本，对于手写、复杂格式效果不好

---

#### 2. 规则引擎验证（快速决策）

**做法**：
- ✅ **规则引擎**：使用规则引擎验证支票信息（金额、账号、签名匹配等）
- ✅ **自动化决策**：不需要人工判断，规则引擎自动决策
- ✅ **决策速度快**：规则引擎决策速度快（毫秒级）

**例子**：
```
OCR 识别结果：
{
  "amount": "1000.00",
  "account_number": "1234567890",
  "signature": "John Doe"
}

规则引擎验证（10ms）：
1. 检查账户余额是否足够（amount <= balance）
2. 检查账号是否有效（account_number in valid_accounts）
3. 检查签名是否匹配（signature matches user_signature）
4. 检查金额是否合理（amount < daily_limit）

结果：
- 如果所有规则通过 → 自动审核通过
- 如果任何规则失败 → 转人工审核
```

**对比 KYC**：
```
KYC 项目：
- 使用 LLM 推理判断 → 慢（5-10秒）
- 判断逻辑复杂，需要上下文理解

银行支票：
- 使用规则引擎判断 → 快（10-100毫秒）
- 判断逻辑简单，只需要匹配和比较
```

**Trade-off**：
- ✅ **优点**：决策速度快，逻辑清晰，可维护性好
- ⚠️ **缺点**：只能处理规则明确的情况，复杂场景需要人工判断

---

#### 3. 风控系统检查（快速风险评分）

**做法**：
- ✅ **风控系统**：预先计算风险评分（账户余额、历史交易、风险等级等）
- ✅ **快速查询**：从缓存或数据库中快速查询风险评分
- ✅ **决策快速**：基于风险评分快速决策

**例子**：
```
规则引擎验证通过
   ↓
风控系统检查（50ms）：
1. 查询账户余额（从缓存，10ms）
2. 查询历史交易（从缓存，10ms）
3. 查询风险评分（从缓存，10ms）
4. 计算风险等级（20ms）

结果：
- 如果风险等级低 → 自动审核通过
- 如果风险等级高 → 转人工审核
```

**对比 KYC**：
```
KYC 项目：
- 可能需要在处理时计算风险评分 → 慢（需要调用多个服务）

银行支票：
- 风险评分预先计算，存储在缓存中 → 快（直接查询缓存）
```

**Trade-off**：
- ✅ **优点**：查询速度快，决策快速，可以预先计算
- ⚠️ **缺点**：需要维护缓存和数据库，需要实时更新

---

#### 4. 自动审核通过（快速反馈）

**做法**：
- ✅ **自动化决策**：如果 OCR 识别 + 规则引擎验证 + 风控检查都通过，自动审核通过
- ✅ **不需要人工**：大部分情况下不需要人工审核
- ✅ **反馈快速**：自动审核几秒内完成

**例子**：
```
OCR 识别（500ms）
   ↓
规则引擎验证（10ms）
   ↓
风控系统检查（50ms）
   ↓
自动审核通过（总时间：560ms ≈ 0.6秒）

返回结果："审核通过，金额已到账"
```

**对比 KYC**：
```
KYC 项目：
- 可能需要 LLM 推理（5-10秒）→ 慢
- 可能需要人工审核（几分钟到几小时）→ 慢

银行支票：
- OCR + 规则引擎 + 风控（0.5-2秒）→ 快
- 大部分情况自动审核通过，不需要人工
```

**Trade-off**：
- ✅ **优点**：反馈速度快，用户体验好，成本低
- ⚠️ **缺点**：只能处理规则明确的情况，复杂场景需要人工审核

---

#### 5. 人工审核兜底（保证质量）

**做法**：
- ✅ **人工审核**：如果自动审核失败，转人工审核
- ✅ **人工快速审核**：人工审核员可以快速查看 OCR 结果和规则验证结果，快速决策
- ✅ **保证质量**：人工审核保证最终质量

**例子**：
```
自动审核失败（规则不匹配、风险等级高）
   ↓
转人工审核（2-5分钟）
   ↓
人工审核员查看：
1. OCR 识别结果（金额、账号、签名）
2. 规则验证结果（哪些规则失败）
3. 风控检查结果（风险评分）
   ↓
人工审核员快速决策：
- 如果 OCR 识别错误 → 手动修正
- 如果规则过于严格 → 手动通过
- 如果风险可控 → 手动通过
```

**对比 KYC**：
```
KYC 项目：
- 人工审核可能需要查看完整图片和上下文 → 慢（5-10分钟）

银行支票：
- 人工审核只需查看 OCR 结果和规则验证结果 → 快（2-5分钟）
```

**Trade-off**：
- ✅ **优点**：保证质量，可以处理复杂情况
- ⚠️ **缺点**：需要人工成本，处理时间较长
