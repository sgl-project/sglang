## 核心问题

**问题**：如果每天几百万请求，每层 99% 通过率，经过多层保护，能保证系统不 down，只是个别用户体验差一些？如何避免 error 累积？Chrome 的设计思路是否类似？

**答案**：**是的，这是"分层容错"（Defense in Depth）的核心设计思路**。通过**隔离（Isolation）**、**熔断（Circuit Breaker）**、**限流（Rate Limiting）**、**降级（Fallback）** 等多层保护，保证**系统可用性**（系统不 down），同时接受**个别请求失败**（个别用户体验差）。

---

## 设计思路：分层容错（Defense in Depth）

### 1. 多层 99% 通过率的数学原理

**关键点**：**不是"累积失败率"，而是"每层独立保护"**。

#### ❌ 错误理解：累积失败率

```
如果每层 99% 通过率，经过 5 层：
最终通过率 = 0.99^5 = 95.1%
失败率 = 4.9%
```

**问题**：这样理解会导致"层数越多，失败率越高"，不符合设计目标。

#### ✅ 正确理解：每层独立保护，失败不传播

```
Layer 1 (Preprocessing): 99% 通过 → 1% 失败（直接返回错误，不进入下一层）
Layer 2 (Rate Limiter): 99% 通过 → 1% 被限流（返回 429，不进入下一层）
Layer 3 (Circuit Breaker): 99% 通过 → 1% 被熔断（走降级方案，不进入下一层）
Layer 4 (API Call): 99% 成功 → 1% 失败（自动重试，或走降级）
Layer 5 (Validation): 99% 通过 → 1% 失败（转人工审核，不直接失败）
```

**关键设计**：
- **每层独立**：Layer 1 失败 → 直接返回，**不进入** Layer 2
- **失败不传播**：Layer 2 失败 → 走降级方案，**不影响** Layer 3
- **系统不 down**：即使 Layer 4 失败，Layer 5 的降级方案保证系统可用

#### Layer 1 失败后的具体流程

**场景**：Layer 1 (Preprocessing) 检测到图片格式不支持（如 `.webp` 格式）

**流程**：

```
1. 检测失败
   ↓
2. 抛出异常：ImageFormatError("IMAGE_FORMAT_UNSUPPORTED")
   ↓
3. 错误处理（不进入 Layer 2）
   ├─ 记录日志（包含 file_id、error_code、timestamp）
   ├─ 写入 _summary.json：
   │   {
   │     "file_id": "doc_001.jpg",
   │     "status": "fail",
   │     "error_code": "IMAGE_FORMAT_UNSUPPORTED",
   │     "latency_ms": 50  // 只用了 50ms，因为提前失败
   │   }
   ├─ 返回响应给用户：
   │   {
   │     "status": "fail",
   │     "error_code": "IMAGE_FORMAT_UNSUPPORTED",
   │     "message": "Unsupported image format. Supported formats: jpg, png, pdf"
   │   }
   └─ 计入指标：
       - Error Rate：+1（计入失败）
       - Success Rate：-1（不计入成功）
       - error_breakdown["IMAGE_FORMAT_UNSUPPORTED"]：+1
   ↓
4. 不触发重试（不可恢复错误）
   ↓
5. 不触发降级（格式错误无法通过降级解决）
   ↓
6. 不触发告警（除非 IMAGE_FORMAT_UNSUPPORTED 占比 > 2%）
```

**关键点**：

| 维度 | 说明 |
|------|------|
| **响应时间** | **极快**（~50ms），因为提前失败，不进入后续处理 |
| **系统影响** | **无影响**（系统正常，其他请求不受影响） |
| **用户影响** | **个别用户**（该请求失败，但其他用户正常） |
| **是否重试** | **不重试**（不可恢复错误，重试没用） |
| **是否降级** | **不降级**（格式错误无法通过降级解决） |
| **是否告警** | **条件告警**（如果 `IMAGE_FORMAT_UNSUPPORTED` 占比 > 2%，触发 Warning） |
| **是否计入 Error Rate** | **是**（计入失败，影响 Success Rate） |

**对比其他层**：

| 层 | 失败后行为 | 是否重试 | 是否降级 | 系统影响 |
|----|----------|---------|---------|---------|
| **Layer 1: Preprocessing** | 直接返回错误，不进入下一层 | ❌ 不重试 | ❌ 不降级 | 无影响 |
| **Layer 2: Rate Limiter** | 返回 429，不进入下一层 | ❌ 不重试 | ❌ 不降级 | 无影响 |
| **Layer 3: Circuit Breaker** | 走降级方案，不进入下一层 | ❌ 不重试 | ✅ 降级 | 无影响 |
| **Layer 4: API Call** | 自动重试，或走降级 | ✅ 重试 | ✅ 降级 | 无影响 |
| **Layer 5: Validation** | 转人工审核，不直接失败 | ❌ 不重试 | ✅ 转人工 | 无影响 |

#### 数学计算：系统可用性 vs 请求成功率

**假设**：每天 100 万请求，每层 99% 通过率

| 层 | 通过率 | 失败数/天 | 失败原因 | 系统影响 |
|----|--------|----------|---------|---------|
| **Layer 1: Preprocessing** | 99% | 10,000 | 格式错误 | **系统正常**，返回 `IMAGE_FORMAT_UNSUPPORTED` |
| **Layer 2: Rate Limiter** | 99% | 10,000 | 限流 | **系统正常**，返回 `429 Too Many Requests` |
| **Layer 3: Circuit Breaker** | 99% | 10,000 | 熔断 | **系统正常**，走降级方案（OCR-only） |
| **Layer 4: API Call** | 99% | 10,000 | API 失败 | **系统正常**，自动重试或走降级 |
| **Layer 5: Validation** | 99% | 10,000 | 校验失败 | **系统正常**，转人工审核 |

**结果**：
- **系统可用性**：**100%**（系统不 down，所有请求都有响应）
- **请求成功率**：**95%**（5% 的请求失败，但系统正常）
- **个别用户体验**：5% 的用户体验差（失败/降级/转人工），但**不影响其他 95% 的用户**

### 2. 如何避免 error 累积？

**核心机制**：**隔离（Isolation）** + **熔断（Circuit Breaker）** + **限流（Rate Limiting）**

#### 机制 1：隔离（Isolation）

**设计**：**每个请求独立处理，失败不传播**。

```
Request 1 失败 → 不影响 Request 2
Batch 1 失败 → 不影响 Batch 2（Per-File Isolation）
Service A 失败 → 不影响 Service B（Service Isolation）
```

**KYC 示例**：
- **Per-File Isolation**（`src/pipeline.py`）：某个文件失败 **不会** 让整个 batch 崩掉
- **Service Isolation**：OCR 服务失败 → 走降级方案（LLM-only），不影响其他服务

#### 机制 2：熔断（Circuit Breaker）

**设计**：**连续失败超过阈值 → 自动熔断，走降级方案**。

```
正常状态（Closed）：
  - 请求正常通过
  - 失败率 < 5% → 继续正常

熔断状态（Open）：
  - 失败率 > 5% → 自动熔断
  - 所有请求走降级方案（不调用主服务）
  - 系统不 down，但用户体验降级

半开状态（Half-Open）：
  - 60 秒后，尝试少量请求
  - 如果成功 → 关闭熔断，恢复正常
  - 如果失败 → 继续熔断
```

**KYC 示例**：
- **API 失败率 > 5%** → 自动熔断，走 OCR-only 降级方案
- **系统可用性**：100%（降级方案保证系统可用）
- **用户体验**：降级（OCR-only 准确率低），但**不影响其他用户**

#### 机制 3：限流（Rate Limiting）

**设计**：**超过 QPS 限制 → 直接拒绝，不进入处理流程**。

```
QPS 限制 = 1000 req/s
当前 QPS = 1200 req/s

→ 前 1000 个请求正常处理
→ 后 200 个请求直接返回 429（Too Many Requests）
→ 系统不 down，但部分请求被拒绝
```

**KYC 示例**：
- **QPS 限制 = 100 req/s**（保护上游 API）
- **超过限制** → 直接返回 `429`，不进入 pipeline
- **系统可用性**：100%（限流保护系统不 down）
- **用户体验**：部分请求被拒绝，但**不影响已接受的请求**

---

### 3. 系统不 down 的保证机制

**设计目标**：**系统可用性 > 99.9%**（系统不 down），同时接受**请求成功率 < 99%**（个别请求失败）。

#### 保证机制 1：降级方案（Fallback）

```
主服务失败 → 自动切换到降级方案
  - OCR-only（不调用 LLM）
  - 人工审核队列（不直接失败）
  - 缓存结果（不重新计算）
```

**结果**：
- **系统可用性**：100%（降级方案保证系统可用）
- **请求成功率**：可能 < 99%（降级方案准确率低），但**系统不 down**

#### 保证机制 2：超时保护（Timeout）

```
API 调用超时（> 30s）→ 自动取消，走降级方案
  - 不等待（避免阻塞）
  - 不重试（避免累积）
  - 直接降级（保证响应）
```

**结果**：
- **系统可用性**：100%（超时保护避免阻塞）
- **请求成功率**：可能 < 99%（超时请求失败），但**系统不 down**

#### 保证机制 3：资源隔离（Resource Isolation）

```
每个请求独立资源：
  - 独立进程/线程（不共享状态）
  - 独立内存（不互相影响）
  - 独立超时（不互相阻塞）
```

**结果**：
- **系统可用性**：100%（资源隔离避免级联失败）
- **请求成功率**：可能 < 99%（个别请求失败），但**系统不 down**

## 类比 Chrome 的设计思路

### Chrome 的分层容错设计

**Chrome 的核心设计**：**Process Isolation（进程隔离）** + **Sandbox（沙箱）** + **Crash Recovery（崩溃恢复）**

#### 1. Process Isolation（进程隔离）

```
每个 Tab 独立进程：
  - Tab 1 崩溃 → 不影响 Tab 2
  - Tab 1 内存泄漏 → 不影响 Tab 2
  - Tab 1 卡死 → 不影响 Tab 2
```

**类比 KYC**：
- **Per-File Isolation**：每个文件独立处理，失败不传播
- **Service Isolation**：每个服务独立，失败不传播

#### 2. Sandbox（沙箱）

```
Renderer 进程运行在沙箱中：
  - 无法访问文件系统
  - 无法访问网络（通过 IPC）
  - 无法访问其他进程
```

**类比 KYC**：
- **Rate Limiter**：限制每个请求的资源使用
- **Timeout**：限制每个请求的执行时间
- **Circuit Breaker**：限制失败请求的影响范围

#### 3. Crash Recovery（崩溃恢复）

```
Tab 崩溃 → 自动重启：
  - 不丢失用户数据（Session Storage）
  - 不影响其他 Tab
  - 用户感知：页面刷新，但系统不 down
```

**类比 KYC**：
- **自动重试**：请求失败 → 自动重试
- **降级方案**：主服务失败 → 走降级方案
- **转人工审核**：校验失败 → 转人工，不直接失败

#### 4. 为什么 Chrome 不会反复出现同一个 bug？

**设计机制**：
1. **Bug 修复**：发现 bug → 修复 → 发布新版本
2. **自动更新**：用户自动更新到新版本
3. **Crash Reporting**：崩溃报告 → 定位 bug → 修复

**类比 KYC**：
1. **Error Classification**：错误分类 → 定位根因 → 修复
2. **Monitoring**：监控告警 → 定位问题 → 修复
3. **Postmortem**：事故复盘 → 改进设计 → 避免重复

## 最终目的：系统可用性 vs 个别用户体验

### 设计权衡

| 维度 | 系统可用性（System Availability） | 请求成功率（Request Success Rate） |
|------|----------------------------------|-----------------------------------|
| **目标** | **> 99.9%**（系统不 down） | **> 99%**（个别请求可以失败） |
| **保证机制** | 降级方案、熔断、限流 | 重试、校验、转人工 |
| **失败影响** | **系统级**（所有用户受影响） | **请求级**（个别用户受影响） |
| **优先级** | **更高**（系统不 down 是底线） | **次之**（个别请求失败可接受） |

### KYC 示例

**场景**：每天 100 万请求，API 失败率 5%

**设计选择 1：保证系统可用性（推荐）**
```
API 失败率 > 5% → 自动熔断，走 OCR-only 降级方案
  - 系统可用性：100%（降级方案保证系统可用）
  - 请求成功率：可能 < 99%（OCR-only 准确率低）
  - 用户体验：降级（准确率低），但系统不 down
```

**设计选择 2：保证请求成功率（不推荐）**
```
API 失败率 > 5% → 继续调用主服务，不熔断
  - 系统可用性：可能 < 99%（主服务可能 down）
  - 请求成功率：可能 > 99%（不降级）
  - 用户体验：可能系统 down，所有用户受影响
```

**结论**：**优先保证系统可用性**（系统不 down），接受**个别请求失败**（个别用户体验差）。

## 总结

### 核心设计思路

1. **分层容错**：每层独立保护，失败不传播
2. **隔离机制**：每个请求独立处理，失败不传播
3. **熔断/限流**：超过阈值 → 自动保护，走降级方案
4. **降级方案**：主服务失败 → 走降级方案，保证系统可用

### 数学原理

- **不是"累积失败率"**：层数越多，失败率越高（错误理解）
- **而是"每层独立保护"**：每层失败 → 直接返回/降级，不进入下一层（正确理解）

### 系统保证

- **系统可用性**：> 99.9%（系统不 down）
- **请求成功率**：> 99%（个别请求可以失败）
- **个别用户体验**：可能差（失败/降级/转人工），但**不影响其他用户**

### 类比 Chrome

- **Process Isolation** → **Per-File Isolation**
- **Sandbox** → **Rate Limiter / Timeout / Circuit Breaker**
- **Crash Recovery** → **自动重试 / 降级方案 / 转人工审核**
