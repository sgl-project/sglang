# Day 7｜面试固化：把系统讲成"低风险进化"的评审节奏

## 为什么要练

你不缺技术深度，缺的是"每次都能稳定输出 senior 评审口吻"。

## 目的

把前 6 天的交付物串成 30 秒 / 2 分钟 / 5 分钟三套话术。

## 目标

你能在 5 分钟内覆盖：目标→约束→设计→指标→失败模式→发布回滚→长期演进。

### 1. Goal + 三层指标（1分钟）

#### 脚本模板

```
【系统目标】
我们做了一个 [系统名称] 系统，核心目标是 [业务目标]。
系统解决的核心问题是 [核心问题]。

【指标体系】
我们用三层指标来度量成功：

L0稳定性（监控/可用/延迟）：
- 成功率：99.9%（SLO目标：99.9%）
- p95延迟：285ms（SLO目标：<300ms）
- 错误率：0.1%（SLO目标：<0.1%）
- 回退率：2.5%

Error Budget Policy：预算烧太快就冻结发布转稳定性工作。

L1业务收益（ROI）：
- 每单节省的人审分钟数：5分钟/单
- 错误拦截率带来的风险降低：每月避免50件错误提交
- 成本节省：$0.001/request，每月节省$5000

L2长期健康（可维护/可扩展）：
- 变更失败率：<5%
- 回滚频率：<2%（近30天：1次回滚）
- 回归门禁通过率：>95%
- 告警噪音：<10%（precision）

这些指标帮助我们平衡"发布速度 vs 稳定性"，用Error Budget来决定什么时候继续发布、什么时候停下来还债。
```

### 2. 关键 trade-off（1分钟）

#### 脚本模板

```
【关键 Trade-off】

我们面临的核心 trade-off 是 [核心 trade-off]。

例如：
1. 【准确性 vs 成本】
   - 高准确率需要大模型（成本高）
   - 我们采用多模型策略：主模型高准确率，降级模型低成本
   - 用 Schema 验证确保质量，用降级确保可用性

2. 【延迟 vs 吞吐】
   - 低延迟需要更多资源（成本高）
   - 我们采用 Batch 处理：小 Batch 低延迟，大 Batch 高吞吐
   - 用 Canary 发布逐步验证，用限流防止过载

3. 【功能 vs 稳定性】
   - 新功能带来风险，过度保守影响创新
   - 我们用 Error Budget 平衡：预算充足时发布，不足时还债
   - 用 Feature Flag + Canary 降低风险，用回归门禁确保质量

通过这些 trade-off，我们在 [约束] 下实现了 [目标]。
```

#### 练习示例

```
【关键 Trade-off】

我们面临的核心 trade-off 是"准确性与成本/延迟之间的平衡"。

具体来说：
1. 【准确性 vs 成本】
   - 高准确率需要大模型（Qwen2.5-72B，成本高）
   - 我们采用多模型策略：主模型高准确率，降级模型（Qwen2.5-7B）低成本
   - 用 Schema 验证确保质量，用降级确保可用性

2. 【延迟 vs 吞吐】
   - 低延迟需要更多资源（成本高）
   - 我们采用 Batch 处理：小 Batch 低延迟（50ms），大 Batch 高吞吐（100 req/s）
   - 用 Canary 发布逐步验证，用限流防止过载

3. 【功能 vs 稳定性】
   - 新功能带来风险，过度保守影响创新
   - 我们用 Error Budget 平衡：预算充足时发布，不足时还债
   - 用 Feature Flag + Canary 降低风险，用回归门禁确保质量

通过这些 trade-off，我们在成本约束下实现了99.9%的可用性和30%的成本降低。
```

### 3. Failure modes + 兜底（1.5分钟）

#### 脚本模板

```
【失败模式与兜底】

我们设计了多层次的保护策略，确保失败可控、可恢复：

1. 【限流（Rate Limiting）】
   - 触发：RPS > 1000 或 并发 > 500
   - 动作：返回 429，限制请求速率
   - 验证：压测看 p95 < 300ms，429 rate < 5%

2. 【熔断（Circuit Breaker）】
   - 触发：上游服务失败率 > 5% 或 延迟 > 1000ms
   - 动作：快速失败，不调用上游，返回默认响应
   - 验证：上游故障时，下游延迟 < 100ms

3. 【重试（Retry）】
   - 触发：可恢复错误（网络错误、5xx）
   - 动作：指数退避重试（初始100ms，最大10s），最多3次
   - 验证：成功率从95%提升到99%

4. 【幂等（Idempotency）】
   - 触发：重复 request_id
   - 动作：返回缓存结果，不重复处理
   - 验证：重复请求率 < 0.1%

5. 【降级（Degradation）】
   - 触发：上游服务不可用 或 延迟 > 500ms
   - 动作：小模型/规则/缓存降级
   - 验证：降级后延迟 < 200ms，成功率 > 90%

通过这些保护策略，我们实现了"失败可控、可恢复"，系统不会因为单点故障而崩溃。
```

### 4. Flag + canary + rollback + 回归门禁（1.5分钟）

#### 脚本模板

```
【低风险进化：发布与回滚】

这是"低风险进化"的核心，确保每次变更都能快速验证、快速回滚：

1. 【Feature Flag + Canary 发布】
   - Feature Flag：按 model_version / prompt_version / validator_strictness 放量
   - Canary：1% → 5% → 25% → 100%，每步观察 L0/L1 指标
   - 每步持续时间：10-30分钟，观察 p95、error rate、schema fail、fallback、成本

2. 【回滚条件（Rollback Gates）】
   - 明确阈值：schema-fail × 2 或 p95 + 20% 立即回滚
   - 自动回滚：任何回滚条件满足，立即切回旧版本
   - 验证：回滚后 < 5 分钟恢复正常

3. 【回归门禁（Regression Gate）】
   - Golden Set：50-200 条测试用例（正常/边界/异常/长尾）
   - 门禁指标：schema pass rate > 95%，字段准确率 > 90%，fallback < 5%，成本 < 阈值
   - 通过阈值：所有指标通过才能发布，否则禁止发布

这套流程确保：
- 部署与发布解耦（Feature Flag）
- 小流量验证（Canary）
- 快速回滚（Rollback）
- 质量保证（回归门禁）

这就是"低风险进化"：能在真实流量下试错，但把伤害限制在很小范围，并能快速回滚。
```

### 5. 长期演进（1分钟）

#### 脚本模板

```
【长期演进】

系统的长期演进方向：

1. 【可观测性改进】
   - 完善 Metrics/Logs/Traces
   - 优化 Dashboard，快速定位根因
   - 降低告警噪音（precision > 90%）

2. 【自动化改进】
   - 自动化回归测试
   - 自动化 Canary 发布
   - 自动化回滚

3. 【保护策略改进】
   - 完善限流/熔断/降级策略
   - 优化重试策略
   - 增强幂等性

4. 【成本优化】
   - 模型量化（FP8/INT4）
   - 缓存优化
   - Batch 优化

5. 【可扩展性】
   - 支持多地域部署
   - 支持更多模型/格式
   - 支持更多业务场景

通过这些改进，我们不断降低 Toil（重复劳动），提升系统可靠性和可维护性。
```

## 核心要点总结

**这 7 天不是在"学大厂流程"，而是在把你的深技术变成"可托付的工业能力"**

**SRE 的核心逻辑**：
- SLO 不要求 100% 满足，要允许 error budget
- Canary/Feature Flags 的意义：在真实流量下试错，但把伤害限制在很小范围
- 失败不可怕，关键是要可控、可恢复、可学习

**面试时，展示的是**：
- 可度量（三层指标）
- 可回归（Golden Set + 门禁）
- 可灰度（Feature Flag + Canary）
- 可回滚（明确的回滚条件和流程）
- 可复盘（Runbook + Postmortem）

这就是"低风险进化"的核心。
