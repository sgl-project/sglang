## Definition（定义）

**核心问题**：**重试次数、重试间隔、熔断阈值等参数应该如何设定？有业界标准吗？**

**核心答案**：**大部分参数设定有业界标准和最佳实践，但需要根据具体场景（延迟要求、成本、稳定性）调整**。本文档提供业界标准和推荐配置。

**关键理解**：
- ✅ **业界标准**：大部分参数有推荐范围（基于经验和实践）
- ✅ **场景驱动**：需要根据具体场景调整（延迟 vs 成本 vs 稳定性）
- ✅ **渐进调优**：先用默认值，再根据监控数据调整

## 🎯 核心问题

### 参数设定原则

**核心原则**：
```
1. 先用业界标准默认值（Common Sense）
2. 根据监控数据调整（Metrics-Driven）
3. 平衡延迟、成本、稳定性（Trade-off）
```

**业界标准来源**：
- ✅ **Google Cloud、AWS、Azure** 的推荐配置
- ✅ **Netflix、Uber、Airbnb** 等大公司的实践
- ✅ **开源项目（如 SGLang、Istio、Envoy）** 的默认值
- ✅ **经验法则（Rules of Thumb）**

## 📊 详细参数设定

### 参数 1：重试策略（Retry Strategy）

#### 1.1 重试次数（Max Retries）

**业界标准**：

| 场景 | 推荐值 | 范围 | 说明 |
|------|-------|------|------|
| **一般 API 调用** | **3-5 次** | 1-10 次 | 最常见配置 |
| **关键业务（支付、订单）** | **5-7 次** | 3-10 次 | 更重视成功率 |
| **非关键业务（日志、统计）** | **1-3 次** | 1-5 次 | 可以接受失败 |
| **外部依赖（第三方 API）** | **3-5 次** | 1-7 次 | 平衡成本和成功率 |

**SGLang Router 默认值**：`5 次`

**推荐配置**：
```python
# 一般场景（推荐）
max_retries = 5

# 关键业务（支付、订单）
max_retries = 7

# 非关键业务（日志、统计）
max_retries = 3

# 外部依赖（第三方 API）
max_retries = 5
```

**选择依据**：
- ✅ **重试次数越多** → 成功率越高，但延迟增加、成本增加
- ✅ **重试次数越少** → 延迟降低、成本降低，但成功率降低

**业界经验法则**：
- ✅ **3 次重试**：对于大多数场景足够（成功率 > 99%）
- ✅ **5 次重试**：平衡延迟和成功率（业界最常用）
- ✅ **10 次重试**：过度重试，可能导致"惊群"效应

---

#### 1.2 初始退避时间（Initial Backoff）

**业界标准**：

| 场景 | 推荐值 | 范围 | 说明 |
|------|-------|------|------|
| **一般 API 调用** | **50-100ms** | 10-500ms | 常见配置 |
| **快速失败（实时系统）** | **10-50ms** | 10-100ms | 延迟敏感 |
| **慢速重试（批处理）** | **100-500ms** | 100-1000ms | 成本敏感 |
| **网络抖动** | **50-200ms** | 50-500ms | 等待网络恢复 |

**SGLang Router 默认值**：`50ms`

**推荐配置**：
```python
# 一般场景（推荐）
initial_backoff_ms = 50

# 实时系统（延迟敏感）
initial_backoff_ms = 20

# 批处理（成本敏感）
initial_backoff_ms = 200

# 网络抖动（等待恢复）
initial_backoff_ms = 100
```

**选择依据**：
- ✅ **初始退避时间短** → 快速重试，延迟低，但可能加重服务端压力
- ✅ **初始退避时间长** → 减轻服务端压力，但延迟增加

**业界经验法则**：
- ✅ **50ms**：业界最常用的初始退避时间（平衡延迟和压力）
- ✅ **< 10ms**：太快，可能加重服务端压力
- ✅ **> 500ms**：太慢，用户体验差

---

#### 1.3 最大退避时间（Max Backoff）

**业界标准**：

| 场景 | 推荐值 | 范围 | 说明 |
|------|-------|------|------|
| **一般 API 调用** | **30-60s** | 10-120s | 常见配置 |
| **实时系统** | **5-10s** | 1-30s | 快速失败 |
| **批处理任务** | **60-300s** | 60-600s | 可以等待 |
| **外部依赖** | **30-120s** | 10-300s | 等待服务恢复 |

**SGLang Router 默认值**：`30s` (30_000ms)

**推荐配置**：
```python
# 一般场景（推荐）
max_backoff_ms = 30_000  # 30秒

# 实时系统（快速失败）
max_backoff_ms = 5_000  # 5秒

# 批处理任务（可以等待）
max_backoff_ms = 60_000  # 60秒

# 外部依赖（等待恢复）
max_backoff_ms = 120_000  # 120秒
```

**选择依据**：
- ✅ **最大退避时间短** → 快速失败，用户体验好，但可能错过恢复
- ✅ **最大退避时间长** → 等待更久，成功率更高，但延迟增加

**业界经验法则**：
- ✅ **30 秒**：业界最常用的最大退避时间（平衡等待和失败）
- ✅ **< 5 秒**：太快，可能错过服务恢复
- ✅ **> 120 秒**：太慢，用户可能已经放弃

---

#### 1.4 退避倍数（Backoff Multiplier）

**业界标准**：

| 场景 | 推荐值 | 范围 | 说明 |
|------|-------|------|------|
| **一般 API 调用** | **1.5-2.0** | 1.2-3.0 | 常见配置 |
| **指数退避（标准）** | **2.0** | 1.5-2.5 | 经典指数退避 |
| **温和退避** | **1.2-1.5** | 1.2-1.8 | 延迟增长慢 |
| **快速退避** | **2.5-3.0** | 2.0-3.5 | 延迟增长快 |

**SGLang Router 默认值**：`1.5`

**推荐配置**：
```python
# 一般场景（推荐）
backoff_multiplier = 1.5

# 标准指数退避
backoff_multiplier = 2.0

# 温和退避（延迟敏感）
backoff_multiplier = 1.3

# 快速退避（快速失败）
backoff_multiplier = 2.5
```

**退避时间计算**：
```python
# 退避时间 = initial_backoff_ms * (backoff_multiplier ^ attempt)
# 例如：initial_backoff_ms=50ms, backoff_multiplier=1.5

attempt 0: 50ms * (1.5 ^ 0) = 50ms
attempt 1: 50ms * (1.5 ^ 1) = 75ms
attempt 2: 50ms * (1.5 ^ 2) = 112ms
attempt 3: 50ms * (1.5 ^ 3) = 168ms
attempt 4: 50ms * (1.5 ^ 4) = 252ms
attempt 5: 50ms * (1.5 ^ 5) = 378ms
# 但不超过 max_backoff_ms = 30_000ms
```

**选择依据**：
- ✅ **退避倍数小（1.2-1.5）** → 延迟增长慢，适合延迟敏感场景
- ✅ **退避倍数大（2.0-3.0）** → 延迟增长快，快速失败，减轻服务端压力

**业界经验法则**：
- ✅ **1.5-2.0**：业界最常用的退避倍数（平衡延迟和压力）
- ✅ **2.0**：经典指数退避（2 的幂次）
- ✅ **> 3.0**：太快，可能过早放弃

---

#### 1.5 抖动因子（Jitter Factor）

**业界标准**：

| 场景 | 推荐值 | 范围 | 说明 |
|------|-------|------|------|
| **一般 API 调用** | **0.1-0.3** | 0.0-0.5 | 常见配置 |
| **避免"惊群"** | **0.2-0.3** | 0.1-0.4 | 随机化重试时间 |
| **精确控制** | **0.0-0.1** | 0.0-0.2 | 几乎不抖动 |
| **高并发场景** | **0.3-0.5** | 0.2-0.6 | 更大的随机性 |

**SGLang Router 默认值**：`0.2` (20%)

**推荐配置**：
```python
# 一般场景（推荐）
jitter_factor = 0.2  # 20% 抖动

# 避免"惊群"（高并发）
jitter_factor = 0.3  # 30% 抖动

# 精确控制（延迟敏感）
jitter_factor = 0.1  # 10% 抖动

# 高并发场景（大规模重试）
jitter_factor = 0.4  # 40% 抖动
```

**抖动计算**：
```python
# 抖动后的退避时间 = base_delay * (1 ± jitter_factor)
# 例如：base_delay=100ms, jitter_factor=0.2

最小退避时间: 100ms * (1 - 0.2) = 80ms
最大退避时间: 100ms * (1 + 0.2) = 120ms
实际退避时间: 80ms ~ 120ms（随机）
```

**选择依据**：
- ✅ **抖动因子大（0.3-0.5）** → 随机性大，避免"惊群"效应，适合高并发
- ✅ **抖动因子小（0.0-0.1）** → 精确控制，适合延迟敏感场景

**业界经验法则**：
- ✅ **0.2（20%）**：业界最常用的抖动因子（平衡随机性和精确性）
- ✅ **0.0（无抖动）**：所有请求同时重试，可能导致"惊群"效应
- ✅ **> 0.5（50%）**：随机性太大，延迟不可控

### 参数 2：熔断器（Circuit Breaker）

**业界标准**：

| 场景 | 推荐值 | 范围 | 说明 |
|------|-------|------|------|
| **一般服务** | **5-10 次** | 3-20 次 | 常见配置 |
| **关键服务** | **10-20 次** | 5-30 次 | 更保守 |
| **非关键服务** | **3-5 次** | 1-10 次 | 快速熔断 |
| **外部依赖** | **5-10 次** | 3-15 次 | 平衡成本和稳定性 |

**SGLang Router 默认值**：`10 次`

**推荐配置**：
```python
# 一般场景（推荐）

failure_threshold = 15

# 非关键服务（快速熔断）
failure_threshold = 5

# 外部依赖（第三方 API）

```

**选择依据**：
- ✅ **失败阈值高（10-20）** → 更保守，避免误熔断，但可能影响用户体验
- ✅ **失败阈值低（3-5）** → 快速熔断，保护系统，但可能误熔断

**业界经验法则**：
- ✅ **10 次**：业界最常用的失败阈值（平衡保守性和快速响应）
- ✅ **< 5 次**：太快，可能误熔断（临时性故障）

### 参数 3：限流（Rate Limiting）

#### 3.1 最大并发请求数（Max Concurrent Requests）

**业界标准**：

| 场景 | 推荐值 | 范围 | 说明 |
|------|-------|------|------|
| **一般服务** | **100-500** | 50-1000 | 常见配置 |
| **高并发服务** | **500-2000** | 500-10000 | 需要高并发 |
| **资源受限** | **50-100** | 10-200 | 资源有限 |
| **API Gateway** | **256-1024** | 128-2048 | 网关服务 |

**SGLang Router 默认值**：`256`

**推荐配置**：
```python
# 一般场景（推荐）
max_concurrent_requests = 256

# 高并发服务
max_concurrent_requests = 1000

# 资源受限
max_concurrent_requests = 100

# API Gateway
max_concurrent_requests = 512
```

**选择依据**：
- ✅ **最大并发数高（500-2000）** → 处理更多请求，但资源消耗增加
- ✅ **最大并发数低（50-100）** → 资源消耗少，但吞吐量降低

**业界经验法则**：
- ✅ **256-512**：业界最常用的最大并发数（平衡资源和吞吐量）
- ✅ **根据资源（CPU/内存）** 调整：每个请求的资源消耗 × 可用资源

- ✅ **队列超时 = 请求超时 * 0.5-1.0**（经验公式）

## 💡 总结

### 核心答案

**参数设定是否有业界标准？**

**答案**：**大部分参数有业界标准和最佳实践，但需要根据具体场景调整**。

### 业界标准总结

| 参数 | 业界标准默认值 | 推荐范围 | 选择依据 |
|------|--------------|---------|---------|
| **重试次数** | **5 次** | 3-7 次 | 平衡延迟和成功率 |
| **初始退避** | **50ms** | 20-200ms | 平衡延迟和压力 |
| **最大退避** | **30s** | 10-120s | 平衡等待和失败 |
| **退避倍数** | **1.5-2.0** | 1.3-2.5 | 平衡延迟和压力 |
| **抖动因子** | **0.2（20%）** | 0.1-0.3 | 避免"惊群"效应 |
| **失败阈值** | **10 次** | 5-15 次 | 平衡保守性和快速响应 |
| **成功阈值** | **3 次** | 2-5 次 | 平衡保守性和快速恢复 |
| **熔断超时** | **60s** | 30-120s | 平衡等待和快速恢复 |
| **时间窗口** | **120s** | 60-300s | 平衡响应和准确性 |
| **最大并发** | **256** | 100-1000 | 根据资源调整 |
| **队列大小** | **100** | 50-500 | 平衡内存和吞吐量 |
| **队列超时** | **60s** | 30-120s | 平衡等待和快速失败 |

### 调优流程

**推荐调优流程**：
```
1. 使用业界标准默认值（Common Sense）
   ↓
2. 根据监控数据调整（Metrics-Driven）
   - 错误率、延迟、成功率
   ↓
3. 平衡延迟、成本、稳定性（Trade-off）
   - 延迟敏感 → 快速失败
   - 成本敏感 → 慢速重试
   - 稳定性敏感 → 保守配置
```

### 关键原则

**核心原则**：
- ✅ **先用默认值**：业界标准默认值对大多数场景足够
- ✅ **数据驱动**：根据监控数据调整，不要盲目猜测
- ✅ **平衡取舍**：延迟 vs 成本 vs 稳定性，根据场景选择
- ✅ **渐进调优**：小步快跑，逐步优化，避免过度调优
