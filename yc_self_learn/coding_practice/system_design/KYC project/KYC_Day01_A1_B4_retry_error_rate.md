## Definition（定义）

**概率累积问题** = 如果一个操作有 p 的失败概率，重复执行 n 次，**至少失败一次**的概率 = 1 - (1-p)^n，会随 n 增大而快速上升。

**核心问题**：如果单次失败概率 = 0.01%（万分之一），重试 100 次，错误率会从 0.01% 飙升到接近 1%，甚至看起来「一定发生」。如何设计系统规避这个问题？

---

## 核心问题：概率累积的数学

### 单次 vs 多次的概率

| 场景 | 单次失败概率 | 重复 n 次，至少失败一次的概率 |
|------|-------------|------------------------------|
| **n = 1** | 0.01% | 0.01% |
| **n = 10** | 0.01% | ≈ 0.1% |
| **n = 100** | 0.01% | ≈ 1% |
| **n = 1000** | 0.01% | ≈ 9.5% |

**公式**：P(至少失败一次) = 1 - (1 - 0.0001)^n

**结论**：即使单次概率很小，重复足够多次后，**至少失败一次**的概率会接近 1。

---

## 设计原则：如何规避概率累积问题

### 1. 限制重试次数（MAX_RETRIES）

**不要无限重试，设置上限**（如 MAX_RETRIES = 3）。

| 策略 | 效果 |
|------|------|
| **无限重试** | ❌ 概率累积 → 最终几乎一定失败；资源消耗大 |
| **MAX_RETRIES = 3** | ✅ 限制在 3 次，即使累积概率也控制在可接受范围 |

**KYC 设计**（A4、学习指南）：
- `max_retries=3`（最多重试 3 次）
- 只对**可恢复错误**重试（`API_TIMEOUT`、`API_CONNECTION_ERROR`、`API_SERVER_ERROR`）
- 不可恢复错误（`IMAGE_FORMAT_UNSUPPORTED`、`SCHEMA_VALIDATION_FAILED`）不重试

### 2. Error Rate 按「最终结果」计算，不按「总尝试数」

**关键设计**：Error Rate = **最终失败的请求数** / **原始请求数**，而不是「总失败次数 / 总尝试次数」。

| 计算方式 | 公式 | 问题 |
|----------|------|------|
| ❌ **错误**：按总尝试数 | Error Rate = 总失败次数 / 总尝试次数 | 重试会放大分母和分子，错误率被稀释但「失败次数」看起来很多 |
| ✅ **正确**：按最终结果 | Error Rate = 最终失败的请求数 / 原始请求数 | 重试成功的不计入失败；只有**最终仍失败**的才计入 |

**示例**：

```
原始请求：100 个
- 90 个成功（无需重试）
- 5 个失败但重试后成功（不计入失败）
- 5 个失败且重试 3 次后仍失败（计入失败）

❌ 错误计算：Error Rate = (5 + 5×3) / (100 + 5×3) = 20/115 ≈ 17.4%
✅ 正确计算：Error Rate = 5 / 100 = 5%
```

**KYC 实现**（A2 `_summary.json`）：
- 每个 `result` 只记录**最终状态**：`status: "success"` 或 `"fail"`
- `calculate_error_rate()` = `count(status == "fail") / count(total)`（按最终结果）
- 重试是**内部机制**，不影响 Error Rate 的计算

### 3. 指数退避（Exponential Backoff）

**目的**：避免快速重试导致资源耗尽、上游压力过大。

| 策略 | 效果 |
|------|------|
| **固定间隔重试**（如每 1s） | ❌ 如果上游有问题，100 个请求 × 3 次重试 = 300 次快速请求，可能压垮上游 |
| **指数退避**（1s → 2s → 4s） | ✅ 重试间隔逐渐拉长，给上游恢复时间；降低资源消耗 |

**KYC 设计**（A4、A1）：
- `backoff_multiplier=2`（每次重试间隔翻倍）
- `initial_delay=1.0`（首次重试延迟 1 秒）
- 重试间隔：1s → 2s → 4s（第 3 次重试后放弃）

### 4. 幂等性保证（Idempotency）

**目的**：确保重复执行不会产生副作用，允许安全重试。

| 场景 | 是否幂等？ | 能否安全重试？ |
|------|-----------|---------------|
| **读取操作**（GET） | ✅ 幂等 | ✅ 可安全重试 |
| **写入操作**（POST，无幂等保证） | ❌ 不幂等 | ❌ 重试可能重复写入 |
| **KYC 处理**（读图 → 推理 → 写结果） | ⚠️ 需设计幂等 | ✅ 通过 `trace_id` 去重或结果覆盖 |

**KYC 设计**（学习指南）：
- 通过 `trace_id` 去重：重复 `trace_id` 返回缓存结果
- 结果覆盖：同一 `file_id` 的处理结果可覆盖（最终状态为准）
- 验证：重复处理率 < 0.1%

## 错误预算消耗：重试不应快速消耗预算

**设计原则**：错误预算按**用户可见的失败**计算，不是「内部重试的失败」。

| 场景 | 错误预算消耗 | 说明 |
|------|-------------|------|
| **100 个请求，5 个失败但重试后成功** | 0 个（不消耗） | 用户看到的是成功，不应消耗预算 |
| **100 个请求，5 个最终失败** | 5 个（消耗） | 用户看到失败，消耗预算 |
| **100 个请求，重试 3 次后仍有 5 个失败** | 5 个（不是 5×3=15） | 按最终结果，不是总尝试数 |

**KYC 示例**：
- Error Budget = 1% = 10 个文档/月（1000 文档/月）
- 如果 100 个请求里有 5 个最终失败 → 消耗 5 个预算（不是 5×重试次数）

## 与 A4 自动重试的衔接

A4 的自动重试策略：

| 触发条件 | 重试策略 | Error Rate 计算 |
|----------|----------|----------------|
| **Success Rate < 98%** | 自动重试失败的请求（max_retries=3） | 重试后成功的 → 不计入失败；重试后仍失败的 → 计入 Error Rate |
| **重试后恢复 > 98%** | 告警解除 | Error Rate 下降（因为部分失败被重试成功） |
| **重试后仍 < 98%** | 升级到 Critical | Error Rate 按最终结果计算，触发告警 |

**关键**：重试是**恢复机制**，目的是降低 Error Rate；如果重试成功，Error Rate 应该**下降**，而不是因为「重试次数多」而上升。

## 重试后的 Error Rate 合理范围是多少？

### 一般系统的目标值

| 系统类型 | 重试后的 Error Rate 目标 | 说明 |
|----------|------------------------|------|
| **高可用系统**（如支付、KYC） | **< 0.1%**（万分之一） | 重试后仍有 0.1% 失败，通常对应不可恢复错误（数据问题、业务规则） |
| **一般业务系统** | **< 1%** | SLO = 99% Success Rate，允许 1% 最终失败 |
| **内部工具/批处理** | **< 5%** | 可接受更高失败率，人工介入成本低 |

### KYC 项目的目标

- **目标**：Error Rate < 1%（Production，重试后）
- **当前值**：5%（PoV 阶段，重试后）
- **SLO**：99% Success Rate = 1% Error Rate

**为什么不是万分之一（0.01%）？**

| 原因 | 说明 |
|------|------|
| **不可恢复错误必然存在** | 即使重试，仍有部分失败是**不可恢复**的：<br>- `IMAGE_FORMAT_UNSUPPORTED`：图片格式不支持，重试没用<br>- `SCHEMA_VALIDATION_FAILED`：数据不符合业务规则，重试没用<br>这些错误**必须人工处理或改输入**，无法通过重试消除 |
| **成本 vs 收益** | 要把 Error Rate 从 1% 降到 0.01%，需要：<br>- 更复杂的重试策略（可能不划算）<br>- 更多降级方案（开发成本高）<br>- 更严格的输入校验（可能误杀正常请求）<br>**1% 已经足够好**，再优化 ROI 低 |
| **与 SLO 对齐** | SLO = 99% = Error Rate 1%，这是对用户的承诺；0.01% 是**过度优化**，超出承诺范围 |

### 重试前后的对比（示例）

| 阶段 | Error Rate | 说明 |
|------|------------|------|
| **重试前（单次尝试）** | 2-5% | 包含可恢复错误（API_TIMEOUT 等）和不可恢复错误 |
| **重试后（最终结果）** | **< 1%** | 可恢复错误通过重试成功；剩余的是不可恢复错误 |

**KYC 示例**：
- 重试前：Error Rate = 5%（包含 1% API_TIMEOUT、1% API_CONNECTION、2% SCHEMA_VALIDATION、1% IMAGE_FORMAT）
- 自动重试：API_TIMEOUT 和 API_CONNECTION 重试 3 次后，大部分成功
- **重试后**：Error Rate ≈ 3%（主要是 SCHEMA_VALIDATION 和 IMAGE_FORMAT，不可恢复）
- **目标**：通过优化校验规则、输入过滤，降到 < 1%

### 如何达到 < 0.1%（万分之一）？

如果系统要求**重试后 Error Rate < 0.1%**，需要：

| 策略 | 说明 |
|------|------|
| **更严格的输入校验** | 在 pipeline 入口就拒绝不支持的格式，避免进入处理流程 |
| **降级方案** | 对不可恢复错误提供 fallback（如 OCR-only），而不是直接失败 |
| **人工审核队列** | 对可疑请求转人工，而不是直接标记为失败 |
| **成本** | 开发/运维成本显著上升，需要权衡 ROI |

**结论**：**< 1% 是常见且合理的目标**；< 0.1% 需要额外投入，通常只在关键系统（如支付）才值得。

---

## 设计要点小结

1. **限制重试次数**：MAX_RETRIES = 3，避免无限重试导致概率累积。
2. **Error Rate 按最终结果**：`最终失败的请求数 / 原始请求数`，不是 `总失败次数 / 总尝试次数`。
3. **指数退避**：1s → 2s → 4s，避免快速重试压垮上游。
4. **幂等性保证**：通过 `trace_id` 去重或结果覆盖，允许安全重试。
5. **错误预算按用户可见失败**：重试成功的不消耗预算；只有最终失败才消耗预算。
