## 核心问题

**问题**：不是所有 layer 的错误都发送给用户，那么：
1. 什么时候会造成**公司的损失**？
2. 什么时候会造成**客户的损失**？
3. 什么时候会造成**损失客户**（客户流失）？
4. 什么时候会造成**人力损失**（工程师 on-call）？

**答案**：**不是所有错误都发送给用户，也不是所有错误都触发 on-call**。通过**错误影响分级**，区分"用户可见错误"和"系统内部错误"，并根据**损失评估**决定响应策略。

---

## 错误影响分级（Error Impact Classification）

### 分级维度

| 维度 | 说明 | 示例 |
|------|------|------|
| **用户可见性** | 用户是否直接感知到错误 | `IMAGE_FORMAT_UNSUPPORTED` → 用户可见<br>`API_TIMEOUT`（自动重试成功）→ 用户不可见 |
| **影响范围** | 影响单个请求 vs 影响所有请求 | 单个文件失败 → 影响范围小<br>系统 down → 影响范围大 |
| **可恢复性** | 能否通过重试/降级恢复 | `API_TIMEOUT` → 可恢复<br>`IMAGE_FORMAT_UNSUPPORTED` → 不可恢复 |
| **损失类型** | 公司损失 / 客户损失 / 客户流失 / 人力损失 | 见下方详细说明 |

---

## 损失评估（Loss Assessment）

### 1. 公司损失（Company Loss）

**定义**：**直接的经济损失**（成本增加、收入减少、合规风险）

| 错误类型 | 损失原因 | 损失金额 | 触发条件 |
|---------|---------|---------|---------|
| **API 调用失败（未重试）** | 浪费 API 调用成本，但未产生价值 | $0.001-0.002 / request | `API_TIMEOUT` 且未重试成功 |
| **降级方案成本** | OCR-only 降级方案成本更高 | $0.003 / request（vs $0.0015 正常） | Circuit Breaker 触发，走降级 |
| **人工审核成本** | 转人工审核，增加人力成本 | $5-10 / request | Validation 失败，转人工 |
| **合规风险** | 错误拦截失败，导致合规问题 | $10,000-100,000 / 事件 | `SCHEMA_VALIDATION_FAILED` 漏检 |
| **系统可用性损失** | 系统 down，无法处理请求 | $100-1000 / 小时 | 系统完全不可用 |

**触发 on-call**：
- **Critical**：Error Rate > 5% 或系统可用性 < 95%
- **Emergency**：系统完全不可用

**KYC 示例**：
```
场景：API 失败率 10%，未触发自动重试
  - 公司损失：$0.0015 × 10,000 requests = $15 / 天
  - 触发条件：Error Rate > 5% → Critical 告警 → on-call
  - 响应：自动重试 + 人工介入（如果重试失败）
```

---

### 2. 客户损失（Customer Loss）

**定义**：**客户直接的经济损失**（时间成本、业务中断、数据丢失）

| 错误类型 | 损失原因 | 损失金额 | 触发条件 |
|---------|---------|---------|---------|
| **请求失败（用户可见）** | 用户需要重新提交，浪费时间 | 5-10 分钟 / request | `IMAGE_FORMAT_UNSUPPORTED`、`SCHEMA_VALIDATION_FAILED` |
| **延迟过高** | 用户等待时间过长，影响体验 | p95 > 15s → 用户体验差 | p95 > 15s |
| **降级方案准确率低** | OCR-only 准确率低，用户需要人工修正 | 10-20 分钟 / request | Circuit Breaker 触发，走降级 |
| **数据丢失** | 请求处理失败，数据未保存 | 不可量化（业务影响） | 系统 down，数据未持久化 |

**触发 on-call**：
- **Warning**：Success Rate < 98% 或 p95 > 15s（影响用户体验）
- **Critical**：Success Rate < 95% 或 p95 > 30s（严重影响用户体验）

**KYC 示例**：
```
场景：p95 = 20s（超过 15s 阈值）
  - 客户损失：95% 的用户等待时间 > 15s，体验差
  - 触发条件：p95 > 15s → Warning 告警 → on-call（如果持续）
  - 响应：自动限流 + 人工调查（如果持续）
```

---

### 3. 客户流失（Customer Churn）

**定义**：**客户因为体验差而流失**（长期影响，难以量化）

| 错误类型 | 流失原因 | 流失概率 | 触发条件 |
|---------|---------|---------|---------|
| **持续失败** | 用户多次提交失败，失去信任 | 10-20% / 月（如果持续） | Success Rate < 90% 持续 > 1 周 |
| **延迟过高** | 用户等待时间过长，转向竞品 | 5-10% / 月（如果持续） | p95 > 30s 持续 > 1 周 |
| **系统不稳定** | 用户认为系统不可靠 | 20-30% / 月（如果持续） | 系统可用性 < 95% 持续 > 1 周 |
| **数据丢失** | 用户数据丢失，失去信任 | 50%+ / 事件 | 系统 down，数据未持久化 |

**触发 on-call**：
- **Critical**：Success Rate < 90% 持续 > 1 小时
- **Emergency**：系统完全不可用

**KYC 示例**：
```
场景：Success Rate = 85% 持续 2 天
  - 客户流失风险：15% 的用户体验差，可能流失
  - 触发条件：Success Rate < 90% 持续 > 1 小时 → Critical 告警 → on-call
  - 响应：自动回滚 + 紧急修复 + 客户沟通
```

---

### 4. 人力损失（Human Loss / On-Call）

**定义**：**工程师需要 on-call，影响工作效率和生活质量**

| 错误类型 | 人力损失原因 | 人力成本 | 触发条件 |
|---------|------------|---------|---------|
| **告警疲劳** | 频繁告警，工程师疲于应对 | 2-4 小时 / 天（如果持续） | 告警噪音 > 50%（无效告警占比） |
| **半夜告警** | 半夜被叫醒，影响睡眠 | 不可量化（生活质量） | Critical / Emergency 告警在非工作时间 |
| **紧急修复** | 需要立即修复，打断正常工作 | 4-8 小时 / 事件 | Critical / Emergency 告警 |
| **Postmortem** | 需要写事故报告，分析根因 | 2-4 小时 / 事件 | Critical / Emergency 告警后 |

**触发 on-call**：
- **Warning**：Success Rate < 98% 或 Error Rate > 2%（自动重试/降级，通知 on-call）
- **Critical**：Success Rate < 95% 或 Error Rate > 5%（自动熔断/回滚，立即通知）
- **Emergency**：系统完全不可用（自动回滚，紧急通知）

**KYC 示例**：
```
场景：Error Rate = 6%（超过 5% 阈值）
  - 人力损失：工程师需要 on-call，调查原因，可能需要回滚
  - 触发条件：Error Rate > 5% → Critical 告警 → on-call（立即）
  - 响应：自动熔断 + 降级 + 人工调查 + 可能回滚
```

## 错误响应策略（Error Response Strategy）

### 策略矩阵

| 错误类型 | 用户可见 | 影响范围 | 可恢复性 | 损失类型 | 响应策略 | 是否 on-call |
|---------|---------|---------|---------|---------|---------|-------------|
| **Layer 1: Preprocessing** | ✅ 是 | 单个请求 | ❌ 不可恢复 | 客户损失（时间） | 直接返回错误，不进入下一层 | ❌ 否（除非占比 > 2%） |
| **Layer 2: Rate Limiter** | ✅ 是 | 单个请求 | ❌ 不可恢复 | 客户损失（时间） | 返回 429，不进入下一层 | ❌ 否（除非持续 > 1 小时） |
| **Layer 3: Circuit Breaker** | ⚠️ 部分（降级） | 部分请求 | ✅ 可恢复（降级） | 公司损失（降级成本）+ 客户损失（准确率低） | 走降级方案，不进入下一层 | ⚠️ 条件（如果降级后仍失败） |
| **Layer 4: API Call（自动重试成功）** | ❌ 否 | 单个请求 | ✅ 可恢复 | 公司损失（重试成本） | 自动重试，用户无感知 | ❌ 否 |
| **Layer 4: API Call（自动重试失败）** | ✅ 是 | 单个请求 | ❌ 不可恢复 | 客户损失（时间） | 返回错误，或走降级 | ⚠️ 条件（如果占比 > 2%） |
| **Layer 5: Validation（转人工）** | ⚠️ 部分（延迟） | 单个请求 | ✅ 可恢复（人工） | 公司损失（人力成本） | 转人工审核，不直接失败 | ❌ 否（除非人工队列积压） |
| **系统 down** | ✅ 是 | 所有请求 | ❌ 不可恢复 | 公司损失 + 客户损失 + 客户流失 + 人力损失 | 自动回滚，紧急通知 | ✅ 是（Emergency） |

## 什么时候触发 on-call？

### 触发条件总结

| 严重性 | 触发条件 | 损失类型 | 响应时间 | 是否 on-call |
|--------|---------|---------|---------|-------------|
| **Info** | 指标轻微波动 | 无损失 | 1小时内 | ❌ 否 |
| **Warning** | Success Rate < 98% 或 Error Rate > 2% | 客户损失（体验差） | 15分钟内 | ⚠️ 条件（如果持续） |
| **Critical** | Success Rate < 95% 或 Error Rate > 5% | 公司损失 + 客户损失 + 客户流失风险 | 立即（< 5分钟） | ✅ 是（立即） |
| **Emergency** | 系统完全不可用 | 公司损失 + 客户损失 + 客户流失 + 人力损失 | 立即（< 1分钟） | ✅ 是（紧急） |

### 不触发 on-call 的情况

| 情况 | 原因 | 损失评估 |
|------|------|---------|
| **单个请求失败（用户可见）** | 影响范围小，不影响系统可用性 | 客户损失（时间），但可接受 |
| **自动重试成功** | 用户无感知，系统自动恢复 | 无损失（或公司损失：重试成本） |
| **降级方案成功** | 系统可用，用户体验降级但可接受 | 公司损失（降级成本）+ 客户损失（准确率低），但可接受 |
| **转人工审核** | 系统可用，只是延迟 | 公司损失（人力成本），但可接受 |
| **告警噪音 > 50%** | 无效告警太多，需要优化阈值 | 人力损失（告警疲劳），需要优化 |

## 设计原则

### 1. 不是所有错误都发送给用户

**原则**：**只有"用户可见错误"才发送给用户**。

| 错误类型 | 是否发送给用户 | 原因 |
|---------|--------------|------|
| **Layer 1: Preprocessing** | ✅ 是 | 用户需要知道格式错误，可以重新提交 |
| **Layer 2: Rate Limiter** | ✅ 是 | 用户需要知道被限流，可以稍后重试 |
| **Layer 3: Circuit Breaker（降级成功）** | ⚠️ 部分 | 用户可能感知到准确率低，但不直接失败 |
| **Layer 4: API Call（自动重试成功）** | ❌ 否 | 用户无感知，系统自动恢复 |
| **Layer 4: API Call（自动重试失败）** | ✅ 是 | 用户需要知道失败，可以重新提交 |
| **Layer 5: Validation（转人工）** | ⚠️ 部分 | 用户可能感知到延迟，但不直接失败 |

### 2. 不是所有错误都触发 on-call

**原则**：**只有"影响系统可用性或用户体验"的错误才触发 on-call**。

| 错误类型 | 是否触发 on-call | 原因 |
|---------|----------------|------|
| **单个请求失败** | ❌ 否 | 影响范围小，不影响系统可用性 |
| **自动重试成功** | ❌ 否 | 用户无感知，系统自动恢复 |
| **降级方案成功** | ⚠️ 条件 | 如果降级后仍失败，触发 on-call |
| **Error Rate > 2%** | ⚠️ 条件 | 如果持续 > 1 小时，触发 on-call |
| **Error Rate > 5%** | ✅ 是 | 严重影响系统可用性，必须立即介入 |
| **系统 down** | ✅ 是 | 系统完全不可用，必须立即介入 |

### 3. 损失评估决定响应策略

**原则**：**根据损失类型和损失金额决定响应策略**。

| 损失类型 | 损失金额 | 响应策略 |
|---------|---------|---------|
| **公司损失（小）** | < $100 / 天 | 记录日志，无需立即处理 |
| **公司损失（中）** | $100-1000 / 天 | Warning 告警，自动重试/降级 |
| **公司损失（大）** | > $1000 / 天 | Critical 告警，自动熔断/回滚 |
| **客户损失（小）** | 个别用户 | 记录日志，无需立即处理 |
| **客户损失（中）** | 部分用户 | Warning 告警，自动重试/降级 |
| **客户损失（大）** | 大部分用户 | Critical 告警，自动熔断/回滚 |
| **客户流失风险** | 持续 > 1 周 | Critical 告警，自动回滚 + 客户沟通 |
| **人力损失** | 告警疲劳 | 优化告警阈值，减少无效告警 |

## 总结

### 核心设计思路

1. **错误影响分级**：区分"用户可见错误"和"系统内部错误"
2. **损失评估**：根据损失类型和损失金额决定响应策略
3. **不是所有错误都发送给用户**：只有"用户可见错误"才发送给用户
4. **不是所有错误都触发 on-call**：只有"影响系统可用性或用户体验"的错误才触发 on-call

### 触发 on-call 的条件

- **Warning**：Success Rate < 98% 或 Error Rate > 2%（如果持续 > 1 小时）
- **Critical**：Success Rate < 95% 或 Error Rate > 5%（立即）
- **Emergency**：系统完全不可用（立即）

### 不触发 on-call 的情况

- 单个请求失败（用户可见，但影响范围小）
- 自动重试成功（用户无感知）
- 降级方案成功（系统可用，用户体验降级但可接受）
- 转人工审核（系统可用，只是延迟）
