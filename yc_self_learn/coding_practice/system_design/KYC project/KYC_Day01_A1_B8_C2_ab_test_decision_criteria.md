## 核心问题

**问题**：什么时候会选择进行 A/B Test？一般的考虑是如何的？

**答案**：**不是所有场景都需要 A/B Test**。选择 A/B Test 的决策标准包括：**不确定性高**、**影响大**、**成本可接受**、**时间允许**。业界通常使用**决策框架**来判断是否进行 A/B Test。

---

## 决策标准：什么时候选择 A/B Test？

### 1. 核心决策框架

**决策标准**（按优先级排序）：

| 标准 | 说明 | 示例 |
|------|------|------|
| **不确定性高** | 对设计决策的影响不确定，需要数据验证 | "p95 > 15s 会导致流失率增加吗？" |
| **影响大** | 设计决策的影响范围大，需要量化 | "改变重试策略会影响 100 万用户" |
| **成本可接受** | A/B Test 的成本（时间、资源）可接受 | 实验时长 2 周，样本量 10,000 用户 |
| **时间允许** | 有足够时间进行实验和分析 | 不是紧急修复，可以等待 2 周 |

---

### 2. 决策矩阵

**决策矩阵**（根据不确定性和影响）：

| 不确定性 | 影响大 | 影响小 |
|---------|-------|-------|
| **高** | ✅ **必须 A/B Test** | ⚠️ **考虑 A/B Test** |
| **低** | ⚠️ **考虑 A/B Test** | ❌ **不需要 A/B Test** |

**示例**：

| 场景 | 不确定性 | 影响 | 决策 |
|------|---------|------|------|
| **验证 p95 阈值** | 高（不知道用户等待多久会流失） | 大（影响所有用户） | ✅ **必须 A/B Test** |
| **验证重试次数** | 高（不知道 3 次 vs 5 次的影响） | 大（影响所有请求） | ✅ **必须 A/B Test** |
| **验证告警阈值** | 中（有行业基准，但需要验证） | 大（影响系统稳定性） | ⚠️ **考虑 A/B Test** |
| **验证降级方案** | 高（不知道用户接受度） | 大（影响用户体验） | ✅ **必须 A/B Test** |
| **日常监控** | 低（监控指标已经足够） | 小（只是监控） | ❌ **不需要 A/B Test** |
| **紧急修复** | 低（问题明确） | 大（但时间紧迫） | ❌ **不需要 A/B Test**（直接修复） |

## 考虑因素

### 1. 不确定性（Uncertainty）

**定义**：对设计决策的影响不确定，需要数据验证

**判断标准**：

| 情况 | 不确定性 | 是否需要 A/B Test |
|------|---------|-----------------|
| **有行业基准，但需要验证** | 中 | ⚠️ 考虑 A/B Test |
| **没有行业基准，完全不确定** | 高 | ✅ 必须 A/B Test |
| **有历史数据，但需要验证新场景** | 中 | ⚠️ 考虑 A/B Test |
| **问题明确，不需要验证** | 低 | ❌ 不需要 A/B Test |

**KYC 示例**：
```
场景：验证 p95 阈值

不确定性分析：
  - 行业基准：Google p95 < 10s，Amazon p95 < 15s
  - 历史数据：过去 3 个月，p95 > 15s 时流失率增加 5%
  - 但不确定：p95 = 15s 是否是临界点？p95 = 18s 是否可以接受？

结论：不确定性高 → 必须 A/B Test
```

---

### 2. 影响范围（Impact）

**定义**：设计决策的影响范围（用户数、请求数、成本等）

**判断标准**：

| 影响范围 | 说明 | 是否需要 A/B Test |
|---------|------|-----------------|
| **影响所有用户** | 影响 100% 用户 | ✅ 必须 A/B Test |
| **影响大部分用户** | 影响 > 50% 用户 | ✅ 必须 A/B Test |
| **影响部分用户** | 影响 < 50% 用户 | ⚠️ 考虑 A/B Test |
| **影响个别用户** | 影响 < 1% 用户 | ❌ 不需要 A/B Test |

**KYC 示例**：
```
场景：验证重试次数（3 次 vs 5 次）

影响范围分析：
  - 影响所有请求（100%）
  - 每天 100 万请求
  - 影响 API 调用成本、用户体验、系统稳定性

结论：影响大 → 必须 A/B Test
```

---

### 3. 成本（Cost）

**定义**：A/B Test 的成本（时间、资源、人力）

**判断标准**：

| 成本类型 | 说明 | 可接受范围 |
|---------|------|-----------|
| **时间成本** | 实验时长 | < 4 周（通常 1-2 周） |
| **资源成本** | 样本量、基础设施 | < 10% 总流量（通常 5%） |
| **人力成本** | 设计、分析、维护 | < 1 人周（通常 0.5 人周） |

**KYC 示例**：
```
场景：验证 p95 阈值

成本分析：
  - 时间成本：2 周（可接受）
  - 资源成本：每组 10,000 用户（总用户 100,000，10% 流量，可接受）
  - 人力成本：0.5 人周（设计 + 分析，可接受）

结论：成本可接受 → 可以进行 A/B Test
```

---

### 4. 时间（Time）

**定义**：是否有足够时间进行实验和分析

**判断标准**：

| 情况 | 时间要求 | 是否需要 A/B Test |
|------|---------|-----------------|
| **不是紧急修复** | 有 1-2 周时间 | ✅ 可以进行 A/B Test |
| **紧急修复** | 需要立即修复 | ❌ 不需要 A/B Test（直接修复） |
| **长期优化** | 有 1-3 个月时间 | ✅ 可以进行 A/B Test |
| **短期优化** | 只有几天时间 | ❌ 不需要 A/B Test（用其他方法） |

**KYC 示例**：
```
场景：验证告警阈值

时间分析：
  - 不是紧急修复（系统正常运行）
  - 有 2 周时间进行实验
  - 可以等待结果再调整阈值

结论：时间允许 → 可以进行 A/B Test
```

---

### 5. 风险（Risk）

**定义**：A/B Test 的风险（用户体验、系统稳定性、业务影响）

**判断标准**：

| 风险类型 | 说明 | 是否需要 A/B Test |
|---------|------|-----------------|
| **低风险** | 影响小，可以回滚 | ✅ 可以进行 A/B Test |
| **中风险** | 影响中等，需要谨慎 | ⚠️ 考虑 A/B Test（小样本量） |
| **高风险** | 影响大，难以回滚 | ❌ 不需要 A/B Test（用其他方法） |

**KYC 示例**：
```
场景：验证降级方案

风险分析：
  - 风险：降级方案准确率低，可能影响用户体验
  - 但可以回滚（Feature Flag）
  - 样本量小（5% 流量），影响可控

结论：风险可控 → 可以进行 A/B Test
```

## 业界最佳实践：决策流程

### 1. 决策流程

```
1. 评估不确定性
   ↓
2. 评估影响范围
   ↓
3. 评估成本和时间
   ↓
4. 评估风险
   ↓
5. 决策：是否进行 A/B Test
```

---

### 2. 决策检查清单

**在进行 A/B Test 前，检查以下问题**：

| 问题 | 答案 | 决策 |
|------|------|------|
| **不确定性高吗？** | 是 → 需要数据验证 | ✅ 考虑 A/B Test |
| **影响范围大吗？** | 是 → 影响 > 50% 用户 | ✅ 考虑 A/B Test |
| **成本可接受吗？** | 是 → 时间 < 4 周，资源 < 10% | ✅ 考虑 A/B Test |
| **时间允许吗？** | 是 → 不是紧急修复 | ✅ 考虑 A/B Test |
| **风险可控吗？** | 是 → 可以回滚，影响可控 | ✅ 考虑 A/B Test |
| **有明确的假设吗？** | 是 → 有明确的假设和指标 | ✅ 可以进行 A/B Test |

**如果所有答案都是"是"** → ✅ **进行 A/B Test**

**如果有 1-2 个答案是"否"** → ⚠️ **考虑其他方法**（如行业基准、历史数据）

**如果有 3+ 个答案是"否"** → ❌ **不需要 A/B Test**（用其他方法）

---

### 3. 业界实践对比

| 公司 | 决策标准 | 实践 |
|------|---------|------|
| **Google** | 不确定性高 + 影响大 | 大量 A/B Test，快速迭代 |
| **Amazon** | 影响大 + 成本可接受 | 定期 A/B Test，验证关键决策 |
| **Netflix** | 不确定性高 + 风险可控 | 大量 A/B Test，验证用户体验 |
| **Facebook** | 影响大 + 时间允许 | 定期 A/B Test，验证产品功能 |

**共同点**：
1. **不确定性高** → 必须 A/B Test
2. **影响大** → 必须 A/B Test
3. **成本可接受** → 可以进行 A/B Test
4. **时间允许** → 可以进行 A/B Test

---

## 什么时候不需要 A/B Test？

### 1. 不需要 A/B Test 的场景

| 场景 | 原因 | 替代方法 |
|------|------|---------|
| **紧急修复** | 时间紧迫，需要立即修复 | 直接修复，监控验证 |
| **问题明确** | 不确定性低，问题明确 | 直接修复，不需要验证 |
| **影响小** | 影响 < 1% 用户 | 直接实施，监控验证 |
| **成本高** | 时间 > 4 周，资源 > 20% | 用其他方法（行业基准、历史数据） |
| **风险高** | 难以回滚，影响大 | 用其他方法（小范围试点、Canary 发布） |

---

### 2. 替代方法

**当不需要 A/B Test 时，可以使用以下方法**：

| 方法 | 适用场景 | 示例 |
|------|---------|------|
| **行业基准** | 有业界标准，不确定性低 | Google p95 < 10s → 设定 p95 阈值 = 15s |
| **历史数据** | 有历史数据，可以分析规律 | 过去 3 个月，p95 > 15s 时流失率增加 5% |
| **监控验证** | 实施后监控，验证效果 | 直接实施，监控指标，如果异常 → 回滚 |
| **小范围试点** | 风险高，先小范围试点 | 5% 流量试点，如果成功 → 全量 |
| **Canary 发布** | 风险高，逐步发布 | 10% → 50% → 100%，每步监控 |

---

## 总结

### 核心决策标准

**选择 A/B Test 的决策标准**：

1. **不确定性高** → 需要数据验证
2. **影响大** → 影响 > 50% 用户
3. **成本可接受** → 时间 < 4 周，资源 < 10%
4. **时间允许** → 不是紧急修复
5. **风险可控** → 可以回滚，影响可控

### 决策流程

```
1. 评估不确定性 → 高 → 考虑 A/B Test
2. 评估影响范围 → 大 → 考虑 A/B Test
3. 评估成本和时间 → 可接受 → 考虑 A/B Test
4. 评估风险 → 可控 → 考虑 A/B Test
5. 决策：是否进行 A/B Test
```

### 业界最佳实践

- **Google / Amazon / Netflix / Facebook**：不确定性高 + 影响大 → 必须 A/B Test
- **不是所有场景都需要 A/B Test**：紧急修复、问题明确、影响小 → 不需要 A/B Test
- **替代方法**：行业基准、历史数据、监控验证、小范围试点、Canary 发布
