## K8s是什么

### 简单定义

**Kubernetes (K8s)** 是一个**容器编排平台**，用于自动化部署、扩展和管理容器化应用程序。

### 类比理解

**把K8s想象成"智能的服务器管理员"**：

```
传统方式（手动管理）：
┌─────────────────────────────────┐
│  你手动管理服务器                 │
│  • 手动启动/停止应用              │
│  • 手动分配资源                   │
│  • 手动处理故障                   │
│  • 手动扩展/缩容                  │
└─────────────────────────────────┘

K8s方式（自动化管理）：
┌─────────────────────────────────┐
│  K8s自动管理服务器                │
│  • 自动启动/停止应用              │
│  • 自动分配资源                   │
│  • 自动处理故障（重启、迁移）      │
│  • 自动扩展/缩容                  │
└─────────────────────────────────┘
```

## 为什么叫K8s

**K8s = K + 8个字母 + s = Kubernetes**

- K**ubernete**s
- K + **8个字母** + s = **K8s**

这是Kubernetes的缩写形式。

## K8s的核心概念

### 1. Pod（容器组）

**Pod是K8s的最小部署单位**，通常包含一个或多个容器。

```
Pod = 一个或多个容器的组合
```

**示例**：
```yaml
# 一个Pod，包含SGLang Worker容器
apiVersion: v1
kind: Pod
metadata:
  name: sglang-worker-0
spec:
  containers:
    - name: sglang
      image: lmsysorg/sglang:latest
      resources:
        requests:
          nvidia.com/gpu: 1  # 需要1个GPU
```

### 2. Node（节点）

**Node是运行Pod的服务器**（物理机或虚拟机）。

```
Node = 服务器
```

**示例**：
```
Node 1 (GPU服务器)
├─ Pod: sglang-worker-0 (GPU 0)
├─ Pod: sglang-worker-1 (GPU 1)
└─ Pod: sglang-worker-2 (GPU 2)
```

### 3. Cluster（集群）

**Cluster是一组Node的集合**。

```
Cluster = 多个Node组成的集群
```

**示例**：
```
Cluster (GPU集群)
├─ Node 1 (A100×1)
├─ Node 2 (4090×2)
└─ Node 3 (L40S×2)
```

### 4. Service（服务）

**Service是Pod的访问入口**，提供稳定的IP地址和端口。

```
Service = Pod的"门牌号"
```

**示例**：
```yaml
# Service提供统一的访问入口
apiVersion: v1
kind: Service
metadata:
  name: sglang-service
spec:
  selector:
    app: sglang-worker
  ports:
    - port: 30000
      targetPort: 30000
```

**访问方式**：
```
原来：http://worker-0:30000  (Pod IP会变)
现在：http://sglang-service:30000  (Service IP稳定)
```

### 5. Deployment（部署）

**Deployment管理Pod的创建和更新**。

```
Deployment = Pod的"管理员"
```

**示例**：
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: sglang-deployment
spec:
  replicas: 3  # 3个Pod副本
  selector:
    matchLabels:
      app: sglang-worker
  template:
    metadata:
      labels:
        app: sglang-worker
    spec:
      containers:
        - name: sglang
          image: lmsysorg/sglang:latest
```

**Deployment的作用**：
- ✅ 自动创建Pod
- ✅ 自动重启失败的Pod
- ✅ 支持滚动更新（零停机）

## K8s在Rad-Linter中的作用

### 场景1：管理SGLang Worker

**不用K8s（手动管理）**：
```bash
# 手动启动每个Worker
docker run -d --gpus device=0 sglang-worker-0
docker run -d --gpus device=1 sglang-worker-1
docker run -d --gpus device=2 sglang-worker-2

# 如果Worker挂了，需要手动重启
docker restart sglang-worker-0
```

**用K8s（自动管理）**：
```yaml
# 定义Deployment
apiVersion: apps/v1
kind: Deployment
metadata:
  name: sglang-workers
spec:
  replicas: 3  # 自动创建3个Worker
  template:
    spec:
      containers:
        - name: sglang
          image: lmsysorg/sglang:latest
          resources:
            requests:
              nvidia.com/gpu: 1
```

**K8s自动处理**：
- ✅ 自动创建3个Worker Pod
- ✅ 如果Worker挂了，自动重启
- ✅ 如果Node挂了，自动迁移到其他Node

### 场景2：GPU资源调度

**K8s可以自动决定Pod在哪个Node运行**：

```
请求：需要1个GPU
    ↓
K8s调度器：
  • 检查所有Node的GPU可用性
  • 选择GPU空闲的Node
  • 在该Node上创建Pod
```

**示例**：
```yaml
# K8s自动选择有GPU的Node
apiVersion: v1
kind: Pod
spec:
  containers:
    - name: sglang
      resources:
        requests:
          nvidia.com/gpu: 1  # K8s自动找有GPU的Node
```

### 场景3：服务发现和负载均衡

**K8s提供Service，自动负载均衡**：

```yaml
# Service自动负载均衡到多个Pod
apiVersion: v1
kind: Service
metadata:
  name: sglang-service
spec:
  selector:
    app: sglang-worker
  ports:
    - port: 30000
```

**访问方式**：
```
请求 → sglang-service:30000
    ↓
K8s自动分发到：
  • sglang-worker-0
  • sglang-worker-1
  • sglang-worker-2
```

## K8s vs 其他方案

### K8s vs Docker Compose

| 特性 | Docker Compose | Kubernetes |
|-----|---------------|------------|
| **适用场景** | 单机、开发环境 | 集群、生产环境 |
| **自动重启** | ❌ 需要手动 | ✅ 自动 |
| **自动扩展** | ❌ 不支持 | ✅ 支持 |
| **故障迁移** | ❌ 不支持 | ✅ 支持 |
| **资源调度** | ❌ 不支持 | ✅ 支持 |
| **复杂度** | ⭐⭐ 简单 | ⭐⭐⭐⭐ 复杂 |

**选择建议**：
- **开发/测试**：Docker Compose（简单）
- **生产环境**：Kubernetes（功能强大）

### K8s vs 手动管理

| 特性 | 手动管理 | Kubernetes |
|-----|---------|------------|
| **部署时间** | 手动，慢 | 自动化，快 |
| **故障处理** | 手动，慢 | 自动，快 |
| **扩展性** | 手动，困难 | 自动，容易 |
| **资源利用率** | 低 | 高 |

## 快速理解

### 核心概念速记

```
K8s = 容器编排平台
  │
  ├─ Pod = 容器（应用）
  ├─ Node = 服务器
  ├─ Cluster = 服务器集群
  ├─ Service = 访问入口
  └─ Deployment = Pod管理器
```

### 工作流程

```
1. 你定义：要运行什么应用（Pod）
2. K8s自动：选择在哪个Node运行
3. K8s自动：创建Pod
4. K8s自动：监控Pod健康
5. K8s自动：如果挂了，重启
6. K8s自动：提供Service访问入口
```

### 在Rad-Linter中的实际应用

```
Rad-Linter架构：
  │
  ├─ SGLang Worker (Pod)
  │   └─ K8s管理：自动创建、重启、扩展
  │
  ├─ Envoy Router (Pod)
  │   └─ K8s管理：自动负载均衡
  │
  ├─ Prometheus (Pod)
  │   └─ K8s管理：自动监控
  │
  └─ GPU资源
      └─ K8s调度：自动分配GPU
```

## 总结

### K8s是什么

**Kubernetes (K8s)** 是一个**容器编排平台**，用于：
- ✅ 自动化部署应用
- ✅ 自动管理资源
- ✅ 自动处理故障
- ✅ 自动扩展/缩容

### 在Rad-Linter中的作用

1. **管理SGLang Worker**：自动创建、重启、扩展
2. **GPU资源调度**：自动分配GPU资源
3. **服务发现**：提供稳定的访问入口
4. **负载均衡**：自动分发请求

### 是否需要K8s？

**需要K8s的场景**：
- ✅ 生产环境
- ✅ 多台服务器
- ✅ 需要自动扩展
- ✅ 需要高可用

**不需要K8s的场景**：
- ✅ 开发/测试环境
- ✅ 单台服务器
- ✅ 简单应用
- ✅ 快速验证

**对于Rad-Linter**：
- **快速落地**：可以不用K8s（用Docker Compose）
- **生产环境**：建议用K8s（自动管理、高可用）
