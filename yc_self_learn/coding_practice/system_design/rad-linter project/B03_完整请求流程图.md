## 请求示例

### 场景描述

**医生操作**：医生在RIS系统中完成了一份胸部X光报告，点击"提交"按钮

**系统任务**：Rad-Linter系统需要检查这份报告的质量，确保：
- 报告与影像一致（没有遗漏关键发现）
- 没有左右侧错误
- 测量值准确
- 符合临床规范

### 输入数据

```json
{
  "case_id": "CASE_20250125_001",
  "patient_id": "P001",
  "report_text": "Chest X-ray shows no acute cardiopulmonary abnormality. Heart size is normal. Lungs are clear bilaterally.",
  "dicom_images": [
    "dicom://pacs/study_001/series_001/image_001.dcm",
    "dicom://pacs/study_001/series_001/image_002.dcm"
  ],
  "study_type": "Chest X-ray",
  "timestamp": "2025-01-25T10:30:00Z"
}
```

## 详细步骤说明

### Step 0: Ingestion (数据接入)

**输入**：
```json
POST /api/v1/lint
{
  "patient_id": "P001",
  "report_text": "Chest X-ray shows no acute cardiopulmonary abnormality...",
  "dicom_images": ["dicom://pacs/study_001/..."]
}
```

**处理**：
1. 接收HTTP请求
2. 生成唯一`case_id`: `CASE_20250125_001`
3. 幂等性检查（检查是否已处理过）
4. 保存原始快照到存储（用于审计重放）
5. 加入实时队列`q_realtime`（RabbitMQ/Redis Queue）

**输出**：
- `case_id`: `CASE_20250125_001`
- 队列消息已入队

**耗时**：50-100ms

---

### Step 1: Schema Gate (契约层)

**输入**：队列中的消息

**处理**：
1. 解析JSON
2. PII脱敏（如需要）
3. Schema验证（CaseInputV1）
   - 检查必需字段
   - 验证数据格式
4. 快速失败：如果Schema不通过，返回400错误

**输出**：验证通过的CaseInputV1对象

**耗时**：50-100ms

---

### Step 2: Evidence Builder (证据构建)

**输入**：CaseInputV1

**处理**（并行）：
1. **Text Facts Extraction** (CPU)
   - NER: 提取实体
   - 关系抽取
   - 否定识别
   - 输出: `report_facts_v1.jsonl`

2. **Visual Facts Extraction** (GPU/CPU)
   - 从PACS获取DICOM影像
   - 检测/分割/测量
   - Laterality识别
   - 输出: `visual_facts_v1.jsonl`

3. **Fact Store**
   - 版本化存储
   - 计算hash
   - 保存到数据库

**输出**：
- `report_facts_v1.jsonl`
- `visual_facts_v1.jsonl`

**耗时**：1-3秒（并行处理）

---

### Step 3: Rule Gate (规则闸门)

**输入**：report_facts + visual_facts

**处理**：
1. Laterality检查
2. Measurement检查
3. Required Fields检查
4. 规则引擎输出：
   - Hard Fail: 严重错误，直接拦截
   - Soft Flag: 需要LLM进一步判断
   - Pass: 所有规则通过

**输出**：
- Hard Fail → 直接返回结果
- Soft Flag → 进入LLM Judge
- Pass → 进入Policy Gate

**耗时**：10-50ms

---

### Step 4: LLM Judge (GPU任务) ⭐

**这是GPU调度的核心部分**

#### 4.1 K8s Service负载均衡

**K8s Service自动处理**：
- Service自动发现所有匹配的Pod（通过Selector）
- kube-proxy自动负载均衡（Round Robin）
- Readiness Probe自动过滤不健康的Pod

**无需手动选择Worker**：
- ✅ K8s Service自动处理
- ✅ 无需查询Prometheus
- ✅ 无需计算得分
- ✅ 简单实用

#### 4.2 通过K8s Service路由

**Service配置**：
```yaml
apiVersion: v1
kind: Service
metadata:
  name: sglang-service
spec:
  selector:
    app: sglang-worker
  ports:
    - port: 30000
      targetPort: 30000
  type: ClusterIP
```

**HTTP请求**：
```
POST http://sglang-service:30000/generate
{
  "report_facts": [...],
  "visual_facts": [...]
}

# K8s Service自动路由到某个Pod
# 例如: 10.244.1.2:30000 (Pod: worker-1)
```

#### 4.3 SGLang Worker处理

**GPU推理**：
1. 接收请求
2. 动态批处理（如果有其他请求，合并处理）
3. LLM推理（Pass A/B/C）：
   - Pass A: 对齐检查
   - Pass B: 风险分级
   - Pass C: 解释生成
4. 输出结构化JSON

**输出**：LintResultV1

**耗时**：2-8秒（GPU推理，主要延迟来源）

---

### Step 5: Policy Gate (策略层)

**输入**：LintResultV1

**处理**：
1. 根据severity和confidence决定action
2. 策略决策：
   - Auto-approve: 无风险
   - Suggest edit: 中等风险
   - Block sign-off: 高风险
   - Escalate: 不确定

**输出**：最终action和结果

**耗时**：10-50ms

---

### Step 6: Human-in-the-loop (人工复核)

**输入**：Policy Gate的结果

**处理**：
1. 显示给医生（如果是suggest_edit或block）
2. 医生操作（接受/忽略）
3. 反馈回流（用于模型优化）

**耗时**：取决于医生响应时间

---

### Step 7: Observability & Audit (可观测性)

**处理**（异步）：
1. 审计日志记录
2. Metrics记录（Prometheus）
3. Traces记录（Jaeger）
4. 可重放性保证

**耗时**：异步，不影响主流程

## GPU调度详细流程

### 详细时序

```
时间轴 (毫秒)
│
├─ 0ms: Rule Gate输出Soft Flag，需要LLM Judge
│
├─ 10ms: 请求到达K8s Service
│        - Service: sglang-service:30000
│        - kube-proxy解析Service
│
├─ 20ms: kube-proxy负载均衡
│        - 查找Endpoints
│        - Round Robin选择Pod
│        - 例如: 选择worker-1
│
├─ 30ms: kube-proxy转发请求
│        - 转发到Pod IP: 10.244.1.2:30000
│        - 请求体: {report_facts, visual_facts}
│
├─ 100ms: SGLang Worker接收请求
│         - 加入内部队列
│         - 等待批处理
│
├─ 200ms: SGLang动态批处理
│         - 如果有其他请求，合并处理
│         - 开始GPU推理
│
├─ 2200ms: GPU推理完成
│          - Pass A: 对齐检查完成
│          - Pass B: 风险分级完成
│          - Pass C: 解释生成完成
│
├─ 2250ms: SGLang返回结果
│          - HTTP 200 OK
│          - 响应体: LintResultV1
│
├─ 2270ms: Pod返回结果给kube-proxy
│
└─ 2280ms: kube-proxy返回结果给Service → Policy Gate
```

```
需要LLM Judge?

├─ 是

│  │  │  │

│  │  │     │

│  │  │
│  │  └─ 所有Worker都不可用?
│  │     │
│  │     └─ 是 → Fallback到规则引擎
│  │
│  └─ 否 → 跳过LLM Judge
```

## 关键时间点

### 端到端延迟分解

| 步骤 | 耗时 | 占比 | 说明 |
|-----|------|------|------|
| **Ingestion** | 50-100ms | 1% | 数据接入、队列入队 |
| **Schema Gate** | 50-100ms | 1% | Schema验证 |
| **Evidence Builder** | 1-3s | 15-30% | 证据提取（并行） |
| **Rule Gate** | 10-50ms | 0.5% | 规则检查 |
| **LLM Judge (GPU)** | 2-8s | 60-80% | **主要延迟来源** |
| **Policy Gate** | 10-50ms | 0.5% | 策略决策 |
| **Human-in-the-loop** | 可变 | - | 取决于医生响应 |
| **总计** | **4-12s** | 100% | P95目标: < 8s |

### LLM Judge延迟分解

| 步骤 | 耗时 | 说明 |
|-----|------|------|
| **K8s Service负载均衡** | 10-30ms | kube-proxy自动路由 |
| **SGLang队列等待** | 0-500ms | 等待批处理 |
| **GPU推理** | 2-6s | **主要延迟** |
| **返回结果** | 10-20ms | 网络传输 |
| **总计** | **2-8s** | P95目标: < 8s |

## 总结

### 完整流程要点

1. **串行流程**：Ingestion → Schema → Evidence → Rule → LLM Judge → Policy
2. **GPU调度核心**：K8s Service自动负载均衡到Worker Pods
3. **主要延迟**：LLM Judge的GPU推理（2-8秒）
4. **SLO目标**：P95 < 8秒（端到端）

### 关键设计

1. **两级队列**：实时队列（q_realtime）优先处理
2. **自动负载均衡**：K8s Service自动分发请求
3. **批处理**：SGLang动态批处理提高吞吐
4. **可观测性**：全程记录，支持审计重放
