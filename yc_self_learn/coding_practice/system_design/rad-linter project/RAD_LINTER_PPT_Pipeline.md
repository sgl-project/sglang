## 🎯 1分钟版本（Elevator Pitch）

**总共 1-2 页 slide，核心是快速传达 Pipeline 价值**

### Slide 1: Pipeline 介绍 + 核心流程（60秒讲解）

**标题**：rad-linter 多模态 LLM 训练 Pipeline

**一句话**：从医学影像报告到 LoRA 微调模型的端到端训练 Pipeline

**核心价值**：
- 自动化数据预处理、标签生成、模型训练全流程
- 双层评估机制（Rule-Based + LLM Judge）确保标签质量
- 三面板评估框架（Rule/Judge/Pred）提供全面的质量保证

**Pipeline 流程**（30秒快速讲解）：
```
Step 0: 数据子集 → Step 1: 格式对齐 → Step 2: 视觉特征提取 
→ Step 3: 规则标签构造 → Step 3.5: LLM Judge 标签生成 
→ Step 4: LoRA 训练 → Step 5: 三面板评估
```

**讲解要点**：
- 开场（10秒）：问题陈述（如何训练高质量的医学影像报告 Linter 模型）
- Pipeline 流程（30秒）：6 个步骤快速带过
- 核心亮点（20秒）：双层评估 + 三面板评估

## 工业级训练流程

### 核心原则：可复现性 + 版本管理 + 评估门禁

#### 1. 版本管理与可复现性

**每个 Step 都要版本化**：

```
Step 0-1: 数据版本化
├─ 输入数据：Indiana_CXR_v1.0 (固定 hash)
├─ 子集配置：subset_config_v1.0.yaml
└─ 格式对齐：openi_format_v1.0.json

Step 2: 视觉特征版本化
├─ 模型版本：TorchXRayVision_v1.0 (Docker 固定)
├─ 提取配置：extract_config_v1.0.yaml
└─ 输出版本：visual_facts_v1.0.jsonl

Step 3: 规则版本化
├─ 规则文件：rules_v1.0.py
├─ 规则配置：rule_config_v1.0.yaml
└─ 标签版本：rule_labels_v1.0.jsonl

Step 3.5: Judge 版本化
├─ Judge Server：sglang_judge_v1.0 (Docker 固定版本)
├─ Prompt 版本：prompt_v1.2.jinja
└─ Judge 标签：judge_labels_v1.0.jsonl

Step 4: 训练版本化
├─ 训练配置：train_config_v1.0.yaml
├─ 超参数：hyperparams_v1.0.json
├─ 随机种子：seed=42 (固定)
└─ 模型输出：lora_model_v1.0.pt

Step 5: 评估版本化
├─ 评估配置：eval_config_v1.0.yaml
├─ Golden Set：golden_set_v1.0.jsonl (固定)
└─ 评估报告：eval_report_v1.0.md
```

#### 2. 训练流程（工业级）

```
┌─────────────────────────────────────────────────────────────────┐
│ 训练启动：train_pipeline.sh                                      │

│ - 生成 experiment_id (timestamp + git_commit)                    │
└────────────────────────┬────────────────────────────────────────┘
                         │
                         ▼

                         │

│ Step 4: LoRA 训练（GPU，耗时最长）                               │

│ - 固定随机种子（seed=42）                                         │
│ - 训练监控：loss/accuracy/learning_rate                          │
│ - Checkpoint 保存（每 N 个 epoch）                                │

│ 门禁决策                                                          │
│ - 通过门禁 → 模型版本发布 (lora_model_v1.0)                      │
│ - 未通过门禁 → 分析失败原因，迭代改进                             │
└─────────────────────────────────────────────────────────────────┘
```

#### 3. 关键工业级实践

**A. 实验追踪（Experiment Tracking）**
```
- experiment_id: exp_20250101_abc123
- git_commit: abc123def456
- 所有版本号：数据/模型/配置
- 训练日志：TensorBoard / MLflow
- 评估结果：自动化报告
```

**B. 缓存机制（避免重复计算）**
```
- Step 0-1: 基于输入 hash + 配置 hash 缓存
- Step 2: FactStore 缓存（基于图像 hash）
- Step 3.5: Judge 结果缓存（基于 visual_facts + report_facts hash）
```

**C. 门禁机制（Release Gate）**
```
- 评估通过阈值才能发布
- 不通过则自动失败，不发布
- 记录失败原因，用于后续改进
```

**D. 监控与告警**
```
- GPU 使用率 / 内存使用
- 训练 loss 异常
- Judge 调用失败
- 评估指标异常
```

**E. 版本回滚**
```
- 保存所有历史版本
- 可以回滚到任意版本
- 支持版本对比（A/B 测试）
```

#### 4. 迭代流程

```
1. 修改配置/规则/Prompt
   ↓
2. 更新版本号
   ↓
3. 重新运行 Pipeline（利用缓存加速）
   ↓
4. 评估（Golden Set）
   ↓
5. 门禁检查
   ├─ 通过 → 发布新版本
   └─ 未通过 → 分析失败 → 回到步骤 1
```

#### 5. 总结：工业级训练的关键

1. ✅ **版本化一切**：数据/模型/配置都要版本化
2. ✅ **可复现性**：固定随机种子、Docker 固定版本
3. ✅ **缓存机制**：避免重复计算，加速迭代
4. ✅ **评估门禁**：通过门禁才能发布
5. ✅ **实验追踪**：记录所有实验和结果
6. ✅ **监控告警**：训练过程实时监控

### Part 2: Pipeline 详细设计（12-15分钟）- Slides 3-11

**每页 slide 讲解 1.5-2 分钟，重点讲设计决策和 trade-off**

#### Slide 3: Step 0 - 数据子集处理

**职责**：
- 从 Indiana CXR 数据集中筛选符合条件的影像报告对
- 确保数据质量和可用性

**关键设计**：
- 数据筛选条件（格式、质量、完整性）
- 子集规模控制
- 数据分布平衡

**设计原则**：
- **数据质量优先**：确保输入数据符合要求
- **可复现性**：使用固定随机种子

#### Slide 4: Step 1 - 格式对齐

**职责**：
- 将不同来源的数据转换为统一的 OpenI 格式
- 标准化数据结构

**关键设计**：
- OpenI 格式规范
- 数据转换逻辑
- 错误处理机制

**设计原则**：
- **标准化**：统一数据格式便于后续处理
- **容错性**：处理格式异常和缺失字段

#### Slide 5: Step 2 - 视觉特征提取（重点）

**职责**：
- 使用 TorchXRayVision 模型提取影像视觉特征
- 生成 `visual_facts.jsonl` 文件

**关键设计**：
- **TorchXRayVision 模型**：预训练的医学影像分析模型
- **视觉特征提取**：提取关键病理特征（如 Effusion、Cardiomegaly 等）
- **输出格式**：结构化 JSONL 格式

**设计决策**：
- **为什么选择 TorchXRayVision**？
  - 预训练模型，无需从头训练
  - 专门针对医学影像设计
  - 提取的视觉特征质量高

**设计原则**：
- **模块化**：视觉特征提取独立，可替换不同模型
- **可观测性**：记录提取结果和质量指标

#### Slide 6: Step 3 - 规则基础标签构造（重点）

**职责**：
- 基于规则生成基础标签（Rule-Based Labels）
- 合成验证集（Poison Factory）

**关键设计**：
- **规则引擎**：基于临床知识的确定性规则
- **标签生成**：自动生成标签（Omission、Contradiction 等）
- **Poison Factory**：合成带有已知错误的验证集

**设计决策**：
- **为什么用 Rule-Based**？
  - 可审计性：所有标签都有明确的规则依据
  - 可测试性：规则可以独立测试和验证
  - 确定性：相同的输入产生相同的标签

**设计原则**：
- **可审计性**：规则可追溯、可解释
- **可扩展性**：易于添加新规则

#### Slide 7: Step 3.5 - LLM Judge 标签生成（重点）

**职责**：
- 使用 SGLang Judge Server 生成高质量标签
- 识别争议案例（Judge ≠ Rule）

**关键设计**：
- **SGLang Judge Server**：Docker 化部署，版本固定
- **标签生成策略**：基于 LLM 的高质量标签
- **争议识别**：比较 Judge 和 Rule 标签，识别差异

**设计决策**：
- **为什么用 LLM Judge**？
  - 规则可能无法覆盖所有边界情况
  - LLM 可以提供更灵活的判断
  - 双层机制识别规则局限性

**Trade-off 分析**：

| 方案 | 优势 | 劣势 | 选择 | 理由 |
|------|------|------|------|------|
| **Rule-Based Only** | 可审计、可测试 | 可能无法覆盖边界情况 | 不选择 | 规则有局限性，需要 LLM 补充 |
| **LLM Judge Only** | 灵活、覆盖边界情况 | 不可审计、不可预测 | 不选择 | 缺乏可审计性 |
| **Rule + LLM Judge** ✅ | 可审计 + 高质量标签 | 成本较高（需要 LLM 调用） | **选择** | **平衡可审计性和标签质量** |

**设计原则**：
- **版本固定**：Docker 版本确保可复现性
- **容错性**：LLM 调用失败时的降级策略

#### Slide 8: Step 4 - LoRA 训练

**职责**：
- 使用生成的标签训练 LoRA 模型
- 模型版本管理

**关键设计**：
- **LoRA 微调**：参数高效微调
- **训练数据**：使用 Step 3/3.5 生成的标签
- **模型版本管理**：跟踪不同版本的模型

**设计原则**：
- **可复现性**：固定随机种子和超参数
- **版本管理**：记录模型版本和训练配置

#### Slide 9: Step 5 - 三面板评估（重点）

**职责**：
- 评估模型性能

### Pipeline 流程讲解（每个 Step 20-30秒）

**Step 0: 数据子集处理**
"第一步是数据子集处理。我们从 Indiana CXR 数据集中筛选符合条件的影像报告对，确保数据质量和可用性。"

**Step 1: 格式对齐**
"第二步是格式对齐。我们将不同来源的数据转换为统一的 OpenI 格式，标准化数据结构，为后续处理做准备。"

**Step 2: 视觉特征提取**（重点）
"第三步是视觉特征提取。我们使用 TorchXRayVision 模型提取影像的视觉特征，生成结构化的 visual_facts.jsonl 文件。这是 Pipeline 的关键步骤，因为视觉特征是后续标签生成的基础。"

**Step 3: 规则基础标签构造**（重点）
"第四步是规则基础标签构造。我们基于临床知识设计确定性规则，自动生成标签。这个设计的优势是可审计、可测试，所有标签都有明确的规则依据。"

**Step 3.5: LLM Judge 标签生成**（重点）
"第四点五步是 LLM Judge 标签生成。我们使用 SGLang Judge Server 生成高质量标签，并通过比较 Judge 和 Rule 标签识别争议案例。这个双层机制的设计平衡了可审计性和标签质量。"

**Step 4: LoRA 训练**
"第五步是 LoRA 训练。我们使用生成的标签训练 LoRA 模型，采用参数高效微调，节省计算资源。"

**Step 5: 三面板评估**（重点）
"第六步是三面板评估。我们设计了三个维度的评估：Rule Adherence 评估模型是否学会了规则，Silver Agreement 评估模型实际性能，Judge-Rule Gap 揭示规则局限性。这个设计不仅评估模型性能，还帮助我们改进规则。"

### 总结（20秒）

"通过这个 Pipeline，我们实现了从数据预处理到模型评估的全流程自动化。双层标签生成机制确保标签质量，三面板评估框架提供全面的质量保证。整个 Pipeline 采用模块化设计，易于维护和扩展。"
