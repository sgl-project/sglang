# Day 4｜发布：Feature Flag + Canary，把"上线"变成可控实验

## 为什么要练

Google SRE 明确讲：变更带来风险，canary 的目标是用小流量获得真实输入下的信心。Feature toggle/flag 的核心是部署与发布解耦，随时可回切。

## 目的

你能说清"怎么低风险进化"。

## 目标

写一份《灰度/回滚策略》（含阈值）。

---

## Feature Flags（功能开关）

### 定义

Feature Flag 允许你在不重新部署的情况下，动态控制功能是否启用。实现**部署与发布的解耦**。

### 应用场景

**按维度放量**：
- [ ] **model_version**：模型版本切换
- [ ] **prompt_version**：Prompt 版本切换
- [ ] **validator_strictness**：Validator 严格程度（high/medium/low）
- [ ] **其他**：`____`

### Feature Flag 配置示例

```yaml
feature_flags:
  - name: model_version_v2
    type: percentage
    enabled: true
    percentage: 5  # 5% 流量使用 v2
    
  - name: prompt_version_v3
    type: user_segment
    enabled: true
    segments:
      - internal_users: 100%  # 内部用户 100%
      - external_users: 1%    # 外部用户 1%
      
  - name: validator_strictness
    type: percentage
    enabled: true
    percentage: 10  # 10% 流量使用高严格度
```

### 实现方式

- [ ] Split.io
- [ ] 自建（Redis/数据库）

## Canary 发布策略

### 定义

Canary 发布是一种渐进式部署策略，先在少量流量上验证新版本，然后逐步增加流量，直到 100%。

### 发布流程

```
┌─────────────┐
│  部署新版本  │
│  (但不启用)  │
└──────┬──────┘
       │
       ▼
┌─────────────┐      ┌─────────────┐
│  1% 流量    │ ────▶│   观察指标   │
│  (Canary)   │      │ L0/L1       │
└──────┬──────┘      └──────┬──────┘
       │                    │
       │ 通过?              │
       ├────────────────────┤
       │                    │
       ▼                    ▼
┌─────────────┐      ┌─────────────┐
│  5% 流量    │ ────▶│   继续观察   │
└──────┬──────┘      └──────┬──────┘
       │                    │
       ▼                    │
┌─────────────┐             │
│  25% 流量   │             │
└──────┬──────┘             │
       │                    │
       ▼                    │
┌─────────────┐             │
│  100% 流量  │             │
└─────────────┘             │
                            │
                            ▼
                   ┌─────────────────┐
                   │   任何阶段异常   │
                   │   立即回滚      │
                   └─────────────────┘
```

### Canary 流量阶段

**阶段 1：1% 流量**
- 持续时间：`____` 分钟/小时
- 观察指标：L0/L1
- 通过条件：所有指标正常

**阶段 2：5% 流量**
- 持续时间：`____` 分钟/小时
- 观察指标：L0/L1
- 通过条件：所有指标正常

**阶段 3：25% 流量**
- 持续时间：`____` 分钟/小时
- 观察指标：L0/L1
- 通过条件：所有指标正常

**阶段 4：100% 流量**
- 持续时间：`____` 分钟/小时
- 观察指标：L0/L1
- 通过条件：所有指标正常

  - Delta: `±____%`
  - **触发回滚**：`✅ 是` / `❌ 否`

- [ ] **p99 延迟增加 > `____%`**

  - **触发回滚**：`✅ 是` / `❌ 否`

- [ ] **Schema Fail Rate 增加 > `____%`**（例如：2×）

## 回滚流程

### 自动回滚

**触发条件**：
- 任一回滚条件满足
- 持续时间 > `____` 分钟（避免短暂波动）

**回滚动作**：
1. 立即将 Canary 流量切回旧版本（0%）
2. 发送告警通知（Slack/电话）
3. 记录回滚日志和原因

### 手动回滚

**触发场景**：
- On-Call 人员判断需要回滚
- 业务方反馈严重问题

**回滚步骤**：
1. 确认回滚原因
2. 执行回滚（Feature Flag → 0% 或切换版本）
3. 验证回滚成功（指标恢复正常）
4. 记录回滚日志

### 回滚验证

**验证指标**：
- [ ] Error Rate 恢复到基线水平
- [ ] Latency 恢复到基线水平
- [ ] 服务恢复正常

**验证时间**：回滚后 `____` 分钟
