# Day 6｜事故响应：Runbook + Postmortem，把"出事"变成组织学习

## 为什么要练

Google SRE workbook 把 on-call/incident/postmortem 作为核心实践：出事不可怕，关键是快速止血 + 防复发。

## 目的

证明你是"交给你我能睡得着"的人。

## 目标

一份可执行 runbook + 一份复盘模板，并做一次桌面演练。

---

## Runbook（运维手册）

### 定义

Runbook 是一份可执行的操作手册，当告警触发时，On-Call 人员可以按照步骤快速止血。

### 告警触发 → 快速止血流程

```
告警触发
    │
    ▼
┌─────────────┐
│ 查看 Dashboard│ ← 先看哪个 dashboard
└─────┬───────┘
      │
      ▼
┌─────────────┐
│  判断严重性  │ ← 影响范围、用户影响
└─────┬───────┘
      │
      ├─────────────────┬─────────────────┐
      │                 │                 │
      ▼                 ▼                 ▼
┌───────────┐   ┌───────────┐   ┌───────────┐
│  立即回滚  │   │  触发降级  │   │  定位根因  │
│ (Critical)│   │  (Warning)│   │   (Info)  │
└───────────┘   └───────────┘   └─────┬─────┘
                                      │
                                      ▼
                              ┌─────────────┐
                              │  Trace→Log  │
                              │   定位问题   │
                              └─────────────┘
```

## 告警响应流程

### Step 1: 告警接收

**告警来源**：
- [ ] PagerDuty / OpsGenie
- [ ] Slack / 邮件
- [ ] 电话 / 短信

**告警信息**：
- 告警类型：`____`（Critical / Warning / Info）
- 告警内容：`____`
- 触发时间：`____`

**第一步动作**：
1. 确认收到告警（ACK）
2. 查看告警详情
3. 判断严重性（Critical / Warning / Info）

---

### Step 2: 查看 Dashboard

**需要查看的 Dashboard**：

1. **实时监控 Dashboard（On-Call Dashboard）**
   - URL：`____`
   - 关键指标：RPS、p95、Error Rate、Queue Depth
   - 判断：是否有异常

2. **业务健康 Dashboard**
   - URL：`____`
   - 关键指标：Fallback Rate、Schema Fail、Cost
   - 判断：业务是否受影响

3. **链路追踪 Dashboard**
   - URL：`____`
   - 功能：查看 Trace、定位慢 Span
   - 判断：哪个阶段出现问题

**Dashboard 检查清单**：
- [ ] 查看 RPS（是否突增/突降）
- [ ] 查看 p95/p99 Latency（是否超标）
- [ ] 查看 Error Rate（是否异常）
- [ ] 查看 Queue Depth（是否积压）
- [ ] 查看 Fallback Rate（是否升高）
- [ ] 查看最近告警列表

---

### Step 3: 判断严重性

#### Critical（严重）→ 立即回滚

**触发条件**（任一满足）：
- [ ] Error Rate > `____%`（持续 5 分钟）
- [ ] p95 > `____ms`（持续 10 分钟）
- [ ] 服务不可用
- [ ] 大量用户反馈

**动作**：
1. 立即回滚（Feature Flag → 0% 或切换版本）
2. 发送通知（Slack/电话）
3. 验证回滚成功
4. 记录回滚日志

#### Warning（警告）→ 触发降级

**触发条件**（任一满足）：
- [ ] Error Rate > `____%`（持续 10 分钟）
- [ ] p95 > `____ms`（持续 15 分钟）
- [ ] Queue Depth > `____`
- [ ] 部分用户受影响

**动作**：
1. 触发降级（使用备用方案）
2. 发送通知（Slack）
3. 持续监控
4. 定位根因（Trace → Log）

#### Info（信息）→ 定位根因

**触发条件**：
- [ ] 指标异常但未超阈值
- [ ] 告警噪音（误报）

**动作**：
1. 查看 Trace 和 Log
2. 定位问题根因
3. 记录日志
4. 评估是否需要修复

---

### Step 4: 立即止血（Critical）

#### 回滚流程

**回滚条件**（任一满足即回滚）：
- [ ] Error Rate > `____%`
- [ ] p95 > `____ms`
- [ ] 服务不可用
- [ ] 关键业务指标严重退化

**回滚步骤**：

1. **确认回滚原因**
   - 问题描述：`____`
   - 影响范围：`____`
   - 用户影响：`____`

2. **执行回滚**
   - Feature Flag 方式：
     - 将新版本 Feature Flag 设置为 `0%`
     - 或切换到旧版本
   
   - 版本切换方式：
     - 切换到上一个稳定版本：`v____`
     - 命令：`____`

3. **验证回滚成功**
   - [ ] Error Rate 恢复到基线：`____%`
   - [ ] Latency 恢复到基线：p95 = `____ms`
   - [ ] 服务恢复正常
   - [ ] 用户反馈正常

4. **记录回滚日志**
   - 回滚时间：`____`
   - 回滚原因：`____`
   - 回滚版本：`v____` → `v____`
   - 回滚验证：`✅ 成功` / `❌ 失败`

**回滚时间目标**：`< ____ 分钟`（从告警到回滚完成）

---

### Step 5: 触发降级（Warning）

#### 降级流程

**降级条件**（任一满足即降级）：
- [ ] 上游服务不可用（熔断打开）
- [ ] Latency > `____ms`
- [ ] Error Rate > `____%`
- [ ] Queue Depth > `____`

**降级步骤**：

1. **确认降级原因**
   - 问题描述：`____`
   - 影响范围：`____`

2. **触发降级**
   - 降级策略：
     - [ ] 小模型降级
     - [ ] 规则降级
     - [ ] 缓存降级
     - [ ] 人工降级
   
   - 降级命令/配置：`____`

3. **验证降级生效**
   - [ ] 降级策略已触发
   - [ ] 延迟降低：p95 = `____ms`（降级后）
   - [ ] 服务可用性恢复
   - [ ] 用户可正常使用（降级功能）

4. **记录降级日志**
   - 降级时间：`____`
   - 降级原因：`____`
   - 降级策略：`____`
   - 降级验证：`✅ 成功` / `❌ 失败`

---

### Step 6: 定位根因（Trace → Log）

#### 定位流程

**定位步骤**：

1. **查看 Trace**
   - 输入 `trace_id` 或 `request_id`：`____`
   - 查看完整调用链
   - 识别慢 Span（超过阈值）

2. **关联 Log**
   - 使用 `trace_id` 关联日志
   - 查看错误日志
   - 查看相关上下文

3. **分析根因**
   - 问题出现在哪个阶段：`____`（Preprocess / OCR/VLM / LLM / Validate / Store）
   - 根因分析：`____`
   - 触发条件：`____`

4. **收集证据**
   - 相关 Trace ID：`____`
   - 相关 Log 片段：`____`
   - 相关 Metrics：`____`

#### 常见问题定位

| 问题现象 | 可能原因 | 检查项 | 解决方案 |
|---------|---------|--------|----------|
| **p95 突增** | 慢 Span | 查看 Trace，找出最慢的 Span | 优化慢 Span |
| **Error Rate 升高** | 上游服务失败 | 查看 Log，检查错误类型 | 检查上游服务 |
| **Schema Fail 升高** | Validator 规则变化 | 查看 Schema Fail 日志 | 检查 Validator |
| **Fallback 升高** | 主服务不可用 | 查看熔断状态 | 检查主服务 |
| **Queue Depth 升高** | 处理能力不足 | 查看 RPS vs 处理能力 | 扩容或限流 |

---

### Step 7: 故障恢复

**恢复步骤**：

1. **修复问题**
   - 问题根因：`____`
   - 修复方案：`____`
   - 修复时间：`____`

2. **验证修复**
   - [ ] 问题已解决
   - [ ] 指标恢复正常
   - [ ] 回归测试通过

3. **逐步发布**
   - 使用 Canary 发布（1% → 5% → 25% → 100%）
   - 每步观察指标
   - 确认无异常

4. **关闭降级/回滚**
   - 移除降级策略
   - 确认服务正常

## 桌面演练（Tabletop Exercise）

**场景 1：p95 突增**
- 告警：p95 > 500ms（持续 10 分钟）
- 演练步骤：按照 Runbook 执行

**场景 2：Error Rate 升高**
- 告警：Error Rate > 5%（持续 5 分钟）
- 演练步骤：按照 Runbook 执行

**场景 3：上游服务不可用**
- 告警：熔断打开
- 演练步骤：按照 Runbook 执行

**场景 4：Schema Fail 升高**
- 告警：Schema Fail Rate > 2%
- 演练步骤：按照 Runbook 执行

- [ ] 能触发降级流程

## Runbook 维护

### 定期审查

- **频率**：每月一次
- **审查内容**：
  - 更新阈值
  - 更新命令/配置
  - 补充新场景

### 持续改进

- 每次事故后，更新 Runbook
- 补充新的故障场景
- 优化响应流程
