# Day 5｜保护与容错：Engineering for Failure（免疫系统）

## 为什么要练

SRE/大厂不是追求"永不失败"，而是"失败可控、可恢复"。Error budget 也强调"发布速度 vs 稳定性"的现实权衡。

## 目的

把你提过的 rate limit/protection 讲成"策略→触发→动作→验证"。

## 目标

产出一张《保护矩阵》。

---

## 保护策略矩阵

### 核心原则

**Engineering for Failure**：假设系统会失败，设计保护机制使失败可控、可恢复。

---

### 3. 熔断（Circuit Breaker）

**策略**：当上游服务持续失败时，快速失败，避免级联故障。

#### 触发条件

| 触发场景 | 阈值 | 当前值 | 状态 |
|---------|------|--------|------|
| **失败率超过** | `> ____%` | `____%` | `关闭` / `打开` / `半开` |
| **连续失败数** | `> ____` | `____` | `关闭` / `打开` / `半开` |
| **响应时间超过** | `> ____ms` | `____ms` | `关闭` / `打开` / `半开` |

#### 动作

**熔断状态**：

1. **关闭（Closed）**：正常状态
   - 请求正常通过
   - 监控错误率和延迟

2. **打开（Open）**：熔断状态
   - 立即拒绝请求（不调用上游）
   - 返回默认响应或错误
   - 持续时间：`____` 秒

3. **半开（Half-Open）**：试探状态
   - 允许少量请求通过
   - 试探请求数：`____` 个
   - 如果成功 → 关闭
   - 如果失败 → 重新打开

**熔断响应**：
- [ ] **直接返回错误**：503 Service Unavailable
- [ ] **降级处理**：使用缓存/规则/备用服务
- [ ] **快速失败**：不等待超时

#### 验证

**熔断效果**：
- 故障隔离：上游故障不影响下游（延迟降低）
- 快速失败：响应时间从 `____ms` → `____ms`
- 自动恢复：熔断后 `____` 秒自动进入半开状态

### 4. 幂等（Idempotency）

**策略**：确保相同请求只处理一次，避免重复计费/重复写库。

#### 触发条件

**需要幂等的操作**：
- [ ] **写操作**（Create/Update）
- [ ] **计费操作**（Cost calculation）
- [ ] **外部 API 调用**（如果支持幂等）

#### 动作

**幂等实现**：

1. **Request ID 去重**
   - 客户端提供 `request_id`
   - 服务端存储已处理的 `request_id`
   - 存储时间：`____` 小时（TTL）

2. **幂等键（Idempotency Key）**
   - 键组成：`user_id + operation + params_hash`
   - 存储位置：Redis / 数据库

3. **状态检查**
   - 如果 `request_id` 已存在 → 返回之前的结果
   - 如果不存在 → 处理请求并存储结果

**幂等范围**：
- [ ] **全局幂等**：所有请求共享
- [ ] **用户级幂等**：每个用户独立
- [ ] **时间窗口幂等**：`____` 分钟内

#### 验证

**幂等效果**：
- 重复请求率：`____%`
- 重复处理率：`< ____%`（应该接近 0%）
- 计费准确性：100%（无重复计费）

### 5. 降级（Degradation）

**策略**：当主服务不可用时，使用备用方案，保证基本功能。

#### 触发条件

| 触发场景 | 阈值 | 当前值 | 状态 |
|---------|------|--------|------|
| **上游服务不可用** | 熔断打开 | - | `降级` |
| **延迟过高** | `> ____ms` | `____ms` | `降级` |
| **错误率过高** | `> ____%` | `____%` | `降级` |
| **手动触发** | - | - | `降级` |

#### 动作

**降级策略**：

1. **小模型降级**
   - 主模型：`____`（成本高、质量好）
   - 降级模型：`____`（成本低、质量可接受）
   - 降级条件：主模型延迟 > `____ms` 或错误率 > `____%`

2. **规则降级**
   - 主方案：AI 提取
   - 降级方案：规则提取（正则/模板）
   - 降级条件：AI 服务不可用

3. **缓存降级**
   - 主方案：实时处理
   - 降级方案：返回缓存结果
   - 缓存 TTL：`____` 小时

4. **人工降级**
   - 主方案：自动处理
   - 降级方案：转人工审核
   - 降级条件：自动处理失败

5. **功能降级**
   - 主方案：完整功能
   - 降级方案：简化功能
   - 降级条件：`____`

**降级通知**：
- [ ] 记录降级日志
- [ ] 发送降级告警
- [ ] 更新状态到监控系统

#### 验证

**降级效果**：
- 降级触发率：`____%`
- 降级成功率：`> ____%`（降级后仍能处理）
- 降级延迟：`< ____ms`（降级后延迟应降低）
- 用户影响：`____`（评估用户感知）

## 保护策略组合

### 策略优先级

```
请求进入
    │
    ▼
┌───────────┐
│  限流检查  │ ← 第一道防线
└─────┬─────┘
      │
      ▼
┌───────────┐
│  熔断检查  │ ← 快速失败
└─────┬─────┘
      │
      ▼
┌───────────┐
│  幂等检查  │ ← 去重
└─────┬─────┘
      │
      ▼
┌───────────┐
│  处理请求  │
│   (重试)   │
└─────┬─────┘
      │
      ▼
┌───────────┐
│  降级处理  │ ← 最后兜底
└───────────┘
```

### 策略配置矩阵

| 策略 | 触发条件 | 动作 | 优先级 | 验证指标 |
|------|---------|------|--------|----------|
| **限流** | RPS > `____` | 返回 429 | 1 | p95、429 Rate |
| **熔断** | 失败率 > `____%` | 快速失败 | 2 | 延迟降低 |
| **幂等** | 重复 request_id | 返回缓存 | 3 | 重复处理率 |
| **重试** | 可恢复错误 | 指数退避重试 | 4 | 成功率提升 |
| **降级** | 服务不可用 | 使用备用方案 | 5 | 降级成功率 |
